1. Library Dependency Graph

The fork creates two new libraries under ydb/core/kqp/opt/cbo/. The key invariant: the fork depends on shared DQ infrastructure (dq/opt/core) but never on the YQL-owned CBO code (yql/essentials/core/cbo or ydb/library/yql/dq/opt).

Why this split?

The original ydb/library/yql/dq/opt is a single library containing both CBO solver code and shared DQ infrastructure (dq_opt.h, dq_opt_phy.h, etc.). To fork the CBO without duplicating infrastructure, we split dq/opt into:

  • dq/opt/core — infrastructure only (uses SRCDIR to compile files from the parent dir without moving them)
  • dq/opt — CBO solver (now depends on core)

The fork's solver/ depends on dq/opt/core for infrastructure but compiles its own copies of the CBO solver files.

2. File Map & Classification

Every file in the fork falls into one of three tiers:

  • CBO — Core optimizer logic: algorithms, types, cost model, hypergraph
  • STATS — Statistics infrastructure: stat inference, propagation, selectivity. Not CBO per se but tightly coupled to CBO types
  • PLUMB — Join rewrite plumbing: included because CBO type changes propagate through function signatures

Boundary Point Details

kqp_statistics_transformer.cpp — FromYqlColumnStatMap() (file-scope static)

Iterates THashMap<TString, NYql::TColumnStatistics> and field-by-field copies each entry into NKikimr::NKqp::TColumnStatistics. Both structs have identical fields (NumUniqueVals, HyperLogLog, CountMinSketch, EqWidthHistogramEstimator, Type); shared pointers are copied as-is. Lives in the transformer rather than kqp_statistics.cpp because the argument type requires #include <yql/essentials/core/yql_statistics.h>, which transitively includes yql/essentials/core/cbo — a dependency that kqp/opt/cbo explicitly denies via CHECK_DEPENDENT_DIRS.

kqp_statistics.cpp — TKqpStatsStore::GetStats() / SetStats()

Keyed by NYql::TExprNode::UniqueId() (a ui64). Stores NKikimr::NKqp::TOptimizerStatistics. This is the KQP parallel to TTypeAnnotationContext::StatisticsMap — decoupled so KQP optimizer passes own their stats without mutating the shared YQL context.

dq_opt_join.cpp:182-189 — EJoinAlgoType static_cast

TEquiJoinLinkSettings::JoinAlgo is NYql::EJoinAlgoType (from the YQL equi-join infrastructure). TOptimizerHints::TJoinAlgoHint::Algo is NKikimr::NKqp::EJoinAlgoType (from the fork). Both enums have identical integer values by design. The cast occurs in two directions: hint → link settings, and link settings → local variable for KQP enum comparisons.

kqp_statistics_transformer.cpp:1012-1017 — FSM wiring

Builds NKikimr::NKqp::TOrderingsStateMachine instances (shufflings + sortings) from collected FDs and stores them on TKqpStatsStore.ShufflingsFSM / .SortingsFSM. The YQL TTypeAnnotationContext also gets its own FSMs built from the same data, but those use NYql::NDq::TOrderingsStateMachine. The two FSM types are structurally identical but live in different namespaces.

4. Substantive Change Rationale

For each file with non-mechanical diffs, here is why the change was necessary — expressed as dependency chains.

kqp_statistics.h (+80/-34)

Completely restructured. Root cause: TOptimizerStatistics in yql/essentials/core/yql_statistics.h uses NYql::NDq::TJoinColumn, NYql::NDq::TOrderingsStateMachine::TLogicalOrderings, and NYql::TColumnStatistics. If we just included that header, every CBO type in the fork would be an NYql type, defeating the purpose.

→ So we duplicated TOptimizerStatistics and all its nested types into NKikimr::NKqp, using the fork's own TJoinColumn and TOrderingsStateMachine. Added TKqpStatsStore as a KQP-owned stats map (parallel to TTypeAnnotationContext::StatisticsMap). The one boundary conversion (FromYqlColumnStatMap) lives in kqp_statistics_transformer.cpp rather than here because including yql_statistics.h in the kqp/opt/cbo library would transitively pull in yql/essentials/core/cbo, violating the CHECK_DEPENDENT_DIRS restriction.

cbo_optimizer_new.h (+45/-37)

Root cause: The original defines NYql::EJoinAlgoType (which lives in yql/essentials/core/yql_cost_function.h, not even in this file). The fork's copy needs its own NKikimr::NKqp::EJoinAlgoType so that all CBO types (TJoinAlgoHints, IProviderContext) use the KQP enum.

→ Added the EJoinAlgoType enum + AllJoinAlgos constexpr directly. Removed #include <yql_cost_function.h>. Changed all NDq::TJoinColumn references to the fork's TJoinColumn. Forward-declared NYql::TExprContext (needed by IOptimizerFactory).

cbo_interesting_orderings.h (+38/-6)

Root cause: The original #include <yql_cost_function.h> pulls in NYql::NDq::TJoinColumn. But the fork needs its own TJoinColumn in NKikimr::NKqp to avoid using YQL types.

→ Inlined the entire TJoinColumn struct (with operator==, operator<, THashFunction implementations in the .cpp). Removed TShuffling::SetNatural() and IsNatural field (unused after orderings rework). Added GetColumns() accessor.

dq_opt_stat.h (+27/-23) and dq_opt_stat.cpp (+235/-237)

Root cause: Every InferStatisticsFor*() function takes TTypeAnnotationContext* and calls typeCtx->GetStats() / typeCtx->SetStats(). But TTypeAnnotationContext is shared YQL state. The fork needs its own stats store.

→ Replaced TTypeAnnotationContext* with TKqpStatsStore* in all 20+ function signatures. Changed all typeCtx->GetStats/SetStats to kqpStats->GetStats/SetStats. For functions that also need typeCtx for non-stats purposes (e.g., InferStatisticsForExtendBase accesses OrderingsFSM), added both parameters. Functions like NeedCalc, IsConstantExpr, IsLiteralDataExpr were moved into an anonymous namespace and reordered due to forward-declaration requirements.

dq_opt_stat_transformer_base.h (+5/-2) and .cpp (+33/-27)

Root cause: TDqStatisticsTransformerBase stores TypeCtx and passes it to all InferStatisticsFor*() calls. After the stat function signatures changed, the transformer also needs TKqpStatsStore*.

→ Added TKqpStatsStore* KqpStats member. Constructor takes it as a new parameter. All InferStatisticsFor*(input, TypeCtx) calls changed to InferStatisticsFor*(input, KqpStats) (or InferStatisticsFor*(input, KqpStats, TypeCtx) for the few that need both).

dq_opt_join.h (+13/-18) and dq_opt_join.cpp (+15/-7)

Root cause: DqRewriteEquiJoin() calls typeCtx.GetStats()/typeCtx.SetStats() to propagate equi-join statistics. It also processes TOptimizerHints which use the fork's EJoinAlgoType.

→ Added TKqpStatsStore& parameter to both DqRewriteEquiJoin overloads. Introduced static_cast between NYql::EJoinAlgoType (in TEquiJoinLinkSettings) and NKikimr::NKqp::EJoinAlgoType (in hints). Removed DqCollectJoinRelationsWithStats declaration (moved to dq_opt_join_cost_based.h where it uses TKqpStatsStore).

dq_opt_join_cost_based.h (+11/-0) and .cpp (+15/-5)

Root cause: DqOptimizeEquiJoinWithCosts() is the main CBO entry point. It calls typesCtx.GetStats() to collect relation stats and typesCtx.SetStats() to store results.

→ Added TKqpStatsStore& parameter to all overloads. DqCollectJoinRelationsWithStats moved here (was in dq_opt_join.h) with TKqpStatsStore parameter. All typesCtx.GetStats/SetStats calls replaced with kqpStats.GetStats/SetStats.

dq_opt_join_cbo_factory.cpp (+2/-3)

Root cause: MakeJoinCostBasedOptimizerPG calls MakePgOptimizerNew() from yql/essentials/parser/pg_wrapper. The fork doesn't need/want the PG optimizer dependency.

→ Replaced PG path with Y_ABORT("PG optimizer is not supported in the ydb-owned CBO"). Removed pg_wrapper/interface/optimizer.h include.

dq_opt_join_hypergraph.h (+0/-1)

Root cause: Original includes yql_cost_function.h (for TJoinColumn). The fork's TJoinColumn lives in cbo_interesting_orderings.h instead.

→ Removed the one include line. TJoinColumn is already available via cbo_optimizer_new.h → cbo_interesting_orderings.h.

dq_opt_predicate_selectivity.cpp (+3/-1)

Root cause: TPredicateSelectivityComputer methods were defined with explicit NYql::NDq:: qualification. In the fork they live in NKikimr::NKqp.

→ Wrapped implementations in namespace NKikimr::NKqp { ... }. Removed explicit NYql::NDq:: qualification on method definitions.

dq_cbo_ut.cpp (+4/-16)

Root cause: Tests use TTypeAnnotationContext as stats storage and call MakePgOptimizerNew for the PG test.

→ Replaced TTypeAnnotationContext typeCtx with TKqpStatsStore kqpStats in all test functions. Removed the DqOptimizeEquiJoinWithCostsPG test (PG not supported). Removed pg_wrapper include. Suite renamed from DQCBO to KqpCBO.

dq_opt_hypergraph_ut.cpp (+1/-3)

Root cause: Tests reference NDq::TJoinColumn and use NYql::NDq namespace.

→ Consolidated three using namespace lines into one using namespace NKikimr::NKqp. Changed include to the fork's dq_opt_join.h. Suite renamed to KqpHypergraphBuild.

5. Why Each Non-CBO File Is Copied

The dependency chain problem

Even files that aren't "CBO" in the algorithmic sense must be copied because of transitive type dependencies:

dq_opt_join.h/cpp (Join rewrite plumbing)

DqRewriteEquiJoin() calls GetStats()/SetStats() on the stats store. If we kept using the original, it would call typeCtx.GetStats() — the shared YQL stats map. We need it to use kqpStats.GetStats() instead. The function also processes TOptimizerHints which contains NKqp::EJoinAlgoType, requiring static_cast at the boundary with TEquiJoinLinkSettings::JoinAlgo (NYql::EJoinAlgoType).

dq_opt_join_tree_node.h/cpp

Defines ToString(EJoinAlgoType) and TJoinOptimizerNodeInternal which references the fork's EJoinAlgoType, EJoinKind, and TJoinColumn. If it used the originals, these types would be NYql::* and wouldn't match the fork's CBO interfaces.

dq_opt_conflict_rules_collector.h/cpp

Uses EJoinKind for commutativity/associativity predicates. These predicates are called by MakeJoinHypergraph() which passes the fork's NKqp::EJoinKind values. A mismatch between enum types would cause compilation errors.

dq_opt_stat.h/cpp (Statistics infrastructure)

Not CBO per se (doesn't decide join order), but provides the statistics data that CBO algorithms consume. Every stat inference function takes TKqpStatsStore* and produces NKqp::TOptimizerStatistics. Can't use the original because it produces NYql::TOptimizerStatistics and stores in TTypeAnnotationContext.

bitset.h

No type dependencies — purely mechanical namespace change. Copied for organizational coherence (solver is self-contained) and to allow KQP to freely modify the bitset utilities. The helpers (Overlaps, IsSubset, GetLowestSetBit, GetNextSubset) are called unqualified inside NKikimr::NKqp blocks; keeping them in the same namespace avoids explicit qualification at every call site. dq_opt_log.h and dq_opt_log.cpp were also initially copied but turned out to be dead files (the .cpp was never in SRCS, the header was only included by that .cpp) and have since been deleted.

7. Entry Points

The KQP optimizer calls into the fork at exactly two function entry points, plus the statistics transformer:

kqp_opt_log.cpp:215 — DqOptimizeEquiJoinWithCosts()

The CBO path. Called when useCBO is enabled for the query. Passes KqpCtx.KqpStats (the KQP-owned stats store), constructs an IOptimizerNew via MakeNativeOptimizerNew(), and gets back a rewritten equi-join tree.

kqp_opt_log.cpp:230 — DqRewriteEquiJoin()

The non-CBO path (rule-based join rewriting). Also uses KqpCtx.KqpStats for statistics propagation even when CBO is off.

ydb/core/kqp/opt/rbo/rules/apply_cbo.cpp — TOptimizeCBOTreeRule::SimpleMatchAndApply()

The RBO → CBO bridge. When the rule-based optimizer produces a TOpCBOTree operator, this rule fires. It converts the RBO join tree into CBO types (TRelOptimizerNode, TJoinOptimizerNode), calls MakeNativeOptimizerNew() to get an IOptimizerNew, runs JoinSearch(), then converts the optimized tree back to RBO operators via ConvertOptimizedTree(). Distinct from the DQ logical rewrite path in kqp_opt_log.cpp.

kqp_statistics_transformer.cpp — InferStatisticsFor*()

The statistics transformer walks the plan bottom-up and calls InferStatisticsFor*() for each operator. It passes KqpStats to all inference functions. This is where table-level stats (from TTypeAnnotationContext) enter the KQP stats store via FromYqlColumnStatMap().

Key observations

  • cbo_interesting_orderings.h and bitset.h are the two true leaves — no intra-fork includes.
  • cbo_optimizer_new.h is the central hub: every solver subgroup depends on it, directly or transitively via dq_opt_join_tree_node.h or dq_opt_join_hypergraph.h.
  • kqp_statistics.h is the second hub: all three solver subgroups include it (directly via dq_opt_stat.h, dq_opt_join.h, dq_opt_join_cost_based.h, or via cbo_optimizer_new.h).
  • The hypergraph group (bitset.h → dq_opt_join_tree_node.h → dq_opt_join_hypergraph.h → dq_opt_dphyp_solver.h) forms a strict linear chain; dq_opt_make_join_hypergraph.h fans in from the side.
  • dq_opt.h (shared, not forked) is included by the two entry-point headers: dq_opt_stat.h and dq_opt_join.h.

10. Design Decision Analysis

Each major structural decision during the fork had at least two viable paths. This section documents the problem each decision faced, the options that were on the table, which one was chosen and why, and what would have gone wrong with each rejected alternative.

1. Copy vs. Wrapper / Delegation

scope of fork
KQP needs its own CBO that can diverge from the YQL/YT CBO over time (KQP-specific join algorithms, KQP-specific statistics, shuffle elimination). The question is how much of the YQL CBO code to reproduce vs. delegate to.
✓ CHOSEN — Full source copy in a new namespace
All CBO source files are physically copied under ydb/core/kqp/opt/cbo/ and …/solver/. Types live in NKikimr::NKqp. The two implementations evolve independently with zero cross-dependency.
Clear ownership boundary; no runtime overhead; each team can evolve their copy without coordinating with the other. ODR safety is automatic (different namespaces).
✗ Thin wrappers / adapter pattern
Keep NYql::NDq types internally; expose KQP-branded adapter classes that forward calls to the real implementation. E.g., NKqp::IOptimizerNew delegates to NYql::IOptimizerNew.
Fails: any KQP-specific change (new parameter, new method) requires touching the YQL interface too, defeating the fork. Adapter indirection also makes debugging harder and incurs vtable overhead on every CBO call.
✗ Template policy / CRTP base classes
Parameterise the CBO on a Statistics policy type so both YQL (NYql::TOptimizerStatistics) and KQP (NKqp::TOptimizerStatistics) can use the same template body.
Fails: the CBO algorithms use statistics in dozens of call sites across many files. Templatising all of them turns every header into a massive template instantiation, exploding compile times. Template errors are also notoriously hard to diagnose.
✗ Subclassing YQL types
Derive NKqp::TOptimizerStatistics : NYql::TOptimizerStatistics, etc. Use dynamic_cast or virtual overrides to extend behaviour.
Fails: object slicing when stored in shared_ptr<NYql::TOptimizerStatistics>. The base class is a plain struct with no virtual destructor. Also, the include path for the base still pulls yql/essentials/core/cbo into the KQP library.
The full-copy approach is the only one that gives genuine independence. It is more code on day one but costs nothing at runtime and allows each CBO to evolve freely.

2. dq/opt Library Split (SRCDIR trick)

shared infrastructure vs. CBO code
ydb/library/yql/dq/opt is a single library containing both pure-infrastructure files (dq_opt.h, dq_opt_phy.h, ...) and CBO files (dq_opt_join_cost_based.*, etc.). The fork's solver needs the infrastructure but must not pull in the YQL CBO. Making the fork depend on the entire dq/opt library would be circular (and forbidden by CHECK_DEPENDENT_DIRS).
✓ CHOSEN — Create dq/opt/core sub-library with SRCDIR
Add a dq/opt/core/ya.make that uses SRCDIR(ydb/library/yql/dq/opt) to compile the infrastructure files from the parent directory without moving them. dq/opt gains a PEERDIR on dq/opt/core; the fork's solver depends only on dq/opt/core.
Infrastructure files don't move (no upstream diff noise); dq/opt is backward-compatible (still compiles all the same SRCS); the fork gets clean infrastructure access without the YQL CBO.
✗ Physically move infra files into dq/opt/core/
Move dq_opt.h, dq_opt_phy.h, etc. to a new subdirectory. Update all consumers.
Hundreds of #include "…/dq_opt.h" across the codebase would need updating. Creates massive upstream diff, merge conflicts, and reviewability issues. The SRCDIR trick achieves the same result with zero file moves.
✗ Fork depends on full dq/opt
Have the solver PEERDIR on ydb/library/yql/dq/opt directly, accepting that both YQL and KQP CBO code are compiled together.
Fails immediately: dq/opt has CHECK_DEPENDENT_DIRS(DENY PEERDIRS ydb/core/kqp/opt/cbo/solver) specifically to prevent this. Even without the guard, it would cause ODR violations — DqOptimizeEquiJoinWithCosts would be defined in both NYql::NDq and NKikimr::NKqp from the same TU if both were linked.
✗ Duplicate infrastructure files too
Copy dq_opt.h, dq_opt_phy.h, etc. into the solver directory alongside the CBO copies.
Infrastructure files don't need any KQP changes (they don't touch statistics or CBO types). Duplicating them adds maintenance burden with zero benefit. Any bug fix to, say, dq_opt_phy.cpp would need to be applied in two places.
The SRCDIR trick is an Arcadia-idiomatic way to split a library into sub-components without moving source files. It is zero-diff for existing consumers and cleanly separates infrastructure from CBO logic.

3. TKqpStatsStore vs. Extending TTypeAnnotationContext

statistics ownership
All InferStatisticsFor*() functions need to read and write per-node statistics. The YQL versions do this through TTypeAnnotationContext::StatisticsMap. The fork's types (NKqp::TOptimizerStatistics) are not the same C++ types as NYql::TOptimizerStatistics, so the map can't hold both without conversion.
✓ CHOSEN — New TKqpStatsStore class
A simple THashMap<ui64, shared_ptr<NKqp::TOptimizerStatistics>> keyed by TExprNode::UniqueId(). Stored on TKqpOptimizeContext. Passed as a pointer to all stat inference functions alongside (or instead of) TTypeAnnotationContext*.
KQP owns its stats without touching shared YQL state. Signature changes are mechanical and localised. The KQP FSMs (ShufflingsFSM, SortingsFSM) live here too, typed as NKqp::TOrderingsStateMachine.
✗ Add a second map to TTypeAnnotationContext
Extend TTypeAnnotationContext with a KqpStatisticsMap: THashMap<ui64, shared_ptr<NKqp::TOptimizerStatistics>> field.
Fails: TTypeAnnotationContext lives in yql/essentials/core. Adding a field typed NKqp::TOptimizerStatistics would pull kqp/opt/cbo into the core YQL library — the exact dependency inversion the CHECK_DEPENDENT_DIRS guards prevent. yql/essentials/core must not know about KQP types.
✗ Store NYql::TOptimizerStatistics in both maps
Keep using TTypeAnnotationContext::StatisticsMap (typed NYql::TOptimizerStatistics) in the KQP CBO, converting at the end.
Defeats the fork: KQP-specific fields (ShuffleOrderingIdx, SortingOrderingIdx, ShuffledByColumns) added to NKqp::TOptimizerStatistics cannot exist in the YQL type. KQP would have no place to store them without modifying YQL's struct.
✗ Use std::any / type-erased slot in TTypeAnnotationContext
Add an std::any KqpExtension field to TTypeAnnotationContext that KQP fills with its own context object.
Loses compile-time type safety; every access requires std::any_cast and an awkward downcast. The extension slot pattern has the same ownership/layering problem: yql/essentials/core gains a runtime dependency on KQP code. Also hides the dependency, making it harder to enforce and audit.
TKqpStatsStore is a clean owned map that mirrors the YQL map's shape but stays fully inside the KQP dependency cone. Passing it alongside TTypeAnnotationContext* (for the few functions that still need YQL context for non-stats purposes) is explicit and auditable.

4. TOptimizerStatistics — Full Independence vs. Type Alias

core stats struct
NYql::TOptimizerStatistics (in yql_statistics.h) uses NYql::NDq::TJoinColumn, NYql::NDq::TOrderingsStateMachine::TLogicalOrderings, and NYql::IProviderStatistics. If kqp_statistics.h simply included yql_statistics.h and aliased the type, the KQP CBO library would transitively depend on yql/essentials/core/cbo (because yql_statistics.h includes cbo_interesting_orderings.h), violating CHECK_DEPENDENT_DIRS.
✓ CHOSEN — Full independent struct
Duplicate TOptimizerStatistics field-by-field in NKikimr::NKqp, using the fork's own TJoinColumn and TOrderingsStateMachine. Add KQP-specific fields (SortingOrderingIdx, ShuffledByColumns) that have no equivalent in the YQL type.
No include dependency on yql_statistics.h from kqp/opt/cbo; KQP can extend the struct freely; no ODR issues.
✗ Type alias using TOptimizerStatistics = NYql::TOptimizerStatistics
Keep it as a pure alias. All code that uses NKqp::TOptimizerStatistics is just sugar over the YQL type.
An alias still requires #include <yql/essentials/core/yql_statistics.h> to be reachable, which includes yql/essentials/core/cbo/cbo_interesting_orderings.h. The CHECK_DEPENDENT_DIRS violation happens at include-time, not at type-reference time. Also, aliases don't allow adding KQP-specific fields.
✗ Keep NYql::NDq::TOrderingsStateMachine in NKqp::TOptimizerStatistics
Make the struct independent in most fields but keep LogicalOrderings typed as NYql::NDq::TOrderingsStateMachine::TLogicalOrderings.
Fails: TLogicalOrderings is a nested type of TOrderingsStateMachine, which lives in yql/essentials/core/cbo/cbo_interesting_orderings.h. Including that from kqp_statistics.h would be the exact CHECK_DEPENDENT_DIRS violation the guard prevents. The KQP fork of TOrderingsStateMachine must be used.
✗ Use void* / opaque handles for ordering state
Store ordering state as void* or shared_ptr<void> inside TOptimizerStatistics, cast at point of use.
Eliminates compile-time type safety across every ordering-related site. The FSM interaction (creating TLogicalOrderings, querying states, combining them) is pervasive in stat inference. Opaque handles would require casts at every site and hide the type relationship that CHECK_DEPENDENT_DIRS is supposed to enforce structurally.
The independent struct is necessary because the include graph constraint is real and structural, not stylistic. The field-by-field duplication is a small one-time cost that enables KQP to add fields like SortingOrderingIdx and ShuffledByColumns without asking the YQL team for permission.

5. EJoinAlgoType — Duplicate Enum vs. Shared Reference

enum independence
NYql::EJoinAlgoType lives in yql/essentials/core/yql_cost_function.h. The fork's cbo_optimizer_new.h and TJoinAlgoHints need a join-algorithm enum. Including yql_cost_function.h from ydb/core/kqp/opt/cbo would require the CBO library to depend on yql/essentials/core — currently it only depends on yql/essentials/ast and utilities, keeping it lightweight.
✓ CHOSEN — Duplicate enum with identical integer values
Define NKikimr::NKqp::EJoinAlgoType with the same enumerator names and same implicit integer values as NYql::EJoinAlgoType. At the one boundary point (TEquiJoinLinkSettings::JoinAlgo), use static_cast.
No new includes in the CBO library. static_cast is zero-cost. Identical integer values mean the cast is semantically correct. Enum names match, so FromString/ToString work the same way.
✗ Include yql_cost_function.h in the CBO library
Just use NYql::EJoinAlgoType everywhere in the fork. Don't duplicate the enum.
Adds yql/essentials/core (and its transitive deps including yql/essentials/core/cbo) to the CBO library's dependency cone. Breaks CHECK_DEPENDENT_DIRS. The CBO library would no longer be lightweight or independently testable.
✗ Type alias using EJoinAlgoType = NYql::EJoinAlgoType
Alias the enum in the fork's namespace. Same effect as the duplicate, but declared as an alias.
An alias still requires #include <yql_cost_function.h> to reach the original definition. The include-path problem is not solved by aliasing — the build system violations happen at the file inclusion level, not at the declaration level.
✗ Use raw int / forward-declared opaque enum
Store join algorithm as an integer or a forward-declared opaque enum whose body is not included in the CBO headers.
Loses type safety at every call site that compares or switches on the enum. The forward-declaration trick for scoped enums requires knowing the underlying type; for unscoped enums it is not possible in standard C++. Makes all algorithm-selection code harder to read and validate.
Duplicating a seven-value enum is a trivial cost. The identical integer values make the static_cast at dq_opt_join.cpp:182–189 provably correct. An explicit comment in cbo_optimizer_new.h documents the invariant that the two enums must stay in sync.

6. TJoinColumn — Inline vs. Shared vs. Own Header

struct placement
NYql::NDq::TJoinColumn is defined in yql/essentials/core/yql_cost_function.h. It is used by TOrdering, TFunctionalDependency, and TFDStorage inside cbo_interesting_orderings.h. The fork needs its own NKqp::TJoinColumn, but yql_cost_function.h cannot be included from the CBO library.
✓ CHOSEN — Inline into cbo_interesting_orderings.h
TJoinColumn struct (with its THashFunction, operator==, operator<) is placed directly at the top of cbo_interesting_orderings.h, before TOrdering and other consumers.
TJoinColumn is tiny (5 fields, two free functions). The primary consumer is TOrdering in the same header. Inlining avoids creating a trivial single-purpose header.
✗ Separate kqp_join_column.h header
Put NKqp::TJoinColumn in its own header, include it from cbo_interesting_orderings.h.
Reasonable but adds a file for a struct that is only ~30 lines. The struct has no other independent consumers within the CBO library; every user of TJoinColumn also uses TOrdering or TFDStorage. One extra indirection for no gain.
✗ Use NYql::NDq::TJoinColumn directly
Keep a PEERDIR on yql/essentials/core from kqp/opt/cbo, use the original struct.
The current kqp/opt/cbo/ya.make deliberately avoids depending on yql/essentials/core (which transitively pulls yql/essentials/core/cbo). Adding that PEERDIR would break CHECK_DEPENDENT_DIRS. It would also mean TOptimizerStatistics::TKeyColumns::ToJoinColumns() returns NYql::NDq::TJoinColumn, mixing types.
✗ Type alias within cbo_interesting_orderings.h
using TJoinColumn = NYql::NDq::TJoinColumn; in the NKikimr::NKqp namespace.
Same problem as before: the alias declaration needs NYql::NDq::TJoinColumn to be defined, which requires including yql_cost_function.h, which is the forbidden dependency. An alias does not change the include requirement.
Inlining TJoinColumn into cbo_interesting_orderings.h mirrors exactly how NYql::NDq::TJoinColumn ended up in yql_cost_function.h originally: as a supporting type for the orderings machinery. The placement is logical and needs no extra files.

7. FromYqlColumnStatMap Placement

boundary conversion locality
Somewhere, NYql::TOptimizerStatistics::TColumnStatMap (from yql_statistics.h) must be converted into NKqp::TOptimizerStatistics::TColumnStatMap (from kqp_statistics.h). The function's argument type forces whoever defines it to #include <yql/essentials/core/yql_statistics.h>. That header includes yql/essentials/core/cbo/cbo_interesting_orderings.h, so the defining file's library must be allowed to depend on yql/essentials/core/cbo.
✓ CHOSEN — File-scope static in kqp_statistics_transformer.cpp
Defined as a static function in the translation unit. kqp_statistics_transformer.cpp is part of ydb/core/kqp/opt (not kqp/opt/cbo), which is allowed to include both YQL and KQP headers. The function is only called from that file anyway.
The only caller is in the same file. static scope prevents accidental use elsewhere. The library constraint is satisfied because the TU lives outside the restricted kqp/opt/cbo boundary.
✗ Member/free function in kqp_statistics.h/cpp
Declare the converter in the CBO library's stats header, so it lives next to the type it creates.
Fails: kqp_statistics.h would need #include <yql/essentials/core/yql_statistics.h> for the argument type. yql_statistics.h includes yql/essentials/core/cbo/cbo_interesting_orderings.h. This violates CHECK_DEPENDENT_DIRS(DENY PEERDIRS yql/essentials/core/cbo) in kqp/opt/cbo/ya.make. The build fails.
✗ Separate boundary-conversion library
Create ydb/core/kqp/opt/cbo/boundary/ with a new ya.make that is allowed to PEERDIR on both sides.
Over-engineering one function. A dedicated library for a 15-line static function adds a new ya.make, a new directory, and an extra PEERDIR chain for every future consumer. The transformer file is already the only caller; moving the definition there is simpler and more auditable.
✗ Store NYql::TColumnStatistics directly (no conversion)
Make NKqp::TOptimizerStatistics::TColumnStatMap store NYql::TColumnStatistics instead of duplicating it.
Same include-graph problem: kqp_statistics.h would need to reach NYql::TColumnStatistics, which lives in yql_statistics.h, which pulls yql/essentials/core/cbo. Also, if KQP ever needs to add column stat fields, it could not without modifying the shared YQL type.
The placement in kqp_statistics_transformer.cpp is the minimum-invasive solution: the function is private to its only caller, no new files are needed, and the include constraint is respected by the file's library affiliation rather than by restructuring the code.

8. ODR Conflict Prevention — Different Namespaces

symbol identity
Both NYql::NDq::DqOptimizeEquiJoinWithCosts (original) and the fork's version of the same function will be compiled and linked into the same ydbd binary. Under the C++ One Definition Rule, two definitions of a function with the same mangled name in the same translation unit are illegal (and in different TUs they must be identical). The functions are intentionally different, so they must have different mangled names.
✓ CHOSEN — Different top-level namespaces
Fork lives in NKikimr::NKqp, original stays in NYql::NDq. The mangled names are different by construction. No runtime mechanism needed.
Zero runtime cost; compile-time enforcement; human-readable in stack traces and diagnostics. Namespacing is the standard C++ mechanism for this.
✗ Inline namespace versioning (NYql::NDq::V2)
Put the fork into inline namespace V2 inside NYql::NDq. Unversioned references in consumer code resolve to V2 by default.
Inline namespaces are designed for ABI versioning of a single evolving library, not for two independent parallel implementations. Having V2 inside NYql::NDq would still give the fork a logical home in the YQL library, blurring ownership. Also, both V1 and V2 would still be reachable from the same namespace — consumer code could accidentally call the wrong version.
✗ Preprocessor #ifdef guards
Use a build-system-controlled macro to include either the YQL or the KQP version of the CBO headers, but not both.
Both CBO implementations are used simultaneously in the same binary (YQL-path queries use NYql::NDq; KQP-path queries use NKikimr::NKqp). A compile-time switch would exclude one entirely. This makes it impossible to run both code paths in the same binary, which is required during the transition period.
✗ Weak symbol linking tricks
Mark one definition as __attribute__((weak)) so the linker prefers the strong one.
Dangerous: weak symbols mean one implementation silently shadows the other. Which one wins depends on link order. Both implementations are correct for their own callers; neither should ever shadow the other. This also does not work portably on all platforms/toolchains the build supports.
Different namespaces are the only mechanism that is both zero-cost and allows both implementations to coexist in the same binary at full strength. The namespace choice also communicates ownership clearly: NYql::NDq is owned by the YQL team, NKikimr::NKqp is owned by the KQP team.

9. dq_opt_join.cpp Namespace Strategy

namespace vs. include plumbing
dq_opt_join.cpp has ~600 lines that use many NYql::NDq types (nodes, expression builders, graph matchers) internally, but its exported functions (DqRewriteEquiJoin, DqBuildPhyJoin, …) must live in NKikimr::NKqp to avoid ODR conflict. If the file simply opens namespace NKikimr::NKqp { ... }, every reference to an NDq type must be re-qualified.
✓ CHOSEN — namespace NKikimr::NKqp + using namespace NYql::NDq inside
The file wraps everything in namespace NKikimr::NKqp { ... } and immediately follows with using namespace NYql::NDq;. All NDq types are visible unqualified inside; exported symbols mangle as NKikimr::NKqp.
Minimal change to the file body: only the namespace wrapper and the using declaration change from the original. Avoids re-qualifying hundreds of NDq:: references throughout the file.
✗ Keep namespace NYql::NDq, rename functions
Leave the original namespace, rename the exported functions (KqpRewriteEquiJoin instead of DqRewriteEquiJoin), update all callers.
Renames proliferate through every KQP call site and break the diff-based comparison with the YQL original. The diff_copies.sh normalization would need to handle function renames in addition to namespace renames. The ODR problem is only solved if the original is not also linked — but it is.
✗ Wrapper file that delegates to the original
Add a new dq_opt_join_kqp.cpp in NKikimr::NKqp that calls the original NYql::NDq::DqRewriteEquiJoin, adapting the TKqpStatsStore argument.
The function body accesses the stats store in ~30 different places. A wrapper would need to either: (a) convert TKqpStatsStore into a TTypeAnnotationContext then convert back (expensive, lossy), or (b) rewrite the entire function body anyway. Neither is a genuine delegation.
✗ Re-qualify every NDq:: reference in the file
Open namespace NKikimr::NKqp but do not add using namespace NYql::NDq. Manually qualify every node matcher, expression builder, and helper.
Makes the file's diff against the YQL original enormous and unreadable, hiding the actual semantic changes. Every added NYql::NDq:: prefix is noise. The file is already ~600 lines; qualifying every reference multiplies its size with no information gain.
The using namespace NYql::NDq inside the outer NKikimr::NKqp block is the minimal diff strategy: the exported symbols get the new mangled names, and the file body is nearly identical to the YQL original, keeping the diff reviewable and merge-friendly.

10. PG Optimizer — Y_ABORT vs. Keep

feature scope
dq_opt_join_cbo_factory.cpp (in both the YQL original and the fork) has a MakeJoinCostBasedOptimizerPG path that calls MakePgOptimizerNew() from yql/essentials/parser/pg_wrapper. KQP does not have a PG optimizer. Keeping the call requires a PEERDIR on pg_wrapper, which adds a substantial dependency and a path that can never be exercised.
✓ CHOSEN — Y_ABORT with explanatory message
Replace the PG path with Y_ABORT("PG optimizer is not supported in the ydb-owned CBO"). Remove the pg_wrapper/interface/optimizer.h include entirely.
Makes the unsupported path loudly visible instead of silently unreachable. Removes the pg_wrapper dependency from the fork. If someone accidentally routes a PG query through the KQP CBO, they get an immediate actionable error rather than undefined behaviour.
✗ Keep the PG path with the PEERDIR
Add yql/essentials/parser/pg_wrapper/interface to solver/ya.make and keep the code unchanged.
Adds a large transitive dependency for dead code. The PG optimizer factory will never be invoked from KQP. Dead code with live dependencies increases compile time, link size, and obscures the intent of the fork.
✗ Return nullptr silently
Replace the body with return nullptr;. Any caller would get a null pointer and likely crash later with a worse diagnostic.
Null pointer dereferences produce crashes far from the unsupported call site, making root-cause analysis hard. Y_ABORT at the entry point is strictly better: same crash, better message, correct location in the stack trace.
✗ Delete the function entirely
Remove MakeJoinCostBasedOptimizerPG from the fork's IOptimizerFactory.
The function is part of the IOptimizerFactory interface. Removing it changes the interface contract, making the fork's factory a strict subset of the YQL one. Code that stores an IOptimizerFactory* and calls the PG method would fail to compile even on code paths that never reach the KQP fork. Keeping the signature but aborting is safer.
Y_ABORT is the correct tool for "this code path exists in the interface but is intentionally not supported here". It preserves the interface contract, produces a clear diagnostic, and removes the unnecessary dependency.

11. IProviderStatistics for S3 — Dual Inheritance

provider stats boundary
TOptimizerStatistics::Specific stores provider-specific stats as a shared_ptr<IProviderStatistics>. The S3 provider creates NYql::TS3ProviderStatistics objects and stores them through the YQL boundary. After conversion to NKqp::TOptimizerStatistics, KQP code that wants to inspect S3-specific stats calls dynamic_cast<NYql::TS3ProviderStatistics*>(stats.Specific.get()). The problem: if Specific is typed NKqp::IProviderStatistics*, the cast requires NKqp::IProviderStatistics and NYql::TS3ProviderStatistics to share a virtual base.
✓ CHOSEN — TKqpS3ProviderStatistics with dual inheritance
struct TKqpS3ProviderStatistics : public NKqp::IProviderStatistics, public NYql::TS3ProviderStatistics. The S3 stats object is both, so it can be stored as the KQP base type and cast to the YQL concrete type.
Zero overhead vs. a straight pointer. The dynamic_cast chain works because the object is actually both types simultaneously. No conversion or copying of S3 data.
✗ Type alias IProviderStatistics = NYql::IProviderStatistics
Make NKqp::IProviderStatistics a pure alias for the YQL type. No wrapper needed; TS3ProviderStatistics already inherits from it.
Reasonable; this alias was present in an earlier version of the fork. Reverted because it creates a direct include dependency on yql_statistics.h from kqp_statistics.h (the same chain that violates CHECK_DEPENDENT_DIRS). Dual inheritance keeps the KQP type truly independent.
✗ Convert NYql::TS3ProviderStatistics at the boundary
Create a NKqp::TS3ProviderStatistics that copies all fields from the YQL version. Store the KQP copy in Specific.
Requires duplicating the S3 statistics struct (which may grow). Every time NYql::TS3ProviderStatistics gains a new field, the copy function silently drops it. Dual inheritance avoids this by reusing the original type without copying.
✗ Store shared_ptr<void> and cast directly
Make Specific a shared_ptr<void>. Any provider-specific type can be stored; callers cast to whatever they expect.
Completely type-unsafe. There is no way to know at compile time what type is actually stored, and a mis-cast gives undefined behaviour without any diagnostic. The virtual base approach keeps type safety via dynamic_cast and the vtable.
Dual inheritance is an unusual but correct use of C++ multiple inheritance: the object is both types. It is contained in kqp_statistics_transformer.cpp (the only creation site), so the unusual pattern is localised and documented.

12. Solver Cannot Deny yql/essentials/core/cbo

transitive dependency ceiling
Ideally the solver library (ydb/core/kqp/opt/cbo/solver) should also have CHECK_DEPENDENT_DIRS(DENY yql/essentials/core/cbo), like kqp/opt/cbo does. But the solver depends on yql/essentials/core (for TTypeAnnotationContext, TExprNode, etc.), and that library depends on yql/essentials/minikql which depends on yql/essentials/parser/pg_wrapper/interface which depends on yql/essentials/core/cbo. The transitive chain makes the denial impossible to satisfy.
✓ CHOSEN — Accept solver as intentional bridge layer; no denial
The CHECK_DEPENDENT_DIRS denial is placed on the inner kqp/opt/cbo library (pure types, no yql/essentials/core dep). The solver is the bridge: it uses both KQP types and YQL infrastructure.
The solver's use of yql/essentials/core/cbo is unavoidable via the transitive chain. Documenting this explicitly is more honest than a guard that would fail on the first build.
✗ Remove yql/essentials/core dep from solver
Eliminate all solver uses of TTypeAnnotationContext. Functions that still need it accept a void pointer or a minimal interface instead.
Significant refactor: InferStatisticsForExtendBase, InferStatisticsForAggregateBase, InferStatisticsForTopBase, and others need TTypeAnnotationContext for orderings FSM data (non-stats uses). Removing this dependency would require either duplicating more of TTypeAnnotationContext into a KQP-owned struct, or passing the FSM data through many extra parameters. The marginal gain (one more denial) does not justify that refactor.
✗ Break the transitive chain upstream
Remove the dep from minikql → pg_wrapper/interface → core/cbo, or from yql/essentials/core → minikql.
These dependencies are structural in the YQL build graph and serve other purposes. Removing them would affect all consumers of those libraries, not just this fork. This is not a change that the KQP team can make unilaterally.
✗ Add the denial and suppress it with ALLOW_DEPENDENT_DIRS
Keep the CHECK_DEPENDENT_DIRS denial in the solver but add an explicit ALLOW exception for the transitive path.
Defeats the purpose of the guard: a denial with a blanket exception for the exact dependency being denied is meaningless. The exception would need to list yql/essentials/core/cbo, which is the forbidden dependency. Better to document the reason in a comment and omit the guard.
The solver's transitive exposure to yql/essentials/core/cbo is structural and cannot be removed without a major refactor of the YQL build graph. The guard on kqp/opt/cbo (the pure-types library) provides the meaningful protection: it ensures the CBO types themselves are defined independently of the YQL CBO types. The solver is documented as the intentional bridge layer.
