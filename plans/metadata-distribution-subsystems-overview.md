# План: обзорная статья «Подсистемы распространения метаданных» (StateStorage, Board, SchemeBoard)

## Цель

Написать обзорную **концептуальную** статью для **пользователей, техписателей и администраторов БД**, которая объясняет:

- что такое подсистемы распространения метаданных в кластере {{ ydb-short-name }} и зачем они нужны;
- ключевые понятия: **реплика**, **кольцо**, **группа колец**;
- принцип работы (распределённый кворумный сервис, выборка реплик по хешу id, отказоустойчивость, обслуживание стоек и роллинг-рестарты);

Статья НЕ описывает код, НЕ содержит ссылок/вставок кода и НЕ дублирует инструкции по конфигурированию кластера (они уже есть в разделах devops/configuration-management).

## Структура статьи (черновик содержания)

1. **Введение** — что такое метаданные кластера и зачем их распространять; общая идея трёх подсистем как распределённых кворумных сервисов.
2. **Общий принцип работы** (единый для всех трёх подсистем):
   - распределённый сервис на множестве нод; запись адресуется по id (например, TabletId);
   - детерминированная выборка реплик по хеш-функции от id — по id всегда вычисляется набор хранящих реплик;
   - кворум: достаточно ответа большинства из выбранных реплик (`nto_select/2 + 1`);
   - следствие: безопасный вывод нод на обслуживание и роллинг-рестарты.
3. **Ключевые понятия**:
   - **Реплика** — актор на отдельной ноде, хранящий метаданные и отвечающий на чтение/запись.
   - **Кольцо** — набор реплик для распределения нагрузки по нодам; из каждой записи в кольце выбирается одна реплика; реплики кольца по возможности в одной стойке; нельзя смешивать реплики разных колец одной подсистемы в одной стойке — это позволяет выводить стойку целиком без потери отказоустойчивости.
   - **Группа колец** — для каждой группы собирается отдельный кворум; нужна для бесшовной смены конфигурации.
4. **Группы колец и смена конфигурации**:
   - бесшовное добавление/удаление групп; свойство `write_only` (новая/удаляемая группа не участвует в кворуме на чтение);
   - автоматическая реконфигурация (Self Heal) и ручная — ссылки на существующие разделы;
   - применение в конфигурации 2 ДЦ / bridge — отдельная группа колец на каждый ДЦ (pile), быстрое переключение primary.
5. **Отказоустойчивость и модель отказа** — связь с доменами/областями отказа, обслуживание стоек.
6. **Подсистемы**:
   - **StateStorage** — разрешение имён таблеток (ActorId по TabletId), хранение лидера/поколения/шага, участие в выборе лидера таблетки; волатильность данных.
   - **Board** — распространение key-value метаданных, в т.ч. информация об endpoint-ах; модель подписок/lookup.
   - **SchemeBoard** — распространение схемных метаданных, подписки на изменения схемы.
   - Краткая сравнительная таблица: назначение, тип данных, типичные потребители.
7. **Связанные материалы** — ссылки на reconfiguration, self-heal, bridge, topology, glossary.
