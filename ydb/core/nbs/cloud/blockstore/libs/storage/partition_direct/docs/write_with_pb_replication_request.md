# Запись в режиме репликации persistent buffer

Документ описывает логику работы записи блоков с использованием встроенной репликации persistent buffer (PB). Реализация: `TWriteWithPbReplicationRequestExecutor` в `partition_direct`.

## Идея

Вместо того, чтобы клиент сам рассылал запись на N PB-хостов, он отправляет один запрос на один PB-хост — **координатор**, а тот сам реплицирует данные на остальные. Преимущества:

- полезная нагрузка уходит с клиента по сети один раз вместо N;
- координация межрепликной доставки выполняется на стороне PB.

При сбоях или замедлениях PB-репликации запрос откатывается на классический путь: прямая рассылка одиночных записей на резервные хосты с механикой повторов и hedge.

## Конфигурируемые параметры
- ```pbuffer_reply_timeout_microseconds``` - время, через которое pbuffer должен предоставить ответ, даже если не все участвующие пбуфферы вернут свои ответы.
- ```write_hedging_delay``` - время, через которое будут отправлены одиночные hedge запросы. Hedge запросы имеют смысл только в случае зависших/потерянных основных запросов, поэтому его нужно выставлять чуть большим, чем pbuffer_reply_timeout_microseconds.
- ```write_request_timeout``` - максимальное время выполнения запроса. По его истечении будет возвращена ошибка.

## Конфигурация хостов

vchunk различает две группы PB-хостов:

- **Desired PB** — основные хосты, на которые идёт первичная рассылка через репликацию.
- **Handoff PB** — резерв; используются как приоритетные кандидаты для прямых записей при сбоях.

Множество хостов, доступных для прямых записей, инициализируется как объединение Desired и Handoff и сокращается по мере получения ответов и отправки прямых запросов.

## Кворум

Запрос считается успешно выполненным, как только подтверждено достаточное количество хостов (кворум). Параллельно с PB-репликацией и/или прямыми записями исполнитель отслеживает:

- множество хостов с подтверждённой записью (Completed);
- множество хостов, на которые сейчас отправлены и ещё не вернулись прямые запросы (in-flight);
- множество хостов, ещё доступных для прямых записей.

Решение «слать ещё или ждать» основано на сумме Completed и in-flight: исполнитель не отправляет больше прямых запросов, чем нужно для кворума.

## Поток выполнения

1. **Старт.** Планируются общий request-timeout и hedge-таймер. Отправляется PB-репликация по списку Desired PB.
2. **PB-репликация.** Из списка Desired выбирается координатор. Координатор сразу исключается из множества доступных для прямых записей: он уже взял на себя попытку. Все Desired-хосты помечаются как «запрошенные» (на них делегирована запись).
3. **Обработка ответа PB-репликации.** Ответ может прийти один или несколько раз: транспортный слой допускает многократный вызов колбэка по мере получения подтверждений от реплик.
   - Если пришла фатальная ошибка координатора — уход в добор прямыми записями.
   - Иначе для каждого хоста в ответе: исключаем его из доступных (вне зависимости от успеха), успешные добавляем в Completed.
   - При достижении кворума — финальный ответ клиенту.
   - Иначе — попытка добрать недостающее прямыми записями.
4. **Добор прямыми записями.** Проверяется необходимость отправки и доступность хостов:
   - если Completed + in-flight уже достаточно для кворума — ничего не делаем, ждём ответы;
   - если хостов в резерве не хватает — отвечаем клиенту ошибкой (кворум недостижим);
   - иначе выбираем недостающее число хостов с приоритетом Handoff PB и отправляем им одиночные записи.
5. **Обработка ответа прямой записи.** Хост снимается с in-flight. При успехе добавляется в Completed и, если набран кворум, отправляется финальный ответ. При ошибке — повторная попытка добора.
6. **Hedge.** По истечении `HedgingDelay` (если клиенту ещё не ответили) запускается тот же механизм добора прямыми записями параллельно ещё не вернувшейся PB-репликации.

## Условия завершения

| Событие | Результат |
|---|---|
| Набран кворум подтверждений | `S_OK` |
| Доступных хостов меньше, чем нужно для добора | `E_FAIL` («Direct additional requests are not available») |
| Сработал общий request-timeout | E_TIMEOUT («Write request timeout») |
| Не смогли доставить сообщение | ERROR («Undelivered») |

## Уровень выше: vchunk

vchunk создаёт исполнитель на каждый запрос записи и передаёт ему `TWriteRequestBundle` — общий объект-«пакет», который связывает исполнитель с vchunk-ом (через слабую ссылку на `IWriteClient`) и хранит контекст запроса (range, Lsn, sglist, Wilson-span, promise клиенту партиции).

У бандла два публичных метода доставки результата:

- `Reply` — **финальный** ответ. Бандл закрывает sglist и вызывает обработку ответа на уровне vchunk, передавая итоговый статус, набор «запрошенных» хостов и набор подтвердивших на момент ответа.
- `NotifyBelated` — **запоздавшее** уведомление. Бандл вызывает обработку "запоздавших" ответов с **дельтой** хостов, подтвердивших запись уже после финального ответа.

Разделение обеспечивается базовой логикой исполнителя: первое срабатывание `ReplyOrNotifyBelated` идёт по `Reply`, последующие — по `NotifyBelated`. Если к моменту срабатывания vchunk уже уничтожен (слабая ссылка не разлочивается), `Reply` всё равно отдаёт promise клиенту партиции с ошибкой `E_CANCELLED`, а `NotifyBelated` молча игнорируется.

### Финальный ответ

При получении финального ответа vchunk в `OnWriteBlocksResponse`:

- декрементирует счётчик inflight-записей;
- обновляет dirty-map: фиксирует «запрошенные» и «подтверждённые» хосты для записанного диапазона блоков по этому Lsn;
- обновляет счётчики операций;
- отдаёт результат клиенту партиции через `Bundle->SendFinalReply` (promise);
- триггерит фоновые операции наа flush/erase.

Применение `RequestedWrites`/`CompletedWrites` к dirty-map происходит один раз — при финальном ответе.

### Запоздавшие подтверждения

Назначение belated-нотификаций — не «дополнительный отчёт клиенту», а корректное управление состоянием PB. Финальный ответ клиенту мог уйти раньше, чем подтвердились все запрошенные реплики (например, кворум собрался раньше). Запоздалое подтверждение от ещё одной реплики означает, что данные на ней теперь тоже лежат и соответствующий PB-слот нужно будет очистить.

В обработке "запоздавших" ответов vchunk вызывает обновление соответствующей очереди (`ReadyToEraseBelated`) и иницирует erase.

Внутри механизма её обновления решение принимается по состоянию inflight-записи для этого Lsn:
- **Lsn неизвестен** (запись уже полностью отработана и удалена из inflight) — хост попадает в `ReadyToEraseBelated`. Из этой очереди его обработает отдельная подсистема belated-erase в фоне.
- **По данному Lsn уже идёт или завершён erase PB** (состояние `PBufferErasing` или `PBufferErased`). Стандартная фаза для этого Lsn уже отыграна и хост туда уже не «впишешь» — он попадает в **belated erase queue** (`ReadyToEraseBelated`).
- **Запись ещё активна в обычной фазе** (`PBufferIncompleteWrite`/`PBufferWritten`/`PBufferFlushing`/`PBufferFlushed`). В belated erase queue хост **не** добавляется — стандартная волна erase для этого Lsn покроет его сама, когда дойдёт до соответствующего состояния.

Без этого механизма «опоздавшие» хосты на завершённых Lsn остались бы с непрочищенным PB: vchunk бы не знал, что они тоже содержат свежие данные, и стандартная волна erase для этого Lsn их бы уже не покрыла.

Запоздавшие подтверждения могут прийти как на дополнительные ответы от `WriteToManyPBuffers`, так и от запоздалых одиночных прямых записей в один PB.

## Уровень ниже: ic_storage_transport

PB-репликация на транспортном уровне — это **один логический запрос**, независимо от числа PB-хостов. Координация реплик идёт внутри PB-координатора; со стороны клиента отправляется одно сообщение и регистрируется один колбэк.

### Карта запросов

Транспортный актор хранит таблицу активных PB-репликационных запросов. Для каждого запроса фиксируется:

- сам запрос с пользовательским колбэком;
- множество PB-id, от которых ещё не получены подтверждения (`WaitingReplies`).

Это множество служит счётчиком незакрытых хостов на уровне транспорта. Пока оно непусто, запись удерживается в таблице, а колбэк остаётся живым.

### Контракт колбэка

Колбэк PB-репликации может вызываться **несколько раз** на один запрос. Транспорт допускает доставку частичных ответов: PB-координатор может прислать подтверждения от части реплик раньше, чем от остальных. Каждый частичный ответ содержит результаты от подмножества PB.

Именно этот контракт определяет идемпотентную форму обработки ответа в исполнителе: маски обновляются постепенно, кворум проверяется после каждого пакета, и итоговый сценарий «один Reply + несколько NotifyBelated» — это естественное следствие,
а не аномалия.

### Обработка ответа

При каждом приходящем ответе транспорт:

- вызывает зарегистрированный колбэк с полученными данными;
- удаляет подтвердившие PB-id из `WaitingReplies`;
- если `WaitingReplies` опустел — удаляет запись из таблицы.

При отказе на этапе захвата данных (sglist уже закрыт клиентом) формируется синтетический ERROR-ответ, отправляемый по тому же пути — это унифицирует обработку и исключает специальные ветки.

При уничтожении транспортного актора все ещё активные запросы получают синтетический ERROR на все оставшиеся PB-id, чтобы исполнитель гарантированно получил финальный сигнал и не завис.

### Таймауты

Транспортный актор не контролирует таймаутами сами по себе. Таймауты живут в двух местах:

- на стороне PB-координатора — туда явно передаётся `PbufferReplyTimeout`;
- на стороне исполнителя — общий `RequestTimeout`.

Если один из этих таймаутов сработают раньше реального ответа, исполнитель продолжит свою работу в соответствии с логиой таймаута, но запись в таблице транспорта останется до прихода реального ответа или уничтожения актора. Поздний ответ в этом случае дойдёт до исполнителя через колбэк и обработается как обычный ответ — частично пополнит Completed и (если уже было `Reply`) пройдёт через путь belated.
