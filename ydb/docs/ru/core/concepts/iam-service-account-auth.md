# IAM-аутентификация с делегированием сервисному аккаунту

Стандартная аутентификация по IAM-токену не подходит для долгосрочных операций — облачные токены имеют ограниченный срок действия, и при их истечении операция требует перезапуска. Для таких случаев {{ ydb-full-name }} поддерживает IAM-аутентификацию с делегированием сервисному аккаунту.

Эта схема используется в:

- [Асинхронной репликации](async-replication.md) и [трансфере данных](transfer.md) между базами.
- [Внешних источниках данных {{ ydb-short-name }}](query_execution/federated_query/ydb.md#iam-auth) для долгосрочных запросов.

## Как работает {#how-it-works}

При создании ресурса указывается начальный IAM-токен в параметре `INITIAL_TOKEN_SECRET_PATH`. Этот токен используется один раз — для проверки доступности целевой базы данных и определения её идентификатора ресурса в облаке — и может истечь к моменту дальнейшего использования ресурса. Дальнейшая аутентификация выполняется сервисным аккаунтом, указанным в `SERVICE_ACCOUNT_ID`, чьи токены обновляются автоматически. Это позволяет долгосрочным операциям работать бесконечно долго без перезапуска.

## Подготовка сервисного аккаунта {#service-account}

В облаке необходимо подготовить сервисный аккаунт, которому будут делегированы права доступа к целевой базе:

1. Назначьте сервисному аккаунту нужные роли на целевой базе данных. Конкретный набор ролей определяется сценарием использования и описан в документации соответствующей фичи.
2. Включите [делегирование сервисного аккаунта](https://yandex.cloud/ru/docs/iam/concepts/service-control) для целевой базы данных.

## Параметры {#parameters}

- `SERVICE_ACCOUNT_ID` — идентификатор подготовленного сервисного аккаунта.
- `INITIAL_TOKEN_SECRET_PATH` — имя [секрета](datamodel/secrets.md) с начальным IAM-токеном. Получить токен можно командой `yc iam create-token`.

После успешного создания ресурса обновлять IAM-токен в секрете не требуется — дальнейшая аутентификация выполняется сервисным аккаунтом автоматически.
