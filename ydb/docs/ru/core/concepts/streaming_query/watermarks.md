# Водяные знаки

Водяной знак (watermark) в [потоковых запросах](index.md) YDB представляет собой специальное событие, которое несет нижнюю оценку времени ещё не обработанных данных. Концепция заимствована из модели [stream processing](https://en.wikipedia.org/wiki/Stream_processing) и позволяет системе определять, когда все данные для заданного временного окна были получены.

Водяной знак со значением времени X означает, что все входные данные с временем события меньше X были получены. Это позволяет, например, функции [HoppingWindow](../../yql/reference/syntax/select/group-by.md#hopping_window) закрыть все завершённые окна и вернуть результат пользователю.

## Время события

В системе потоковой обработки важно знать, что стоит за понятием *времени события*, например, при работе с окнами [HoppingWindow](../../yql/reference/syntax/select/group-by.md#hopping_window). Сейчас в YDB в качестве такой временной метки выступает время записи события в [топик](../datamodel/topic.md) (`write_time`). Это значение присваивается системой автоматически в момент записи сообщения и не зависит от содержимого сообщения.

## Водяной знак

### Отставание водяного знака

В условиях распределенных систем, когда часы на разных устройствах могут дрейфовать, а данные могут задерживаться из-за проблем с сетью, водяной знак не может полагаться на монотонное возрастание времени событий даже в рамках одной партиции. Для этого в формулу расчёта водяного знака закладывается отставание.

Значение отставания следует выбирать исходя из максимально допустимой задержки доставки сообщений. Слишком малое значение приведёт к потере опоздавших событий, слишком большое увеличит задержку выдачи результатов.

### Использование

Для включения механизма водяных знаков необходимо указать выражение для вычисления водяного знака в секции [WITH](../../yql/reference/syntax/select/with.md). Сейчас поддерживается только время записи в [топик](../datamodel/topic.md) с константной задержкой. Это временное ограничение текущей реализации. Более тонкая настройка параметров водяного знака может быть произведена также в секции [WITH](../../yql/reference/syntax/select/with.md).

При использовании [HoppingWindow](../../yql/reference/syntax/select/group-by.md#hopping_window) необходимо корректно указать первый параметр, **time extractor**. В текущей реализации это должно быть только время записи события в [топик](../datamodel/topic.md). Для доступа к этому значению используется функция `SystemMetadata("write_time")`, которая возвращает системное время записи сообщения в топик.

### Пример

Входными данными для примера являются:

```json
{"pass": 1, "payload": "a"} // 1970-01-01T00:00:40Z
{"pass": 1, "payload": "b"} // 1970-01-01T00:00:42Z
{"pass": 0, "payload": "c"} // 1970-01-01T00:00:50Z
{"pass": 1, "payload": "d"} // 1970-01-01T00:00:40Z
```

Пусть для примера каждое событие было записано в топик во время, указанное в комментарии (это значение `write_time`, присвоенное системой)

Тело запроса создаёт [потоковый запрос](../../yql/reference/syntax/create-streaming-query.md), который читает данные из топика, фильтрует их и агрегирует с помощью оконной функции [HoppingWindow](../../yql/reference/syntax/select/group-by.md#hopping_window):

```sql
CREATE STREAMING QUERY example AS
DO BEGIN
    $input =
        SELECT
            t.*,
            SystemMetadata("write_time") AS ts
        FROM
            input_topic
        WITH (
            FORMAT = json_each_row,
            SCHEMA = (
                pass Int64,
                payload String
            ),
            WATERMARK = SystemMetadata("write_time") - Interval("PT5S")
            -- , WATERMARK_ADJUST_LATE_EVENTS
        ) AS t;

    SELECT
        AGGREGATE_LIST(payload) AS result,
        HOP_END() AS ts
    FROM
        $input
    WHERE pass > 0
    GROUP BY
        HoppingWindow(ts, "PT5S", "PT10S");
END DO;
```

В примере используются:
* [`CREATE STREAMING QUERY`](../../yql/reference/syntax/create-streaming-query.md) создаёт потоковый запрос;
* `SystemMetadata("write_time")` возвращает время записи сообщения в топик;
* `WATERMARK = SystemMetadata("write_time") - Interval("PT5S")` задаёт водяной знак с отставанием 5 секунд (подробнее о формате интервалов см. [литералы простых типов](../../yql/reference/builtins/basic.md#data-type-literals));
* [`AGGREGATE_LIST`](../../yql/reference/builtins/aggregation.md#agg-list) собирает значения в список;
* [`HOP_END()`](../../yql/reference/syntax/select/group-by.md#hopping_window) возвращает временную метку конца текущего окна;
* [`HoppingWindow`](../../yql/reference/syntax/select/group-by.md#hopping_window) задаёт скользящее окно с шагом 5 секунд и размером 10 секунд.

Результат:

```json
{"result": ["a", "b"], "ts": 45}
```

Значение `"ts": 45` означает 45 секунд от начала эпохи Unix (1970-01-01T00:00:45Z), что соответствует концу окна `[35; 45)`.

При параметрах `HoppingWindow(ts, "PT5S", "PT10S")` (hop = 5с, interval = 10с) формируются окна `[30; 40)`, `[35; 45)`, `[40; 50)` и т.д. (подробнее об алгоритме построения окон см. [HoppingWindow](../../yql/reference/syntax/select/group-by.md#hopping_window)).

```text
Время:    30    35    40    42    45    50
          |     |     |     |     |     |
Окно 1:   [─────────────────)
          [30;          40)
Окно 2:         [─────────────────)
                [35;          45)  ← закрыто watermark=45
Окно 3:               [─────────────────)
                       [40;          50)

События:               a↑   b↑          c↑   d↑(40, опоздало)
                       40   42          50   40
```

Пояснение:

1. Первое событие порождает водяной знак, равный 35 секундам (write_time = 40с, watermark = 40 − 5 = 35с). Первое событие проходит сквозь фильтр и "застревает" в HoppingWindow. Водяной знак, порожденный первым событием, не оказывает никакого влияния;
2. Второе событие порождает водяной знак, равный 37 секундам. Далее аналогично первому событию;
3. Третье событие порождает водяной знак, равный 45 секундам. Третье событие отбрасывается на фильтре. Водяной знак, порожденный третьим событием, закрывает окно `[35; 45)`, результат отдается пользователю;
4. Четвертое событие порождает водяной знак, равный 35 секундам. Четвертое событие проходит сквозь фильтр и отбрасывается в HoppingWindow как опоздавшее событие, так как оно пришло после водяного знака со значением 45;
    - Чтобы не отбрасывать опоздавшие события, нужно раскомментировать строку с опцией `WATERMARK_ADJUST_LATE_EVENTS` в секции `WITH`. В случае включения опции четвертое событие было бы учтено в окне `[40; 50)`.

### Поведение при перезапуске

При перезапуске потокового запроса значение водяного знака восстанавливается из [чекпоинта](checkpoints.md). Если чекпоинт отсутствует, водяной знак вычисляется заново по поступающим данным.
