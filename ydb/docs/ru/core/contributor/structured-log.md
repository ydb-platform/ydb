# Структурированное логирование
## Назначение
Структурированное логирование предназначено для того, чтобы в логе (журнале работы) отделить текстовое описание произошедших событий от параметров этих событий. Например, если произошла ошибка чтения файла, то её описание - это текст вида "Ошибка чтения файла", а параметры - это путь к файлу и код ошибки.

Такое разделение описания событий и их параметров впоследствии может быть использовано для эффективного поиска сообщений, сбора статистики, оптимизированного хранения и так далее.

## Запись сообщений в лог
## Базовые средства
Базовым средством для записи структурированных сообщений в журнал является макрос `YDB_LOG_CTX_COMP`. Он проверяет необходимость логирования (согласно текущим настройкам), формирует текстовое сообщение, прикрепляет к нему структурированные значения и затем отправляет всю эту информацию актору логирования, а он обрабатывает её согласно своим текущим настройкам (например, записывает в файл).

Полный синтаксис макроса логирования выглядит следующим образом:

```
YDB_LOG_CTX_COMP(CTX, PRIO, COMP, T, ...)
```

В аргументах вызова макроса указывается следующее:
- `CTX` - контекст исполнения актора (необходим для отправки сообщения актору логирования);
- `PRIO`- уровень логирования сообщения (ошибка, предупреждение и так далее). Соответствует уровням логирования из <ссылка на настройки>;
- `COMP`- идентификатор компоненты;
- `T`- текстовое сообщение;
- `...`- прикрепляемые параметры сообщения и их значения (могут отсутствовать).

Пример записи сообщения в лог:
```
YDB_LOG_CTX_COMP(env, PRI_ERROR, 1, "Unable to open file",
    {"path", filename},
    {"errorCode", err});
```

В данном примере происходит запись  сообщения с текстом `Unable to open file` и двумя прикреплёнными параметрами: `path` (значение берётся из переменной `filename`) и `errorCode`(значение берётся из переменной `err`). Для передачи сообщения актору логирования используется контекст исполнения `env`, а в качестве кода компоненты передаётся`1`.

**ЗАМЕЧАНИЕ:**  Обычно при логировании выполняются следующие условия:
1. Уровень сообщения является статическим и известен уже в момент компиляции.
2. Контекст исполнения актора обычно доступен через глобальную переменную `NActors::TlsActivationContext`.
3.  Код компонента является одним и тем же для всего файла `.cpp`.

Поэтому **НАСТОЯТЕЛЬНО РЕКОМЕНДУЕТСЯ** использовать рассматриваемые далее макросы логирования, позволяющие не указывать часть или даже все перечисленные выше параметры.

### Указание уровня сообщения
Поддерживается несколько уровней сообщений (<ссылка на уровни логирования> ). Для каждого из них есть своя версия макроса `YDB_LOG_CTX_COMP`, которая не требует указания  уровня сообщения в виде аргумента.

|Макрос|Уровень сообщения  |
|--|--|
|`YDB_LOG_EMERG_CTX_COMP`  | Возможен сбой системы (например, отказ кластера).|
|`YDB_LOG_ALERT_CTX_COMP`  | Возможна деградация системы, компоненты системы могут выйти из строя. |
|`YDB_LOG_CRIT_CTX_COMP`  | Критическое состояние.|
|`YDB_LOG_ERROR_CTX_COMP`  | Некритическая ошибка. |
|`YDB_LOG_WARN_CTX_COMP`  | Предупреждение, на которое следует отреагировать и исправить, если оно не временное. |
|`YDB_LOG_NOTICE_CTX_COMP`  | Произошло событие, существенное для системы или пользователя.|
|`YDB_LOG_INFO_CTX_COMP`  | Отладочная информация для сбора статистики. |
|`YDB_LOG_DEBUG_CTX_COMP`  | Отладочная информация для разработчиков. |
|`YDB_LOG_TRACE_CTX_COMP`  | Очень подробная отладочная информация.|

**ЗАМЕЧАНИЕ:** Если уровень сообщения известен заранее, то рекомендуется вместо базового макроса `YDB_LOG_CTX_COMP` использовать соответствующий макрос из таблицы выше.

Пример:
```
YDB_LOG_ERROR_CTX_COMP(env, 1, "Unable to open file",
    {"path", filename},
    {"errorCode", err});
```

### Использование стандартного контекста исполнения

В большинстве случаев для передачи сообщений актору логирования используется контекст исполнения, доступный через глобальную переменную `NActors::TlsActivationContext`. Для того, чтобы исходные тексты не были перегружены, введён дополнительный макрос `YDB_LOG_COMP`, который всегда использует `NActors::TlsActivationContext` и не требует указания контекста исполнения. Соответственно, для разных уровней сообщений существуют макросы `YDB_LOG_EMERG_COMP`, `YDB_LOG_ALERT_COMP`  и так далее.

Пример:
```
YDB_LOG_ERROR_COMP(1, "Unable to open file",
    {"path", filename},
    {"errorCode", err});
```

### Использование компонента по умолчанию
В большинстве случаев в одном файле `.cpp` все сообщения пишутся в журнал от имени одного и того же компонента. Для того, чтобы исходные тексты не были перегружены, необходимо  следующее:
1. В начале файла (но после всех директив `#include`) должен быть определён макрос `YDB_LOG_THIS_FILE_COMPONENT`. Он задаёт код компонента для всего файла.
2. Далее в файле должны использоваться макросы логирования, не требующие указания кода компонента. Они аналогичны уже рассмотренным ранее, но их имя не содержит в себе строку `COMP`.

Пример:
```
#define YDB_LOG_THIS_FILE_COMPONENT 1
...
YDB_LOG_ERROR_CTX(env, "Unable to open file",
    {"path", filename},
    {"errorCode", err});
```

**ЗАМЕЧАНИЕ:**
1. Крайне **НЕ РЕКОМЕНДУЕТСЯ** определять макрос `YDB_LOG_THIS_FILE_COMPONENT` в заголовочных файлах, поскольку это может либо приводить к запутанной логике определения кода компонента, либо вызывать ошибки при компиляции (в разных файлах макрос `YDB_LOG_THIS_FILE_COMPONENT` может быть определён по-разному).
2. Наличие макроса `YDB_LOG_THIS_FILE_COMPONENT`  никак не влияет на поведение  макросов логирования, требующих явного указания кода компонента. Более того, это является нормальной практикой, когда в одном файле происходит логирование с указанием разных кодов компонент,  и один из них указан в качестве компонента по умолчанию (соответствующим образом определён макрос `YDB_LOG_THIS_FILE_COMPONENT`).

### Использование контекста и компонента по умолчанию
Если в конкретном файле используется контекст исполнения`NActors::TlsActivationContext`, и при этом известен компонент по умолчанию, то **РЕКОМЕНДУЕТСЯ** использовать макросы логирования, позволяющие не указывать ни контекс исполнения, ни компонент .

Пример:
```
#define YDB_LOG_THIS_FILE_COMPONENT 1
...
YDB_LOG_ERROR("Unable to open file",
    {"path", filename},
    {"errorCode", err});
```
Таким образом, настоятельно рекомендуется пользоваться следующими макросами:

|--|--|
|`YDB_LOG_EMERG`  | Возможен сбой системы (например, отказ кластера).|
|`YDB_LOG_ALERT`  | Возможна деградация системы, компоненты системы могут выйти из строя. |
|`YDB_LOG_CRIT`  | Критическое состояние.|
|`YDB_LOG_ERROR`  | Некритическая ошибка. |
|`YDB_LOG_WARN`  | Предупреждение, на которое следует отреагировать и исправить, если оно не временное. |
|`YDB_LOG_NOTICE`  | Произошло событие, существенное для системы или пользователя.|
|`YDB_LOG_INFO`  | Отладочная информация для сбора статистики. |
|`YDB_LOG_DEBUG`  | Отладочная информация для разработчиков. |
|`YDB_LOG_TRACE`  | Очень подробная отладочная информация.|


## Построение структурированных сообщений
### Переиспользуемые наборы прикрепляемых параметров
Если один и тот же набор параметров прикрепляется к различным сообщениям, то для избежания дублирования кода можно заранее создать этот набор параметров, а затем переиспользовать его при логировании различных событий.

Пример:
```
#define YDB_LOG_THIS_FILE_COMPONENT 1
...
void MyFunction(const std::string& filename) {
    auto context = YDB_LOG_CREATE_MESSAGE(
        {"path", filename});
    ...
    if (err != 0 ) {
		    YDB_LOG_ERROR("MyFunction failed",
			      context,
			      {"errorCode", err});
		    return l
    }
	  ...
    YDB_LOG_NOTICE("MyFunction successed",
        context);
}
```

Результат вызова макроса `YDB_LOG_CREATE_MESSAGE` является экземпляром класса `TStructuredMessage`. Его можно хранить в локальных и глобальных переменных, копировать, передавать и т.д. Если в процессе работы необходимо изменить ранее созданный набор параметров, то для этого следует использовать специальный макрос или предоставляемые методы.

Пример:
```
// Добавление значения
YDB_LOG_UPDATE_MESSAGE(context,
    {"socket", socketNum});
...
// Удаление значения
values.RemoveValue("socket");
```
Макрос `YDB_LOG_UPDATE_MESSAGE` либо добавляет параметр в набор, либо изменяет уже существующее с таким именем.

### Вложенные наборы
Возможно создание вложенных наборов прикрепляемых параметров.

Пример:
```
#define YDB_LOG_THIS_FILE_COMPONENT 1
...
void MyFunction() {
    auto context = YDB_LOG_CREATE_MESSAGE(
        {"path", filename});
    ...
    if (err != 0 ) {
        YDB_LOG_ERROR("MyFunction failed",
            context,
            {"errorCode", err});
        return l
    }
    ...
    YDB_LOG_NOTICE("MyFunction successed",
        context);
}
```
В этом примере к сообщению с текстом `MyFunction successed` будет прикреплены параметр `details.path=100`.

### Контексты логирования
Формирование и переиспользование наборов прикрепляемых параметров неудобно, поскольку эти наборы нужно указывать при логировании каждого сообщения. Для того, чтобы упростить код, можно задать контекст логирования - набор параметров, которые будут автоматически прикрепляться ко всем сообщениям, отправляемым в журнал в данном потоке исполнения.

Пример:
```
#define YDB_LOG_THIS_FILE_COMPONENT 1
...
void MyFunction() {
    YDB_LOG_CREATE_CONTEXT(
        {"path", filename});
    ...
    if (errorCode != 0 ) {
        YDB_LOG_ERROR("MyFunction failed",
            {"error", errorCode});
        return l;
    }
    ...
    YDB_LOG_NOTICE("MyFunction successed");
}
```
В этом примере параметры `path` будет прикрепляться ко всем сообщениям, логируемым в данном потоке исполнения в функции `MyFunction` и вызываемых ею вложенных функциях вплоть до выхода из функции `MyFunction`. В свою очередь, вложенные функции могут пополнять контекст логирования своими параметрами.

## Рекомендации по наполнению логов
### Стиль кодирования
Рекомендуется следующий стиль кодирования при использования макросов `YDB_LOG_...`:
1. Вся обязательная информация (контекст исполнения, компонент, уровень ошибки, текст сообщения) располагается на одной строке.
2. Каждый прикрепляемый параметр располагается на отдельной строке.
3. Закрывающая скобка может располагаться либо на той же строке, где и отдельный параметр, либо на отдельной строке, но тогда она должна выравниваться по началу макроса `YDB_LOG`.
4. Если есть возможность не указывать какие-либо обязательные параметры (контекст исполнения, код компонента, уровень ошибки), то лучше их не указывать и воспользоваться более коротким маркосом. В идеальном случае, нужно использовать макросы вида `YDB_LOG_<level>`.
5. Запрещается определять макрос `YDB_LOG_THIS_FILE_COMPONENT` в заголовочных файлах.

### Стиль текстовых сообщений
Рекомендуется следующий стиль написания текстовых сообщений:
1. Сообщения являются осмысленными (и желательно корректными) предложениями на английском языке.
2. Текст может содержать имена классов и функций.
3. Сообщения являются фиксированным текстом без динамически формируемых фрагментов.
4. Вся динамическая информация, известная на момент исполнения, должна помещаться в виде прикрепляемых параметров.

### Именование параметров

Рекомендуется следующий стиль именования параметров:
1.  Для обозначения одних и тех же сущностей желательно использовать одни и те же имена параметров. Список представлен в таблице далее.
2. Имена параметров пишутся в `camelCase`. Строго ограничения нет, но:
* как правило, имена параметров являются короткими строками, состоящими из 2-3 слов. В данном случае, использование `snake_case` не добавит наглядности;
* использование `camelCase` согласуется со стилем именования переменных в исходном коде системы;
* для многих сущностей уже есть рекомендованные названия параметров, и они в `camelCase`. Желательно сохранять общую стилистику;
3. Желательно не добавлять к именам параметров различные предлоги. Например, если происходит запись в файл, то для параметра, содержащего имя файла, желательно использовать имя `filePath` , но не `toFilePath` или т.п.
4. Если в название параметра входит общепринятая аббревиатура, то она пишется большими буквами. Например,  правильными именами параметров являются `userSID` (поскольку `SID` - общепринятая аббревиатура) и `userId` (`Id` в данном случае id не является аббревиатурой).
5. Не рекомендуется (кроме специальных случаев, рассматриваемых в таблице) использовать отдельные общеупотребимые слова, такие как `id` и т.п.

Рекомендуемые названия параметров:

|Название параметра|Описание  |
|--|--|
|`selfId`  | Идентификатор актора, выполняющего действие (в частности, обрабатывающего сообщение или выполняющего `bootsrap`).|
|`actorClassName`  | Название класса актора.|
|`actorStateName`  | Состояние актора (`bootsrap` или название функции-обработчика сообщения).|
|`ev`  | Информация об обрабатываемом сообщении.|
|`eventType`  | Тип сообщения.|
|`backtrace`  | Стек исключения `TBackTrace::FromCurrentException().PrintToString()`|
|`?добавить?`  | ?добавить? |


### Согласование текстов сообщений и названий параметров
Существует простое эмпирическое правило:
> Если взять текст сообщения, и к нему добавить значения параметров в виде `имя=значение`, то должно получиться осмысленное, законченное, однозначное и понятное человеку описание события.

Например, пусть в исходном тексте сообщение формируется следующим образом:
```
YDB_LOG_ERROR_CTX(env, "Unable to open file",
    {"path", filename},
    {"errorCode", err});
```
Тогда формируемое описание события выглядит так:
``
Unable to open file. path=<...> , errorCode=<...> .
``

### Методология обогащения сообщений контекстной информацией

Предлагается следующая методология для обогащения событий, которые пишет в журнал конкретный актор.

**Шаг 1.** Функция создания контекста логирования.
В акторе должна быть функция, которая формирует текущий контекст логирования в виде структурированного сообщения.

Например, если есть некоторый абстрактный актор чтения файла, то функция создания его контекста логирования может быть такой:
```
TStructuredMessage GetLogContext() const {
    return YDB_LOG_CREATE_MESSAGE(
        {"actorClassName", "TFileReader"},
        {"selfId", SelfId()},
        {"path", filename});
}
```
**ЗАМЕЧАНИЯ:**
1. Вместо имени `GetLogContext` может использоваться `GetLogPrefix` или т.п.
2. Это может быть не функция, а поле класса, создаваемое в конструкторе и изменяемое в ходе работы актора.

**Шаг 2.**  Настройка контекстов логирования.
У каждого актора есть небольшой набор функций, которые могут быть вызваны ядром акторной системой. Сюда входят:

1. Функции 'bootsrap'.
2. Функции обработки сообщений (ищутся по макросу `STATEFN` или т.п.).
3. Прочие виртуальные функции, которые перекрывает актор (они легко ищутся по ключевому слову `override`).

Далее в первой строке каждой такой функции с помощью макроса `YDB_LOG_CREATE_CONTEXT` настраивается контекст логирования.

Пример:
```
STATEFN(StateWork) {
    YDB_LOG_CREATE_CONTEXT(GetLogContext());
    switch (ev->GetTypeRewrite()) {
        ....
    }
}
```

**Альтернативным способом** указания контекста логирования является его явное использование в каждом макросе логирования.
 Пример:
```
YDB_LOG_ERROR_CTX(env, "Unable to open file",
    {"context", GetLogContext()},
    {"errorCode", err});
```
Принципиальным отличием данного способа является то, что имена параметров из контекста будут иметь префикс `context.`.
