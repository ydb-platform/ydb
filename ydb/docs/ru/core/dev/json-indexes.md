# JSON-индексы

JSON-индексы — это специализированный тип [вторичного индекса](../concepts/glossary.md#secondary-index), который позволяет эффективно искать строки по содержимому JSON-документов в колонке типа `Json` или `JsonDocument`: по существованию пути внутри документа и по значению, расположенному в указанном пути. В отличие от обычных вторичных индексов, которые строятся по значениям колонок целиком, JSON-индекс «заглядывает» внутрь JSON-документа и индексирует отдельные пути и значения.

Общее описание поиска по JSON см. в разделе [Поиск по JSON](../concepts/query_execution/json_search.md).

## Характеристики JSON-индексов {#characteristics}

JSON-индекс — это **глобальный**, **синхронный** и **обновляемый** вторичный индекс по одной JSON-колонке. Он строит [инвертированный индекс](https://ru.wikipedia.org/wiki/Инвертированный_индекс) по путям JSON-дерева и позволяет:

* быстро фильтровать строки по существованию пути через [JSON_EXISTS](../yql/reference/builtins/json.md#json_exists);
* быстро фильтровать строки по значению в пути через [JSON_VALUE](../yql/reference/builtins/json.md#json_value) и фильтр-предикаты JsonPath внутри `JSON_EXISTS`.

JSON-индекс запрашивается уже существующими функциями `JSON_EXISTS` и `JSON_VALUE` с грамматикой [JsonPath](../yql/reference/builtins/json.md#jsonpath). [Оптимизатор](../concepts/glossary.md#optimizer) распознаёт такие предикаты по индексированной колонке и подставляет чтение по индексу автоматически — отдельное выражение `VIEW`, в отличие от [полнотекстового индекса](fulltext-indexes.md), не требуется (но поддерживается). Подробнее об обоих режимах — в разделе [Чтение по JSON-индексу](../yql/reference/syntax/select/json_index.md).

## Хранение данных {#storage}

JSON-индекс не хранит копию JSON-документа целиком. Вместо этого он раскладывает документ на отдельные **поисковые записи** и для каждой из них хранит список первичных ключей строк, где она встречается:

* для **каждого пути** в JSON-дереве создаётся запись «путь существует»;
* если путь ведёт к **скалярному значению** (строке, числу, логическому значению или `null`), дополнительно создаётся запись «путь + значение».

Массивы при этом **прозрачны**: элементы массива индексируются под тем же путём, что и сам массив, поэтому индекс одинаково отвечает на вопросы и про `$.items`, и про `$.items[0]`, и про `$.items[*]`.

Благодаря такому устройству вопрос «найти вложенный путь или значение внутри JSON» сводится к поиску по инвертированному индексу, что и обеспечивает скорость.

Рассмотрим документ:

```json
{
    "id": 42042,
    "name": "Michael",
    "email": null,
    "items": [null, "str"],
    "parts": {
        "key": "k1",
        "value": false
    }
}
```

Концептуально для него индексируются следующие записи (без учёта внутреннего представления):

| Поисковая запись | Что позволяет найти |
|---|---|
| `$` | документ не равен `NULL` |
| `$.id` / `$.id == 42042` | путь `id` существует / равен `42042` |
| `$.name` / `$.name == "Michael"` | путь `name` существует / равен `"Michael"` |
| `$.email` / `$.email == null` | путь `email` существует / равен `null` |
| `$.items` / `$.items == null` / `$.items == "str"` | путь `items` существует / содержит элемент `null` / содержит элемент `"str"` |
| `$.parts` | путь `parts` существует |
| `$.parts.key` / `$.parts.key == "k1"` | вложенный путь существует / равен `"k1"` |
| `$.parts.value` / `$.parts.value == false` | вложенный путь существует / равен `false` |

## Создание JSON-индекса {#create}

JSON-индекс можно определить при создании таблицы или добавить к существующей таблице. Тип индекса задаётся как `json`:

```yql
CREATE TABLE `TestTable` (
    `Key` Uint64,
    `Value` JsonDocument,
    PRIMARY KEY (`Key`),
    INDEX `json_idx` GLOBAL SYNC USING json ON (`Value`)
);
```

```yql
ALTER TABLE `TestTable` ADD INDEX `json_idx` GLOBAL SYNC USING json ON (`Value`);
```

Полный синтаксис создания JSON-индекса:

* при создании таблицы: [CREATE TABLE](../yql/reference/syntax/create_table/json_index.md);
* добавление к существующей таблице: [ALTER TABLE](../yql/reference/syntax/alter_table/indexes.md).

## Чтение по JSON-индексу {#read}

JSON-индекс используется при чтении в одном из двух режимов:

* **Автоматический выбор.** Оптимизатор сам распознаёт предикаты `JSON_EXISTS` / `JSON_VALUE` по индексированной колонке и заменяет их на чтение по индексу. Дополнительный синтаксис не нужен:

  ```yql
  SELECT * FROM TestTable WHERE JSON_EXISTS(Value, '$.abc.def');
  SELECT * FROM TestTable WHERE JSON_VALUE(Value, '$.abc' RETURNING Int32) == 10;
  ```

* **Явное указание индекса** через `VIEW IndexName` — как и для других вторичных индексов:

  ```yql
  SELECT * FROM TestTable VIEW json_idx WHERE JSON_EXISTS(Value, '$.abc.def');
  ```

Эти режимы различаются поведением, когда предикат не получается превратить в обращение к индексу: при автоматическом выборе запрос выполняется обычным способом (без ускорения индексом), а при явном `VIEW` возвращается ошибка. Полное описание поддерживаемых предикатов, правил автоматического выбора и отличия «ошибка против отката» — в разделе [Чтение по JSON-индексу](../yql/reference/syntax/select/json_index.md).

## Обновление JSON-индексов {#update}

JSON-индексы автоматически поддерживаются при модификации данных. Таблицы с JSON-индексами поддерживают:

* `INSERT`
* `UPSERT`
* `REPLACE`
* `UPDATE`
* `DELETE`

## Удаление JSON-индексов {#drop}

```yql
ALTER TABLE `TestTable` DROP INDEX `json_idx`;
```

## Ограничения {#limitations}

* Первичный ключ основной таблицы должен состоять из **одной** колонки одного из типов: `Uint64`, `Int64`, `Int32`, `Uint32`. Поддержка произвольного первичного ключа находится в разработке.
* В JSON-индекс может входить ровно **одна** колонка типа `Json` или `JsonDocument` (одна колонка в `ON`).
* Покрывающие колонки (`COVER`) не поддерживаются.
* Тип параметра запроса не может быть обёрнут в `Optional<T>` — опциональные параметры не поддерживаются.
* Целые литералы с абсолютным значением больше `2^53` не помещаются в тип `double` без потери точности — сравнение по равенству с таким литералом не ускоряется индексом и сводится к проверке существования пути.
* Приведение вещественных литералов (`Float`, `Double`) к целочисленным типам при сравнении не допускается.
* Отрицание (`NOT`, `!= true`, `!pred` в JsonPath) нельзя использовать как индексный предикат.

Полный список правил, по которым предикат ускоряется или не ускоряется индексом, приведён в разделе [Чтение по JSON-индексу](../yql/reference/syntax/select/json_index.md).
