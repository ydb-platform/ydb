# min_max индекс

`min_max` — это локальный индекс пропуска ([skip index](https://en.wikipedia.org/wiki/Block_Range_Index)) для [колоночных таблиц](../concepts/datamodel/table.md#column-oriented-tables). Для каждого фрагмента данных (порции) индекс хранит минимальное и максимальное значения индексируемой колонки. На чтении, если искомое значение или диапазон не пересекаются с интервалом `[min, max]` фрагмента, такой фрагмент пропускается без чтения колоночных данных. Это сокращает объём фактически читаемых данных для селективных запросов.

В отличие от глобальных [вторичных индексов](../concepts/glossary.md#secondary-index), `min_max` не материализует отдельную индексную таблицу и не выбирается в запросах с `VIEW`: пропуск фрагментов происходит в слое хранения при чтении основной таблицы.

## Когда применять {#when}

`min_max` особенно эффективен, когда значения индексируемой колонки коррелируют с порядком хранения данных (например, с первичным ключом или временем загрузки). В таком случае интервалы `[min, max]` соседних фрагментов слабо перекрываются, и фильтр позволяет пропускать большие куски данных.

Типичные сценарии:

* Поиск по диапазону значений в колонке, не входящей в первичный ключ, но коррелирующей с ним (например, временная метка события при первичном ключе по идентификатору).
* Точечный поиск по колонке с небольшим числом различных значений на фрагмент (например, `source = "..."`, когда данные приходят пачками от одного источника).
* Колонки, где значения монотонно растут (например, идентификаторы, выдаваемые источником-генератором).

Если данные распределены по колонке равномерно и случайно относительно порядка хранения, эффективность `min_max` будет низкой.

## Какие предикаты ускоряются {#predicates}

`min_max` используется оптимизатором для следующих предикатов на индексируемой колонке:

* равенство: `col = <value>`;
* неравенства: `col < <value>`, `col <= <value>`, `col > <value>`, `col >= <value>`;
* `col BETWEEN <a> AND <b>` (раскладывается в комбинацию неравенств).

Не ускоряются: `LIKE`, поиск подстроки (`StartsWith` / `EndsWith` / `Contains`), сравнение с `NULL`.

## Синтаксис {#syntax}

Создание индекса при создании таблицы:

```yql
CREATE TABLE `<table_name>` (
    ...
    INDEX `<index_name>`
        LOCAL
        USING min_max
        ON ( <column> )
    [,   ...]
)
WITH (STORE = COLUMN);
```

Добавление индекса к существующей таблице:

```yql
ALTER TABLE `<table_name>`
  ADD INDEX `<index_name>`
    LOCAL USING min_max
    ON ( <column> );
```

Полный синтаксис для `CREATE TABLE`: [min_max индекс в CREATE TABLE](../yql/reference/syntax/create_table/min_max_index.md).

Удаление индекса:

```yql
ALTER TABLE `<table_name>` DROP INDEX `<index_name>`;
```

У `min_max` нет настраиваемых параметров — секция `WITH (...)` не используется.

## Поддерживаемые типы колонок {#types}

`min_max` строится по одной колонке любого типа, для которого определены операции сравнения, а именно:

* целочисленные типы: `Int8`, `Int16`, `Int32`, `Int64`, `Uint8`, `Uint16`, `Uint32`, `Uint64`;
* числа с плавающей точкой: `Float`, `Double`;
* `Decimal`;
* строки и бинарные данные: `String`, `Utf8`, `Bytes`;
* дата и время: `Date`, `Datetime`, `Timestamp`, `Date32`, `Datetime64`, `Timestamp64`, `Interval`, `Interval64`;
* `Bool`, `Uuid`.

Не поддерживаются типы без определённого порядка сравнения: `Json` и `JsonDocument`.

## Примеры {#examples}

Создание таблицы с `min_max` индексом по колонке `event_time`:

```yql
CREATE TABLE `/Root/events` (
    id Uint64 NOT NULL,
    event_time Timestamp NOT NULL,
    source Utf8,
    payload String,
    PRIMARY KEY (id),
    INDEX idx_event_time LOCAL USING min_max ON (event_time)
)
WITH (STORE = COLUMN);
```

Селективный запрос по диапазону `event_time` сможет использовать индекс для пропуска фрагментов, целиком лежащих вне запрошенного интервала:

```yql
SELECT id, source
FROM `/Root/events`
WHERE event_time BETWEEN Timestamp("2026-01-01T00:00:00Z")
                     AND Timestamp("2026-01-02T00:00:00Z");
```

Добавление `min_max` индекса к уже существующей таблице (индекс достраивается фоном для новых порций; для существующих данных индекс будет построен при ближайшей перестройке порции):

```yql
ALTER TABLE `/Root/events`
  ADD INDEX idx_source LOCAL USING min_max ON (source);
```

## Ограничения {#limitations}

* `min_max` поддерживается только для [колоночных таблиц](../concepts/datamodel/table.md#column-oriented-tables). Создание `min_max` индекса на строковой таблице будет отклонено.
* В одном `min_max` индексе индексируется ровно одна колонка — нельзя указать несколько колонок в `ON (...)`.
* Колонки покрытия (`COVER (...)`) не поддерживаются.
* `min_max` доступен только в варианте `LOCAL`; глобального `min_max` индекса не существует.
* Индекс нельзя использовать в запросе через `VIEW <index_name>` — это локальный индекс, пропуск фрагментов происходит автоматически в слое хранения при чтении основной таблицы.
* Изменение типа индекса или его параметров после создания не поддерживается — индекс нужно удалить и пересоздать.
* Индекс применяется только к тем фрагментам данных, для которых он был построен на момент записи или перестройки. Для остальных фрагментов пропуск по `min_max` индексу не выполняется, пока порция не будет перестроена.
