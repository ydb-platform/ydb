# Обзор

При выполнении запросов к базе данных текст на языке SQL комплируется, т.е. преобразуется в формат, пригодный для распределённого выполнения на [кластере](../../concepts/glossary.md#cluster). В большинстве случаев есть много вариантов выполнения одного и того же запроса, и [стоимостной оптимизатор](../../concepts/query_execution/optimizer.md) {{ ydb-short-name }} старается выбрать самый эффективный способ.

Главные задачи при оптимизации запросов в {{ ydb-short-name }}:

- понять, каким образом сервер планирует выполнить ваш запрос
- проанализировать, как именно запрос был выполнен на практике
- выявить и использовать возможности для ускорения выполнения

## Как сервер собирается выполнять запрос

После компиляции и оптимизации строится [план выполнения запроса](plans.md) — последовательность [операторов](../../concepts/glossary.md#operator), которые необходимо выполнить для получения результата. Этот план можно получить через команду `EXPLAIN` без реальной обработки данных. Этот план будет отражать структуру выполнения запроса и содержать оценки [стоимостного оптимизатора](../../concepts/query_execution/optimizer.md), полученные на основании существующей **статистики источников данных**, например, количество строк в таблице или её размер в байтах.

Этот план описывается в формате [JSON](https://ru.wikipedia.org/wiki/JSON), который далее преобразуется в более удобные для изучения форматы. Варианты визуализации зависят от того, как именно выполнялся запрос — посредством SDK, CLI или UI, и описаны в разделе [План выполнения запроса](plans.md).

## Как на самом деле был выполнен запрос

Для более детального и предметного анализа поведения запроса {{ ydb-short-name }} собирает **статистику выполнения**:

- длительность отдельных шагов ([операторов](../../concepts/glossary.md#operator))
- объёмы переданных данных
- ожидания, возникшие при взаимодействии отдельных [узлов](../../concepts/glossary.md#node) распределённой системы

Термин **статистика** без уточнения может иметь разный смысл, который обычно понятен из контекста изложения.

На большом кластере каждый шаг плана может выполняться как отдельные [задачи](../../concepts/glossary.md#task) параллельно на большом количестве [узлов](../../concepts/glossary.md#node). Детальная информация для каждого отдельного экземпляра занимала бы слишком большой объём, вместо этого она обрабатывается сервером и для каждого набора одинаковых [задач](../../concepts/glossary.md#task) публикуется в агрегированном виде, обогащая план в формате [JSON](https://ru.wikipedia.org/wiki/JSON).

Но даже в агрегированном виде **статистика выполнения** всё ещё является достаточно объёмной для самостоятельного анализа. Чтобы решить эту проблему, разработан специальный способ визуализации этой статистики вместе с планом в формате [SVG](https://ru.wikipedia.org/wiki/SVG). Основная часть этого раздела рассказывает, как [получить](plans.md#svg-cli) подобный план запроса и как его [интерпретировать](layout.md); расшифровка метрик — в [Визуализации метрик запроса](metrics.md).

## Что можно сделать для ускорения работы

В этом разделе основной фокус сделан на анализе и интерпретации планов выполнения запросов: рассматриваются способы понять, как сервер планирует и реализует выполнение, а также как выявить узкие места с помощью изучения плана и сопутствующих метрик. Практические аспекты ускорения работы разбираются через нахождение потенциальных точек оптимизации на основе анализа этих данных.
