# Обзор

При выполнении запросов к базе данных, текст на языке SQL преобразуется в формат, пригодный для распределённого выполнения на кластере (компилируется). В большинстве случаев есть много выариантов выполнения одного и того же запроса, в этом случае оптимизатор {{ ydb-short-name }} старается выбрать самый эффективный способ.

При оптимизации поведения запроса надо уметь отвечать на следующие вопросы:

- как сервер собирается выполнять запрос
- как на самом деле был выполнен запрос
- что можно сделать для ускорения работы

В примерах этого раздела используется набор данных [TPC-H](../../reference/ydb-cli/workload-tpch.md). Схему таблиц и связи описана в спецификации TPC-H. Некоторые примеры являются запросами из бенчмарка с адаптацией под {{ ydb-short-name }}, другие специально подготовлены для иллюстрации излагаемого материала. Исходный текст SQL всех примеров позволяет самостояльно повторить все описанные примеры.

## Как сервер собирается выполнять запрос

После компиляции и оптимизации строится план выполнения запроса - последовательность действий (операторов), которые необходимо выполнить для получения результаты. Этот план можно получить через команду `EXPLAIN` без реальной обработки данных. Этот план будет отражать структуру выполнения запроса и содержать оценки стоимостного оптимизатора, полученные на основании существующей **статистики источников данных**, например, количество строк в таблице или её размер в байтах.

Этот план описывается в формате [JSON](https://ru.wikipedia.org/wiki/JSON), который далее преобразуется в более удобные для изучения форматы. Варианты визуализации зависят от того, как именно выполнялся запрос - посредством SDK, CLI или UI и описаны в разделе [План выполнения запроса](plans.md)

## Как на самом деле был выполнен запрос

Для более детального и предметного анализа поведения запроса {{ ydb-short-name }} собирает **статистику выполнения**:

- длительность отдельных шагов (операторов)
- объёмы переданных данных
- ожидания, возникшие при взаимодействии отдельных узлов распределённой системы

Термин **статистика** без уточнения, может иметь разный смысл, который обычно понятен из контекста изложения.

На большом кластере каждый шаг плана может выполняться как отдельные задачи параллельно на большом количестве узлов. Детальная информация для каждого отдельного экземпляра занимала бы слишком большой объём, вместо этого оно обрабатывается сервером и для каждого набор одинаковых задач публикуется в агрегированном виде, обогащая план в формате [JSON](https://ru.wikipedia.org/wiki/JSON).

Но даже в агрегированном виде **статистика выполнения** всё ещё является достаточно объёмной для самостоятельного анализа. Чтобы решить эту проблему, разработан специальный способ визуализации этой статистики вместе с планом в формате [SVG](https://ru.wikipedia.org/wiki/SVG). Основная часть этого раздела рассказывает, как [получить](how-to-get.md) подобный план запроса и как его [интерпретировать](layout.md).

## Что можно сделать для ускорения работы

Когда понятно поведение системы и выявлены узкие места, то можно предпринять разные действия для ускорения выполнения запроса. Они могут включать разные действия, в том числе настройку кластера и изменение структуры данных. В этом разделе обсуждается, как можно повлиять на поведение оптимизатора посредством [подсказок](hints.md)
