# План выполнения запроса

План запроса — подробное описание того, как сервер выполнит пользовательский запрос после компиляции и оптимизаций. План помогает заранее оценить порядок операций и ожидаемые объёмы данных.

Когда запрос реально выполняется, план дополнительно обогащается статистикой выполнения и даёт больше данных для анализа.

## Обзор режимов {#modes-overview}

| Цель | Режим | CLI | UI | Раздел |
| --- | --- | --- | --- | --- |
| Оценка плана без выполнения запроса | EXPLAIN | `ydb sql --explain` | кнопка **Explain** | [EXPLAIN через CLI](#explain-cli), [EXPLAIN в UI](#explain-ui) |
| План с фактическими метриками выполнения | ANALYZE | `ydb sql --explain-analyze` | кнопка **Run** | [ANALYZE через CLI](#analyze-cli), [ANALYZE в UI](#analyze-ui) |
| Графический план в формате SVG | SVG | `ydb workload … --plan` | **Open Execution Plan** | [SVG через CLI](#svg-cli), [SVG в UI](#svg-ui) |

## Получение EXPLAIN плана через CLI и SDK {#explain-cli}

Рассмотрим простой запрос:

```sql
SELECT n_name
  FROM nation
  JOIN region ON nation.n_regionkey == region.r_regionkey
  WHERE r_name = "AMERICA"
```

Чтобы узнать план этого запроса, не выполняя его, необходимо запустить команду `ydb sql` с указанием параметра `--explain` (предполагается, что текст запроса находится в файле `file1.sql`):

```bash
ydb sql -f file1.sql --explain
```

В терминале будет выведено следующее:

```text
┌────────┬────────┬────────┬────────────────────────────────────────────────────────────────────────────────────────────────────┐
│ E-Cost │ E-Rows │ E-Size │ Operation                                                                                          │
├────────┼────────┼────────┼────────────────────────────────────────────────────────────────────────────────────────────────────┤
│        │        │        │ ┌> ResultSet                                                                                       │
│ 2003   │ 3      │ 457    │ └─┬> InnerJoin (Map) (nation.n_regionkey = region.r_regionkey)                                     │
│ 0      │ 25     │ 1212   │   ├──> TableFullScan (Table: nation, ReadColumns: ["n_nationkey (-∞, +∞)","n_name","n_regionkey"]) │
│ 0      │ 1      │ 67     │   └─┬> Filter (Blocks: True, r_name == "AMERICA", Pushdown: True)                                  │
│ 0      │ 5      │ 672    │     └──> TableFullScan (Table: region, ReadColumns: ["r_regionkey (-∞, +∞)","r_name"])             │
└────────┴────────┴────────┴────────────────────────────────────────────────────────────────────────────────────────────────────┘
```

В колонке `Operation` показана структура запроса. В ней написано, что сначала сервер планирует (последовательность выполнения идёт снизу вверх):

- прочитать таблицу `region`
- применить предикат (фильтр) к полю `r_name` со значением `AMERICA`
- объединить результат с содержимым таблицы `nation`

В других колонках — прогнозные оценки оптимизатора:

| Колонка | Назначение |
| --- | --- |
| `E-Cost` | Оценка стоимости фрагмента плана |
| `E-Rows` | Ожидаемое число строк |
| `E-Size` | Ожидаемый объём данных |

Оценки стоимостной оптимизатор выводит из **статистики источников данных** и структуры запроса.

При вызове с `--format json-unicode` (или при работе через SDK) сервер возвращает план в JSON; таблица в терминале — человекочитаемое представление того же плана. JSON можно получить так:

```bash
ydb sql -f file1.sql --explain --format json-unicode
```

Вывод достаточно объёмный и здесь приведён лишь частично:

```json
{
    "Plan" : {
        "Plans" : [
            {
                "PlanNodeId" : 6,
                "Plans" : [
                    {
"..."
                    }
                ],
                "Node Type" : "ResultSet",
                "PlanNodeType" : "ResultSet"
            }
        ],
        "Node Type" : "Query",
        "PlanNodeType" : "Query"
    }
}
```

Если вы взаимодействуете с сервером программным образом, то план присылается именно в этом формате.

## Получение EXPLAIN плана в UI {#explain-ui}

В графическом интерфейсе {{ ydb-short-name }} также можно получить план выполнения, для этого надо использовать кнопку `[Explain]`. В UI доступно больше вариантов плана. По умолчанию показывается закладка `Computation Graph`, граф вычислений в виде операторов и связей между ними:

![Граф выполнения в UI](../../_assets/rts-ui-graph.png)

Если переключиться на закладку `Explain`, вы увидите план, похожий на вывод CLI:

![Получение EXPLAIN плана в UI](../../_assets/rts-ui-analyze.png)

Оставшиеся закладки позволяют также исследовать исходную структуру плана в формате `JSON` на одноимённой закладке, а также ознакомиться с низкоуровневым форматом, в который компилируется программа на закладке `AST` (обсуждение этого формата выходит за рамки данной документации).

Графический SVG-план с метриками выполнения описан в разделах [Получение графического плана при выполнении запроса через CLI](#svg-cli) и [Получение графического плана при выполнении запроса в UI](#svg-ui). Вкладка `Computation Graph` — отдельное представление, не то же самое, что SVG из workload или UI.

## Получение плана при выполнении запроса через CLI {#analyze-cli}

Чтобы выполнить запрос и получить его план, следует запустить команду `ydb sql` с другой опцией `--explain-analyze`. Вывод в терминале при этом несколько изменится - в дополнение к оценкам оптимизатора появятся актуальные метрики, собранные во время работы запроса:

```text
┌───────┬────────┬────────┬────────┬────────┬─────────────────────────────────────────────────────────────────────────────────────────────────────────────────┐
│ A-Cpu │ A-Rows │ E-Cost │ E-Rows │ E-Size │ Operation                                                                                                       │
├───────┼────────┼────────┼────────┼────────┼─────────────────────────────────────────────────────────────────────────────────────────────────────────────────┤
│       │        │        │        │        │ ┌> ResultSet                                                                                                    │
│ 1     │ 5      │ 2003   │ 3      │ 457    │ └─┬> InnerJoin (Map) (A-SelfCpu: 0.367, nation.n_regionkey = region.r_regionkey, A-Size: 54)                    │
│       │ 25     │ 0      │ 25     │ 1212   │   ├──> TableFullScan (Table: nation, ReadColumns: ["n_nationkey (-∞, +∞)","n_name","n_regionkey"], A-Size: 577) │
│ 0     │ 1      │ 0      │ 1      │ 67     │   └─┬> Filter (r_name == "AMERICA", A-SelfCpu: 0.333, Blocks: True, Pushdown: True, A-Size: 16)                 │
│       │ 1      │ 0      │ 5      │ 672    │     └──> TableFullScan (Table: region, ReadColumns: ["r_regionkey (-∞, +∞)","r_name"], A-Size: 13)              │
└───────┴────────┴────────┴────────┴────────┴─────────────────────────────────────────────────────────────────────────────────────────────────────────────────┘
```

Исходный план в формате JSON объёмный; здесь приведена его часть.

## Получение плана при выполнении запроса в UI {#analyze-ui}

Аналогично, план запроса доступен в UI при обычном выполнении кнопкой `[Run]`. Как и в случае с CLI, в план добавляются актуальные метрики — вывод похож на рисунок в разделе [Получение EXPLAIN плана в UI](#explain-ui).

## Получение графического плана при выполнении запроса через CLI {#svg-cli}

Графический план в формате SVG можно получить посредством утилиты командной строки. Для этого надо использовать команду [workload](../../reference/ydb-cli/commands/workload/index.md) с опцией `--plan`. На диске появятся файлы с базовым именем, заданным в опции, в нескольких форматах; расширение `.svg` соответствует графическому представлению.

Для произвольного SQL-запроса:

```bash
ydb workload query run --plan filename -q "SELECT count(*) FROM lineitem"
```

Для запросов из [TPC-H](../../reference/ydb-cli/workload-tpch.md) бенчмарка:

```bash
ydb workload tpch run --plan filename
```

## Получение графического плана при выполнении запроса в UI {#svg-ui}

Чтобы в [{{ ydb-short-name }} UI](../../reference/embedded-ui/ydb-monitoring.md) отображался графический план выполнения запроса, включите настройку `Experiments | Execution plan`.

{% note info %}

Графический план в UI пока относится к экспериментальным возможностям (раздел `Experiments`). Со временем он должен перейти в обычные настройки интерфейса.

{% endnote %}

![Включение графического представления в UI](../../_assets/rts-ui-settings.png)

После этого выполните запрос, например:

```sql
SELECT count(*) FROM lineitem
```

В дополнительном меню справа (кнопка [...], отображается на всех вкладках интерфейса) появятся пункты:

- `Open Execution Plan` — открыть план в новой вкладке браузера;
- `Download Execution Plan` — сохранить план в файл.

Там же есть пункт `Download Diagnostics`.

![Просмотр и сохранение плана](../../_assets/rts-get-execution-plan.png)

Для этого запроса план будет выглядеть примерно так (конкретные значения метрик зависят от конфигурации и нагрузки кластера):

![План выполнения запроса](../../_assets/rts-count-lineitem.svg){inline=false}
