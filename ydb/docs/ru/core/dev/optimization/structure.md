# Структура плана запроса

На [диаграмме](layout.md) левая колонка показывает граф стадий и [операторов](../../concepts/glossary.md#operator); ниже — их семантика, каналы связи и нетривиальные топологии.

## Стадии {#stages}

Ниже представлен запрос и его графическое представление:

```sql
SELECT count(*) cnt, l_orderkey, sum(l_quantity)
  FROM lineitem
  WHERE l_commitdate >= date("1993-01-01") and l_commitdate < date("1993-02-01")
  GROUP BY l_orderkey
  HAVING sum(l_quantity) > 200
  ORDER BY cnt DESC, l_orderkey
```

![Простая агрегация](../../_assets/rts-structure-1.svg){inline=false}

В примере три нумерованные вычислительные [стадии](../../concepts/glossary.md#processing-stage) ```0```, ```1``` и ```2``` и отдельная стадия у таблицы ```lineitem``` (стадии хранения не нумеруются). Любую стадию можно временно выделить на схеме; здесь выделена стадия ```1```.

Интерактивные элементы на встроенных изображениях недоступны — см. [Расположение информации на плане](layout.md).

На схеме — структура выполнения запроса, полученная после его компиляции: [логические операторы](../../concepts/glossary.md#logical-operator), затем [физические](../../concepts/glossary.md#physical-operator), сгруппированные в стадии выполнения.

Стадия — основная единица плана; задачи стадии выполняются параллельно, часто на разных узлах. Данные передаются от стадии к стадии снизу вверх: от хранилища к результату.

Стадии бывают двух типов:

1. Выполняемые в системе хранения ([колоночные таблицы](../../concepts/glossary.md#column-oriented-table) или [шарды](../../concepts/glossary.md#data-shard)) — более тёмная заливка.
2. Вычисляемые без хранения (compute) — светлее; такие стадии нумеруются подряд, с ```0```.

Вкратце по плану:

{% note info %}

Нумерованный список ниже на странице идёт **сверху вниз**, на графике стадии и [операторы](../../concepts/glossary.md#operator) расположены **снизу вверх**. Сопоставлять пункты со схемой удобно, начиная с **нижней** части плана.

{% endnote %}

1. Таблица ```lineitem``` сканируется целиком: условие ```WHERE``` не опирается на [первичный ключ](../../concepts/glossary.md#primary-key).
2. Фильтр по ```l_commitdate``` выполняется в хранилище (pushdown предиката), чтобы сократить объём передаваемых данных.
3. В стадии ```0``` — предварительная агрегация с группировкой по ```l_orderkey``` (```GROUP BY```).
4. Данные перераспределяются (hash shuffle) в стадию ```1```; подробнее — в разделе [Каналы связи](#connections).
5. В стадии ```1``` — финальная агрегация, фильтр ```HAVING``` и локальная сортировка.
6. Стадия ```2``` сливает отсортированные потоки в один глобально упорядоченный результат.

{% note tip %}

Неочевидный элемент схемы можно подсветить курсором: у большинства узлов есть всплывающая подсказка. Это не замена разделам документации, но удобно для быстрой ориентации и метрик.

{% endnote %}

## Каналы связи {#connections}

Канал связи на диаграмме показан пятиугольником с направлением потока. В этом примере выделен канал ![](../../_assets/rts-hash-conn-u.svg) между стадиями ```0``` и ```1```.

У линейного плана из четырёх стадий три соединения:

1. ```E``` — чтение из хранилища (вне compute).
2. ```H``` — hash shuffle, разбиение по одному или нескольким полям (список полей — во всплывающей подсказке по пятиугольнику).
3. ```Me``` — слияние (merge) уже отсортированных потоков; для этого типа указываются поля сортировки.

![Каналы связи](../../_assets/rts-structure-2.svg){inline=false}

Соединение состоит из двух частей:

1. **Выходной канал** — исходящий трафик из стадии ```0```. Синяя стрелка вправо-вверх с номером стадии-получателя (здесь ```1```).
2. **Входной канал** — приём в стадию ```1```. Зелёная стрелка влево-вверх с номером стадии-отправителя (```0```).

На графике **успешно** завершённого запроса объём данных на выходе канала и на входе должен совпадать: {{ ydb-short-name }} гарантирует доставку между [узлами](../../concepts/glossary.md#node) без потерь и дубликатов.

Пока запрос выполняется, показатели входа и выхода могут расходиться из-за задержки буферов и асинхронного обновления статистики. На графике ошибочного запроса расхождения тоже возможны.

Входной канал сопоставляется [оператору](../../concepts/glossary.md#operator), который принимает данные. В простом случае у стадии один вход: поток из канала подаётся в первый по порядку обработки (нижний в списке) [оператор](../../concepts/glossary.md#operator). В сложных структурах выполнения запроса зачастую несколько входов; у [оператора](../../concepts/glossary.md#operator) справа дублируется номер стадии-источника исходящего канала (здесь ```0```), он подсвечивается при выборе канала. Разбор на примере с ```JOIN``` — в [разделе ниже](#join).

Итог по выбранному каналу между ```0``` и ```1```:

1. Начинается в стадии ```0```, направлен в стадию ```1```.
2. Проходит через hash shuffle.
3. Входит в стадию ```1``` со стадии ```0```.
4. Подаётся в [оператор](../../concepts/glossary.md#operator) агрегации.

Между стадией хранения и первой вычислительной стадией действуют те же логические каналы, но с иными деталями реализации, чем между двумя compute-стадиями (см. [выше](#stages)). Планировщик {{ ydb-short-name }} по возможности ставит связанные задачи рядом с шардами хранения, чтобы сократить сетевой трафик.

На схеме такие каналы похожи на межстадийные: другие оттенки, другие пиктограммы, без дублирования номера стадии хранения. Сейчас вычислительная стадия связана не более чем с одной стадией хранения; её данные всегда приходят в самый нижний [оператор](../../concepts/glossary.md#operator) стадии, отдельная навигация не нужна.

## Объединение стадий {#join}

Ниже — запрос с ```JOIN``` двух таблиц.

```sql
PRAGMA ydb.OptimizerHints = 'JoinType(nation region shuffle)';

SELECT n_name
  FROM nation
  JOIN region ON nation.n_regionkey == region.r_regionkey
  WHERE r_name = "AMERICA"
```

Таблицы `region` и `nation` небольшие (5 и 25 строк), запрос быстрый, трафик мал, но структура плана показательна. Для примера задан **Shuffle Join** через [подсказку оптимизатора](hints.md). Без этой прагмы план будет другим: при малом объёме данных оптимизатор выберет иной план (его можно сравнить, убрав `PRAGMA`).

![Объединение стадий](../../_assets/rts-nation-region.svg){inline=false}

Стадия ```2``` (на схеме изначально выделена) получает данные сразу от стадий ```0``` и ```1```. Стадии расположены по вертикали, и направление движения данных — снизу вверх, однако связи между ними не обязательно последовательные. Если у стадии есть несколько предшественников, слева появляется отступ, а входные ```H```-каналы обозначаются пятиугольником ![](../../_assets/rts-hash-conn-l.svg) с направлением влево; в этом примере таких соединений два.

Данные сходятся в [физическом операторе](../../concepts/glossary.md#physical-operator) InnerJoin (эквивалент ```JOIN``` в тексте запроса). Указаны ключи ```n_regionkey``` и ```r_regionkey```; справа — номера стадий ```0``` и ```1```.

Красные круги с числами ```1``` и ```5``` — напоминание из раздела [Агрегаты](metrics.md#aggregates): число ненулевых метрик (здесь — по трафику в каналах) меньше числа задач стадии. Часть задач не получила данных, частая причина этого - перекос или избыточный параллелизм.

На этой схеме следуте обратить внимания на два типичных случая, которые вызывают перекос по данным (`data skew`) и, таким образом, замедляют выполнения запроса, потому что те задачи, которые получили меньший объём трафика, завершатся быстрее и будут ждать другие, которым досталось больше данных:

### 1. Неравномерное распределение данных в хранилище

В столбце `Tasks` видно, что у стадий хранения указан параллелизм 1 (по одному шарду на таблицы `region` и `nation`). У связанных с ними вычислительных (`compute`) стадий с номерами `0` и `1` запланированы по 3 задачи. Все прочитанные из каждой таблицы строки были сгруппированы в один пакет (`batch`) и отправлены из колоночного шарда в случайную **одну** из трёх задач. Оставшимся задачам ничего не досталось, поэтому они не отчитались в статистике по этому каналу.

Когда в таблице данных больше, и пересылка использует более одного сообщения, то обычно данные распределяются равномерно между вычислительными задачами. Однако, если при хранении возникает перекос (одни шарды содержат больше данных чем другие), то похожая неравномерность также может проявляться и плохо влиять на производительность, поэтому она выделяется на схеме, чтобы обратить на это внимание.

### 2. Неравномерное распределение ключей

Даже если данные в хранилище распределены равномерно, они перераспределяются в процессе последующей обработки, при этом это перераспределение также может вызывать перекос, вызванный алгоритмическими особенностями реализации.

Соединение **hash shuffle** используется для корректной обработки больших объёмов данных несколькими задачами одновременно. Смысл этого действия заключается в том, чтобы разделить всё множество строк на несколько непересекающихся групп таким образом, чтобы строки с одинаковыми значениями некоторых колонок оказались бы в одной группе. Тогда каждая задача может независимо от других обрабатывать свою группу и получить правильный результат.
Такие колонки ещё называют, потому что они используются в качестве ключей для **hash shuffle** и последующих операторов.

В данном примере **hash shuffle** используется для того, чтобы сгруппировать необходимые строки в задаче, реализующей оператор `JOIN`. Он реализован как вычисление значения хеш-фукнции от всех ключевых колонок и далее остатка от деления этого значения на общее количество таких задач.

Но общее количество различных значений (кардинальность) хеш-функции не может превышать общего количества различных ключей в наших данных, поэтому даже если данных изначально много и все они распределены в хранилище равномерно, но количество ключей невелико и сильно отличается для разных значений ключей, у нас снова возможен перекос, который мы наблюдаем на это плане.

После применения условия `WHERE` из таблицы `region` остаётся всего одна строка, которая естественным образом попадает только в одну из созданных задач, о чём сигнализирует красный круг с отметкой `1` на входе стадии `2` (мы наблюдаем низкую кардинальность по ключу `r_regionkey`).

Из таблицы `nation` выбираются все 25 строк. Однако для ключа `n_regionkey` у нас есть всего 5 разных значений. Поэтому из всех 8 задач стадии `2` данные из таблицы `nation` приходят только в 5 (остальным задачам ничего не досталось). Все 8 задач выполняют оператор `JOIN`, однако непустой результат будет только у одной из них — именно той, в которую попала строка из таблицы `region`. Эта задача вернёт 5 выбранных строк таблицы `nation`, связанных с данным регионом. Поэтому на выходе стадии `2` тоже появляется красная отметка `1`.

## Множественные выходы {#multiout}

{% include [tpch-dataset-note.md](_includes/tpch-dataset-note.md) %}

Следующий случай выполнения запроса: одна таблица читается один раз, а результат расходится к двум (или более) потребителям, выше потоки снова сходятся.

Такую структуру удобно показывать **клонами** стадии: один экземпляр — основной, остальные — сокращённое представление и отдельный выход. Пример — TPC-H Q17, адаптированный под синтаксис {{ ydb-short-name }}:

```sql
SELECT SUM(l_extendedprice) / 7.0 AS avg_yearly
  FROM lineitem
  CROSS JOIN part
  CROSS JOIN (
    SELECT l_partkey, 0.2 * AVG(l_quantity) AS quantity_threshold
      FROM lineitem
      GROUP BY l_partkey
    ) AS threshold
    WHERE part.p_partkey = lineitem.l_partkey
      AND p_brand = 'Brand#35'
      AND p_container = 'LG DRUM'
      AND l_quantity < quantity_threshold
      AND part.p_partkey = threshold.l_partkey
```

![Множественные выходы](../../_assets/rts-tpch-q17.svg){inline=false}

```lineitem``` читается один раз в стадии ```0``` и используется в стадиях ```1``` и ```3```. На плане стадия ```0``` показана дважды:

- **Основной экземпляр** — полный набор метрик: входы, CPU, память.
- **Клоны** — тот же цвет, что у стадий хранения, и одна метрика: выход, ведущий в нужную нижестоящую стадию.

Выделение любого фрагмента стадии ```0``` подсвечивает все её копии; несколько выходов подсвечиваются независимо, видно, куда идёт каждый.

Оптимизатор может объединять и дедуплицировать не только чтение из хранилища, но и целые подструктуры — это хорошо видно, например, на TPC-H Q21.

## Составные структуры {#complex}

В примерах выше была одна структура выполнения. Если в одном вызове можно передать несколько выражений SQL, то сервер выполнит их по очереди, и на диаграмме появятся **несколько несвязанных** структур, каждая из которых представляет отдельное выполнение запроса.

```sql
SELECT count(*) FROM lineitem;
SELECT count(*) FROM orders;
```

![Два графа выполнения](../../_assets/rts-count-count.svg){inline=false}

Бывают случаи, когда **одно** SQL-выражение приводит к появлению сразу нескольких ветвей вычислений: оптимизатор может вынести часть операций в отдельный прекомпьют — самостоятельную цепочку, создающую промежуточный результат для последующего использования в основном процессе выполнения запроса. Например, так устроена TPC-H Q15 в синтаксисе {{ ydb-short-name }}:

```sql
$revenue0 = (
  SELECT l_suppkey AS supplier_no, sum(l_extendedprice * (1 - l_discount)) AS total_revenue
    FROM lineitem
    WHERE l_shipdate >= date('1996-01-01') AND l_shipdate < date('1996-01-01') + interval('P90D')
    GROUP BY l_suppkey
);

SELECT s_suppkey, s_name, s_address, s_phone, total_revenue
  FROM supplier
  CROSS JOIN $revenue0 AS revenu0
  CROSS JOIN (
    SELECT max(total_revenue) AS max_total_revenue
      FROM $revenue0
    ) as max_revenue
  WHERE s_suppkey = supplier_no AND total_revenue = max_total_revenue
  ORDER BY s_suppkey;
```

![Прекомпьют](../../_assets/rts-tpch-q15.svg){inline=false}

Связь прекомпьюта с основной структурой выполнения становится очевидной при выделении имени прекомпьюта или его выходного канала с результатом: одновременно подсвечивается и вход [оператора](../../concepts/glossary.md#operator), который использует этот результат. Такой вход отмечен символом ```P```. В данном примере это стадия ```3``` в верхней структуре; данные из прекомпьюта поступают в правую часть Map Join.

В тяжёлых запросах (в том числе из TPC-DS) прекомпьютов несколько; они могут выполняться последовательно или параллельно, результаты подмешиваются в следующие стадии или порождают новые прекомпьюты. Формат плана рассчитан на то, чтобы по схеме можно было восстановить реальную структуру выполнения и соответствие задач на кластере {{ ydb-short-name }}.
