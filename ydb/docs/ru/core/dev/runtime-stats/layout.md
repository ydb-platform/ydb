# Расположение информации на плане

## Отдельные регионы (колонки) {#regions}

Снова посмотрим на план элементарного запроса

```sql
SELECT count(*) FROM lineitem
```

![План выполнения запроса](../../_assets/rts-count-lineitem.svg){inline=false}

Визуально план отображается в "таблично-деревянной" структуре. Он поделён на отдельные вертикальные области (колонки), которые содержат следующую информацию:

- `ResultSet` cтруктура запроса
- `Operators` операторная статистика
- `Tasks` прогресс выполнения задач
- `Stages` детальная информация по стадиям
- `Timeline` временные графики

## Структура запроса {#structure}

Самая левая колонка таблицы показывает структуру запроса. Более подробно это обсуждается в разделе [Структура плана запроса](structure.md). Сейчас нам достаточно знать, что план преставляется в виде дерева и состоит из стадий. Горизонтальная строка в таблице содержит информацию об одной стадии.

Название этой колонки соответствует одному из следующих категорий запросов:

- `ResultSet` (как в данном примере) для запросов с возвращаемым результатом (начинаются с `SELECT`)
- `Sink` для модифицирующих запросов (`INSERT`, `UPDATE` или `DELETE`)
- `Precompute` для выполняющихся отдельно частей запросов, как описано в разделе [Прекомпьюты](structure.md#precomputes)

## Операторная статистика {#operators}

Каждая стадия может состоять из одного или нескольких операторов. Если оператор сообщает статистику, то она отображается в следующей колонке `Operators`.

## Прогресс выполнения задач {#progress}

Запросы в {{ ydb-short-name }} выполняеются в распределённом варианте и один и тот же шаг может выполняться одновременно несколькими отдельными задачами (tasks) на одном или нескольких узлах кластера.

Колонка `Tasks` показывает сколько всего задач было запланировано для данной стадии. Если навести курсор мыши на это число, то во всплывающем окне будет также показано, сколько задач уже завершило своё выполнение. Кроме этого прогресс завершения задач отображается прерывистой полосой в левой области колонки.

Кроме того, колонка раскрашивается в разные цвета согласно интенсивности потребления процессорного времени стадией, как это описано в разделе [Потребление CPU](metrics.md#cpu).

## Детальная информация по стадиям {#stages}

Колонка `Stages` содержит агрегированную информацию по всем задачам для данной стадии. Здесь отображается достаточно большое количество метрик, многие из которых описаны в разделе [Визуализация метрик](metrics.md). Их объединяет то, что все они показывают итоговые значения в какой-то момент времени (обычно - в конце выполнения запроса).

Одинаковые метрики нормируются между стадиями в одном и том же масштабе для быстрого понимания ,например, того, какая стадия потратила больше всех процессорного времени или памяти. Подробности изложены в разделе [Масштаб метрик](metrics.md#scale)

## Временные графики {#timeline}

Последняя колонка с временной шкалой отображает те же самые данные что и в предыдущей колоноке `Stages` в течение всего времени работы запросы. В этой колонке также показана временная сетка с адаптивным шагом для облегчения понимания того, как во времени работли отдельные шаги запроса. Продолжительность работы запроса показывается в сером прямоугольнике в левом верхнем углу этой колонки.

## Интерактивные возможности {#interactive}

{% note info %}

Описанная в этом разделе интерактивнпя функциональность непосредственно в документации недоступна. Чтобы воспользоваться этой функциональностью надо открыть изображение с визуализированным планов в отдельной странице браузера.

{% endnote %}

Для выполнения необходимой бизнес задачи запрос может получить большим и сложным. Соответственно его план тоже будет занимать много стадий и иметь сложную структуру.

Чтобы упростить понимание плана, визуальное представление предоставляет некоторые возможности для того, чтобы скрыть ненужные данные, оставив только важные части:

- У каждой стадии есть кнопка (../../_assets/rts-button-minus.svg) которая "скрывает" поддерево плана, начиная с этой стадии (а сама кнопка заменяется на (../../_assets/rts-button-plus.svg) и позволяет отобразить поддерево обратно)
- у каждой стадии есть кнопка (../../_assets/rts-button-up.svg) которая включает "slim" режим, стадия уменьшается по высоте до одной метрики (output), возврат в обычный режим происходит по любому действию на стадии
- у всего плана в левой верхней части есть кнопки (../../_assets/rts-button-up.svg) для того чтобы включить "slim" режим для всей стадии или (../../_assets/rts-button-down.svg) вернуть все стадии в обычный режим (при этом дополнительно раскрываются все поддеревья)
