# Структура плана запроса

## Стадии {#stages}

Ниже — более сложный запрос и его графическое представление:

```sql
SELECT count(*) cnt, l_orderkey, sum(l_quantity)
  FROM lineitem
  WHERE l_commitdate >= date("1993-01-01") and l_commitdate < date("1993-02-01")
  GROUP BY l_orderkey
  HAVING sum(l_quantity) > 200
  ORDER BY cnt DESC, l_orderkey
```

![Простая агрегация](../../_assets/rts-structure-1.svg){inline=false}

В примере три нумерованные вычислительные [стадии](../../concepts/glossary.md#processing-stage) ```0```, ```1``` и ```2``` и отдельная стадия у таблицы ```lineitem``` (стадии хранения не нумеруются). Любую стадию можно временно выделить на схеме; здесь выделена стадия ```1```.

{% note info %}

Интерактивные элементы на встроенных в документацию изображениях недоступны. Чтобы воспользоваться интерактивной функциональностью, откройте визуализацию плана в отдельной вкладке браузера.

{% endnote %}

На схеме — граф вычислений после компиляции запроса: логические операторы, затем физические, сгруппированные в стадии выполнения.

Стадия — неделимая единица плана; в распределённом {{ ydb-short-name }} она обычно выполняется несколькими параллельными задачами, в том числе на разных узлах. Данные идут от стадии к стадии и в общем случае образуют [DAG](https://ru.wikipedia.org/wiki/Ориентированный_ациклический_граф); в этом примере проще — линейная цепочка. Направление потока на графике — снизу вверх: от хранилища к результату запроса.

Стадии двух типов:

1. Выполнение в системе хранения ([колоночные таблицы](../../concepts/glossary.md#column-oriented-table) или [шарды](../../concepts/glossary.md#data-shard)) — более тёмная заливка.
2. Вычисление без хранения (compute) — светлее; такие стадии нумеруются подряд, с ```0```.

Вкратце по плану:

{% note info %}

Нумерованный список ниже на странице идёт **сверху вниз**, на графике стадии и операторы расположены **снизу вверх**. Сопоставлять пункты со схемой удобно, начиная с **нижней** части плана.

{% endnote %}

1. Таблица ```lineitem``` сканируется целиком: условие ```WHERE``` не опирается на [первичный ключ](../../concepts/glossary.md#primary-key).
2. Фильтр по ```l_commitdate``` выполняется в хранилище (pushdown предиката), чтобы сократить объём передаваемых данных.
3. В стадии ```0``` — предварительная агрегация с группировкой по ```l_orderkey``` (```GROUP BY```).
4. Данные перераспределяются (hash shuffle) в стадию ```1```; подробнее — в разделе [Каналы связи](#connections).
5. В стадии ```1``` — финальная агрегация, фильтр ```HAVING``` и локальная сортировка.
6. Стадия ```2``` сливает отсортированные потоки в один глобально упорядоченный результат.

{% note tip %}

Неочевидный элемент схемы можно подсветить курсором: у большинства узлов есть всплывающая подсказка. Это не замена разделам документации, но удобно для быстрой ориентации и метрик.

{% endnote %}

## Каналы связи {#connections}

Тот же запрос, но в фокусе — связи между стадиями. Стадии — вершины графа, [каналы связи](../../concepts/glossary.md#channels) — рёбра, по которым передаются данные.

Канал на диаграмме показан пятиугольником с направлением потока. В этом примере выделен канал ![](../../_assets/rts-hash-conn-u.svg) между стадиями ```0``` и ```1```.

У линейного плана из четырёх стадий три соединения:

1. ```E``` — чтение из хранилища (вне compute).
2. ```H``` — hash shuffle, разбиение по одному или нескольким полям (список полей — во всплывающей подсказке по пятиугольнику).
3. ```Me``` — слияние (merge) уже отсортированных потоков; для этого типа указываются поля сортировки.

![Каналы связи](../../_assets/rts-structure-2.svg){inline=false}

Соединение состоит из двух частей:

1. **Выходной канал** — исходящий трафик из стадии ```0```. Синяя стрелка вправо-вверх с номером стадии-получателя (здесь ```1```).
2. **Входной канал** — приём в стадию ```1```. Зелёная стрелка влево-вверх с номером стадии-отправителя (```0```).

На графике **успешно** завершённого запроса объём данных на выходе канала и на входе должен совпадать: {{ ydb-short-name }} гарантирует доставку между [узлами](../../concepts/glossary.md#node) без потерь и дубликатов. Остальные метрики по каналам могут различаться и дополняют картину выполнения.

Пока запрос выполняется, показатели входа и выхода могут расходиться из-за задержки буферов и асинхронного обновления статистики. На графике ошибочного запроса расхождения тоже возможны.

Входной канал сопоставляется оператору, который принимает данные. В простом случае у стадии один вход: поток из канала подаётся в первый по порядку обработки (нижний в списке) оператор. В сложных графах входов несколько; у оператора справа дублируется номер стадии-источника исходящего канала (здесь ```0```), он подсвечивается при выборе канала. Разбор на примере с ```JOIN``` — в [разделе ниже](#join).

Итог по выбранному каналу между ```0``` и ```1```:

1. Начинается в стадии ```0```, направлен в стадию ```1```.
2. Проходит через hash shuffle.
3. Входит в стадию ```1``` со стадии ```0```.
4. Подаётся в оператор агрегации.

Между стадией хранения и первой вычислительной стадией действуют те же логические каналы, но с иными деталями реализации, чем между двумя compute-стадиями (см. [выше](#stages)). Планировщик {{ ydb-short-name }} по возможности ставит связанные задачи рядом с шардами хранения, чтобы сократить сетевой трафик.

На схеме такие каналы похожи на межстадийные: другие оттенки, другие пиктограммы, без дублирования номера стадии хранения. Сейчас вычислительная стадия связана не более чем с одной стадией хранения; её данные всегда приходят в самый нижний оператор стадии, отдельная навигация не нужна.

## Объединение стадий {#join}

Ниже — запрос с ```JOIN``` двух таблиц; план уже не линейный.

```sql
PRAGMA ydb.OptimizerHints = 'JoinType(nation region shuffle)';

SELECT n_name
  FROM nation
  JOIN region ON nation.n_regionkey == region.r_regionkey
  WHERE r_name = "AMERICA"
```

Таблицы ```region``` и ```nation``` небольшие (5 и 25 строк), запрос быстрый, трафик мал, но структура плана показательна. Для примера задан **Shuffle Join** через подсказку оптимизатора. Без этой прагмы план будет другим: при малом объёме данных оптимизатор выберет иной план (его можно сравнить, убрав ```PRAGMA```).

![Объединение стадий](../../_assets/rts-nation-region.svg){inline=false}

Стадия ```2``` (на схеме изначально выделена) принимает данные от стадий ```0``` и ```1```. Стадии по-прежнему упорядочены по вертикали, но связи не линейны: применяется топологическая сортировка — выход стадии ведёт ко входу стадии **выше** на графике, поток по-прежнему снизу вверх. Если у стадии несколько предшественников, слева появляется отступ, входные ```H```-каналы рисуются пятиугольником ![](../../_assets/rts-hash-conn-l.svg) влево; здесь таких соединений два.

Данные сходятся в операторе InnerJoin (эквивалент ```JOIN``` в тексте запроса). Указаны ключи ```n_regionkey``` и ```r_regionkey```; справа — номера стадий ```0``` и ```1```.

Красные круги с числами ```1``` и ```5``` — напоминание из раздела [Агрегаты](metrics.md#aggregates): число ненулевых метрик (здесь — по трафику в каналах) меньше числа задач стадии. Часть задач не получила данных; причину стоит разобрать, это часто симптом перекоса или избыточного параллелизма.

Два типичных объяснения на этой схеме:

1. **Мало данных.** В ```Tasks``` видно: у стадий хранения параллелизм 1 (по одному шарду на ```region``` и ```nation```). У соответствующих compute-стадий ```0``` и ```1``` создано по 3 задачи; при большом объёме нагрузка разошлась бы между ними. Здесь все строки уместились в один пакет из колоночного шарда в **одну** из трёх задач, остальные задачи не отчитались в статистике по этому каналу.

2. **Hash shuffle для ```JOIN```.** Строки с одинаковым ключом с **обоих** входов должны оказаться в **одной** задаче стадии соединения. Номер задачи задаётся детерминированно хешем ключей; число различных хешей не превышает число различных значений ключа. После ```WHERE``` из ```region``` остаётся одна строка — трафик уходит в одну задачу, метка ```1``` на входе стадии ```2``` согласуется с низкой кардинальностью ```r_regionkey```.

Из ```nation``` участвуют все значения ```n_regionkey```; при 8 задачах стадии ```2``` данные из ```nation``` приходят только в 5 (остальным задачам строк не досталось). ```JOIN``` завершает одна задача — та, в которую попала строка из ```region```; она формирует 5 строк результата. На выходе стадии ```2``` снова отметка ```1``` — по той же логике счётчиков метрик, что в [Агрегаты](metrics.md#aggregates).

## Множественные выходы {#multiout}

[Выше](#stages) говорилось, что граф — DAG. До сих пор это была линия или дерево. Следующий случай — **ромб**: одна таблица читается один раз, а результат расходится к двум (или более) потребителям, выше потоки снова сходятся.

Такую топологию на экране удобно показывать **клонами** стадии: один экземпляр — основной, остальные — сокращённое представление и отдельный выход. Пример — TPC-H Q17, адаптированный под синтаксис {{ ydb-short-name }}:

```sql
SELECT SUM(l_extendedprice) / 7.0 AS avg_yearly
  FROM lineitem
  CROSS JOIN part
  CROSS JOIN (
    SELECT l_partkey, 0.2 * AVG(l_quantity) AS quantity_threshold
      FROM lineitem
      GROUP BY l_partkey
    ) AS threshold
    WHERE part.p_partkey = lineitem.l_partkey
      AND p_brand = 'Brand#35'
      AND p_container = 'LG DRUM'
      AND l_quantity < quantity_threshold
      AND part.p_partkey = threshold.l_partkey
```

![Множественные выходы](../../_assets/rts-tpch-q17.svg){inline=false}

```lineitem``` читается один раз в стадии ```0``` и используется в стадиях ```1``` и ```3```. На плане стадия ```0``` показана дважды:

- **Основной экземпляр** — полный набор метрик: входы, CPU, память.
- **Клоны** — тот же цвет, что у стадий хранения, и одна метрика: выход, ведущий в нужную нижестоящую стадию.

Выделение любого фрагмента стадии ```0``` подсвечивает все её копии; несколько выходов подсвечиваются независимо, видно, куда идёт каждый.

Оптимизатор может объединять и дедуплицировать не только чтение из хранилища, но и целые подграфы — это хорошо видно, например, на TPC-H Q21.

## Составные графы {#complex}

В примерах выше был один граф вычислений. В одном вызове можно передать несколько выражений SQL — сервер выполнит их по очереди, на диаграмме появятся **несколько несвязанных** графов:

```sql
SELECT count(*) FROM lineitem;
SELECT count(*) FROM orders;
```

![Два графа выполнения](../../_assets/rts-count-count.svg){inline=false}

Сложнее, когда **одно** SQL-выражение порождает несколько графов: оптимизатор может вынести часть в **прекомпьют** — отдельный граф с промежуточным результатом, от которого строится основной граф. Пример — TPC-H Q15 в синтаксисе {{ ydb-short-name }}:

```sql
$revenue0 = (
  SELECT l_suppkey AS supplier_no, sum(l_extendedprice * (1 - l_discount)) AS total_revenue
    FROM lineitem
    WHERE l_shipdate >= date('1996-01-01') AND l_shipdate < date('1996-01-01') + interval('P90D')
    GROUP BY l_suppkey
);

SELECT s_suppkey, s_name, s_address, s_phone, total_revenue
  FROM supplier
  CROSS JOIN $revenue0 AS revenu0
  CROSS JOIN (
    SELECT max(total_revenue) AS max_total_revenue
      FROM $revenue0
    ) as max_revenue
  WHERE s_suppkey = supplier_no AND total_revenue = max_total_revenue
  ORDER BY s_suppkey;
```

![Прекомпьют](../../_assets/rts-tpch-q15.svg){inline=false}

Связь прекомпьюта с основным графом видна по выделению имени прекомпьюта или его исходящего канала с результатом: подсвечиваются и вход оператора, который читает этот результат. Такой вход помечен символом ```P```. В примере это стадия ```3``` верхнего графа; значение из прекомпьюта идёт в правую часть Map Join.

В тяжёлых запросах (в том числе из TPC-DS) прекомпьютов несколько; они могут выполняться последовательно или параллельно, результаты подмешиваются в следующие стадии или порождают новые прекомпьюты. Формат плана рассчитан на то, чтобы по схеме можно было восстановить реальную структуру выполнения и соответствие задач на кластере {{ ydb-short-name }}.
