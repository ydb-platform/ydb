# Анти-паттерны при партицировании {#anti-patterns}

Типичные ошибки при проектировании ключа и настроек партиционирования:

* **Низкая кардинальность первого компонента первичного ключа** — например, ключ `(status, id)`, где `status` почти всегда один и тот же: почти все строки попадают в один диапазон, одна партиция становится «горячей», автопартицирование не может равномерно разнести нагрузку.

* **Горячий ключ** — большая часть записей уходит в узкий диапазон ключей (один пользователь, один shard времени и т.д.). Даже при включённом [`AUTO_PARTITIONING_BY_LOAD`](../auto/by-load.md) эффективность ограничена: полезно пересмотреть модель ключа и схему доступа; см. [{#T}](../../../primary-key/row-oriented.md).

* **Игнорирование лимита `AUTO_PARTITIONING_MAX_PARTITIONS_COUNT`** — при достижении потолка таблица перестаёт делиться, рост задержек и размера партиций неизбежен, пока лимит не увеличат.

Страница [{#T}](default-no-settings.md) описывает отдельный случай: оставить настройки партиционирования по умолчанию без явной конфигурации.
