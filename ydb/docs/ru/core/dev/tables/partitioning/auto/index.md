# Автоматическое партицирование {#automatic-partitioning}

Для строковых таблиц {{ ydb-short-name }} может автоматически **разделять** (split) и **объединять** (merge) партиции, чтобы подстроиться под объём данных и интенсивность запросов. Это задаётся параметрами `AUTO_PARTITIONING_*` в описании таблицы (см. [{#T}](../../../../concepts/datamodel/table.md#partitioning_row_table)).

Типичное время операции разделения или слияния — порядка **500 мс**; в этот интервал данные по затронутой партиции могут быть кратковременно недоступны для чтения и записи (подробности — в [{#T}](../../../../concepts/datamodel/table.md#partitioning_row_table)).

Ограничения автоматического партицирования (лимиты базы и таблицы, очередь split/merge) — в [{#T}](limits.md).

## По объёму (size-based) {#auto-by-size}

Режим **`AUTO_PARTITIONING_BY_SIZE`** включает разделение партиции, когда её размер превышает заданный порог **`AUTO_PARTITIONING_PARTITION_SIZE_MB`**, и объединение соседних партиций, когда суммарный размер достаточно мал (см. точные условия в описании параметров в [`AUTO_PARTITIONING_BY_SIZE`](../../../../concepts/datamodel/table.md#auto_partitioning_by_size)).

Практические ориентиры:

* Рекомендуемый диапазон **`AUTO_PARTITIONING_PARTITION_SIZE_MB`** в документации по умолчанию ориентирован на значения порядка **от десятков мегабайт до 2000 MB**; конкретное значение зависит от профиля нагрузки и допустимой частоты split/merge.
* Слишком **большой** порог при неравномерном доступе к ключам приводит к «тяжёлым» партициям и сильнее нагружает один data shard; слишком **маленький** — к частым split/merge.

Независимо от пользовательского порога, в продукте также участвует внутренняя логика, завязанная на ориентир **~2000 MB** для части решений о разделении — см. [{#T}](../../../../concepts/datamodel/table.md#partitioning_row_table).

При изменении порога на **уже большой таблице** избегайте резкого скачка (например, с 2000 MB до 100 MB сразу): это может вызвать волну операций split. Лучше снижать порог **постепенно** и мониторить стабилизацию; см. также [{#T}](../../../../troubleshooting/performance/schemas/splits-merges.md).

Одновременно на таблице выполняется ограниченное число операций split — остальные партиции ждут в очереди. Подробнее — в [{#T}](limits.md).

## По нагрузке (load-based) {#auto-by-load}

Режим **`AUTO_PARTITIONING_BY_LOAD`** позволяет разделять партицию, когда она **перегружена по CPU**, даже если размер партиции ещё не большой. По умолчанию для строковых таблиц этот режим **выключен** (`DISABLED`); для высокой конкурентной записи/чтения по узкому диапазону ключей его обычно имеет смысл **включить** вместе с [`AUTO_PARTITIONING_MIN_PARTITIONS_COUNT`](../../../../concepts/datamodel/table.md#auto_partitioning_min_partitions_count), выставленным исходя из ожидаемого числа партиций, чтобы избежать их излишнего сжатия при спаде нагрузки.

Каноническое описание порогов и алгоритма (сэмпл ключей, учёт реплик, merge при низкой загрузке) — в [`AUTO_PARTITIONING_BY_LOAD`](../../../../concepts/datamodel/table.md#auto_partitioning_by_load).

### Типичные симптомы перегруженной партиции

* высокая загрузка CPU на отдельных партициях при умеренной общей загрузке кластера;
* рост задержек на «горячих» ключах;
* ошибки вида **`STATUS_OVERLOADED`** при записи в перегруженную партицию.

Один data shard использует **не более одного ядра CPU** для операций изменения и чтения данных на партиции: простое добавление CPU на узел не снимает bottleneck с одной партиции — нужны split и/или ключ, лучше способствующий распределению нагрузки по шардам.
