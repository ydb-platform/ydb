# Ограничения автошардирования {#auto-sharding-limits}

«Автошардирование» здесь означает автоматические **split/merge** партиций строковой таблицы. На практику накладываются лимиты уровня **базы данных** и **таблицы**, а также внутренние ограничения алгоритма.

## Лимиты уровня базы данных

* **Максимальное число таблеток в базе** (`MaxShards`) — при достижении лимита автоматическое разделение партиций прекращается. См. [{#T}](../../../../concepts/limits-ydb.md#schema-object).
* **Максимальное число шардов таблицы** (`MaxShardsInPath`) — верхняя граница числа партиций (data shard) на одном пути таблицы, включая индексы.
* **Жёсткий лимит размера партиции на уровне кластера** — по умолчанию **2 ГБ** (`ForceShardSplitDataSize`). Если размер партиции превышает этот порог, она разделится, даже если пользовательский [`AUTO_PARTITIONING_PARTITION_SIZE_MB`](../../../../concepts/datamodel/table.md#auto_partitioning_partition_size_mb) выше или достигнут [`AUTO_PARTITIONING_MAX_PARTITIONS_COUNT`](../../../../concepts/datamodel/table.md#auto_partitioning_max_partitions_count). Подробнее — в [{#T}](../../../../concepts/datamodel/table.md#partitioning_row_table).

## Параметры таблицы

Пока число партиций находится в коридоре [`AUTO_PARTITIONING_MIN_PARTITIONS_COUNT`](../../../../concepts/datamodel/table.md#auto_partitioning_min_partitions_count) — [`AUTO_PARTITIONING_MAX_PARTITIONS_COUNT`](../../../../concepts/datamodel/table.md#auto_partitioning_max_partitions_count), {{ ydb-short-name }} может **разделять** и **объединять** партиции; при достижении максимума разделение останавливается — данные и нагрузка могут расти за счёт размера партиций и нагрузки на shard.

**Рекомендуемая разница между min и max:** если разница между минимумом и максимумом **больше ~20%**, при сильно меняющейся во времени нагрузке компонента [Hive](../../../../concepts/glossary.md#hive) возможны лишние циклы split/merge. Подробнее и что делать — в [{#T}](../../../../troubleshooting/performance/schemas/splits-merges.md).

## Скорость адаптации по нагрузке

При включённом [`AUTO_PARTITIONING_BY_LOAD`](index.md#auto-by-load):

* решение о разделении партиции по нагрузке принимается **не быстрее чем за две минуты**;
* за одну операцию партиция разделяется **не более чем на две** части.

При значительном или многократном росте нагрузки адаптация партицирования может занимать **единицы и десятки минут**, в течение которых перегруженные партиции могут отвечать ошибками `OVERLOADED` — как из-за перегрузки, так и из-за самого процесса разделения.

Разделение и объединение партиций включают **компакшен** данных партиции — дополнительную нагрузку на диск.

У [scheme shard](../../../../concepts/glossary.md#scheme-shard) на принятие решения о разделении таблетки может уходить порядка **15 секунд** (см. [{#T}](../../../../troubleshooting/performance/schemas/splits-merges.md)).

## Очередь операций split/merge

Одновременно на **таблице** может выполняться ограниченное число операций split — остальные партиции ждут в очереди. Дополнительно Scheme shard ограничивает общее число параллельных split/merge в базе.

## Вторичные индексы

Параметры партиционирования задаются для **таблицы**; индекс имеет свой набор партиций и свои правила по умолчанию — см. [{#T}](../../../../concepts/datamodel/table.md#partitioning_row_table).

{% note warning %}

Колоночные таблицы: автоматическое репартицирование по тем же правилам **не поддерживается** — число партиций задаётся при проектировании таблицы. См. [{#T}](../../../primary-key/column-oriented.md).

{% endnote %}
