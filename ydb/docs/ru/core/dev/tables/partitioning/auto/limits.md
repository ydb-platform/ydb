# Ограничения автошардирования {#auto-sharding-limits}

«Автошардирование» здесь означает автоматические **split/merge** партиций строковой таблицы. На практику накладываются:

1. **Параметры таблицы** [`AUTO_PARTITIONING_MIN_PARTITIONS_COUNT`](../../../../concepts/datamodel/table.md#auto_partitioning_min_partitions_count) и [`AUTO_PARTITIONING_MAX_PARTITIONS_COUNT`](../../../../concepts/datamodel/table.md#auto_partitioning_max_partitions_count)**:** пока число партиций внутри допустимого коридора, {{ ydb-short-name }} может делить и склеивать партиции; при достижении потолка деление останавливается — данные и нагрузка могут расти за счёт размера партиций и нагрузки на shard.

2. **Рекомендуемая разница между min и max:** если разница между минимумом и максимумом **больше ~20%**, при сильно меняющейся во времени нагрузке компонента [Hive](../../../../concepts/glossary.md#hive) возможны лишние циклы split/merge. Подробнее и что делать — в [{#T}](../../../../troubleshooting/performance/schemas/splits-merges.md).

3. **Внутренние задержки решений:** у scheme shard может уходить порядка **15 секунд** на принятие решения о разделении таблетки; это влияет на динамику при перегрузках (см. troubleshooting выше).

4. **Колоночные таблицы:** автоматическое репартицирование по тем же правилам **не поддерживается** — число партиций задаётся при проектировании таблицы.

5. **Вторичные индексы:** параметры партиционирования задаются для **таблицы**; индекс имеет свой набор партиций и свои правила по умолчанию — см. [{#T}](../../../../concepts/datamodel/table.md#partitioning_row_table).
