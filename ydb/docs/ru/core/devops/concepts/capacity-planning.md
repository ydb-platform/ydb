# Планирование ёмкости хранилища

Из данной статьи вы узнаете, как оценить количество оборудования для хранения заданного объёма данных с учётом репликации и других накладных расходов, и как понять, какой полезный объём данных поместится на заданное количество оборудования.

## Модель распределённого хранилища {#storage-model}

На уровне оборудования кластер {{ ydb-short-name }} включает в себя некоторое количество физических дисков (HDD / SSD / NVMe). Остальные компоненты (CPU, RAM, сеть) в данной статье не рассматриваются.

Кластеры {{ ydb-short-name }} являются мультитенантными, и один физический диск может совместно использоваться несколькими потребителями — изолированными базами данных. Поверх каждого физического диска запущено по одному компоненту для работы с ним — [PDisk](../../concepts/glossary.md#pdisk). Ресурсы одного PDisk (IOPS, полоса на чтение и на запись, емкость) делятся на равные доли, именуемые [слотами](../../concepts/glossary.md#slot). Слоты резервируются для создания логических (виртуальных) дисков [VDisk](../../concepts/glossary.md#vdisk). За каждым VDisk закреплён один слот.

Для управления ресурсами пользователю доступна сущность [группы хранения](../../concepts/glossary.md#storage-group). Группа хранения представляет собой совокупность из нескольких VDisk на разных серверах, обеспечивающих отказоустойчивость наподобие RAID. Количество VDisk в группе хранения определяется режимом избыточного кодирования (erasure coding):

| Режим | Дата-центры | VDisk в группе хранения |
| --- | --- | --- |
| `block-4-2` | 1 | 8 |
| `mirror-3-dc` | 3 | 9 |
| `mirror-3-of-4` | ? | ? |
| `none` | 1 | 1 |

Подробнее о доступных режимах хранения и гарантиях отказоустойчивости см. [{#T}](../../concepts/topology.md).

С точки зрения пользователя группу хранения можно воспринимать как фиксированный набор ресурсов: IOPS, Read/Write Throughput, Space. Каждой базе данных принадлежит одна или более групп хранения. Каждая группа хранения принадлежит одной определённой базе данных.

Пользователями распределённого хранилища являются [таблетки](../../concepts/glossary.md#tablet) (от англ. tablet) — специальные сущности, каждая из которых управляет фрагментом пользовательских или системных данных. Каждая таблетка может записывать данные в несколько групп хранения. В каждую группу хранения пишут множество таблеток.

## Используемые величины {#metrics}

**Tablet Storage Used** — это суммарный объём данных, хранимый таблетками базы, включая сами данные и индексы.

**Distributed Storage Used** — это суммарный размер, аллоцированный VDisk'ами базы. Этот размер включает в себя:

- собственно данные, записанные таблетками;
- избыточность при репликации или кодировании с исправлением ошибок;
- служебные метаданные и журналы;
- аллоцированное, но реально не используемое место.

Для существующих баз эти значения можно посмотреть в {{ ydb-short-name }} UI на странице базы на вкладке Info → Storage.

Распределённое хранилище реализует сложную отложенную сборку мусора, поэтому в любой момент времени часть занятого места приходится на устаревшие данные, ещё не удалённые сборщиком.

## Оценка накладных расходов {#overhead}

**Redundancy Factor**, `RF` — фактор избыточности.

Для режима `block-4-2` (один дата-центр) `RF = 1.5`. Для режима `mirror-3-dc` (три дата-центра) в штатном режиме работы `RF = 3` (данные хранятся в 3 копиях). Однако при недоступности одного дата-центра данные пишутся в 4 копии и нагрузка на каждый из оставшихся дата-центров удваивается. Для планирования длительных отказов следует закладывать `RF = 6`.

Другие накладные расходы могут существенно варьироваться в зависимости от характера хранимых данных и нагрузки. Для грубой оценки сверху рекомендуется считать `Overhead = ???`.

Связать эти величины можно приблизительно через соотношение:

```text
Distributed Storage Used = Tablet Storage Used × RF × Overhead
```

## Оценка требуемого оборудования {#hardware-estimation}

{% note warning %}

Формулы ниже применимы для дисков достаточно большого объёма (от 800 ГБ). Использование дисков меньшего размера не рекомендуется, если необходимо достижение оптимальной производительности. Подробнее о требованиях к дисковой подсистеме читайте в разделе [{#T}](system-requirements.md).

{% endnote %}

<!--
1 (SysLog) + 5 (SysReserveSize) + 200 (MaxCommonLogChunks) = 206 chunks (128 MiB each) ≈ 28.4 GB
-->

Часть ёмкости PDisk зарезервирована для системных нужд, суммарно около 28.4 GB. Остальное место поровну распределено между слотами. Количество слотов на одном PDisk должно определяться исходя из параметров оборудования. Для оценки вслепую можно полагать `ExpectedSlotCount = 8` для HDD-дисков и `16` для SSD/NVMe. Исходя из этого размер слота можно определить по формуле:

```text
SlotSize = (DriveSize - 28.4 GB) / ExpectedSlotCount
```

Чтобы оценить необходимое количество групп хранения, требуется также учесть, что при заполнении любого из VDisk на 87% вся группа хранения получает предупреждение «место скоро закончится» (Capacity Alert = Cyan). На 90% включается логика по снижению пишущей нагрузки и таблетки переназначаются на другие группы хранения. Таким образом общее количество групп хранения в кластере можно оценить как:

```text
TotalGroups = Distributed Storage Used / (SlotSize × VDisksInGroup × 0.87)
```

Здесь `VDisksInGroup = 8` для `block-4-2` и `9` для `mirror-3-dc`.

Помимо этого, в кластере должен оставаться запас пустых слотов для нормальной работы механизма [SelfHeal](../../maintenance/manual/selfheal.md), выполняющего автоматическую реконфигурацию групп хранения для замены вышедших из строя или длительное время недоступных дисков. Минимально необходимое количество пустых слотов `EmptySlots` можно оценить как сумму максимального количества слотов в одной стойке ([домене отказа](../../concepts/glossary.md#fail-domain)) плюс 2.7% от общего количества слотов в кластере.

```text
TotalSlots = TotalGroups × VDisksInGroup + EmptySlots
```

Таким образом количество физических дисков можно вычислить как:

```text
TotalPDisks = TotalSlots / ExpectedSlotCount
```

## Оценка полезной ёмкости хранилища {#usable-capacity}

Обратная задача — оценить, какой объём полезных данных можно разместить на заданном количестве оборудования.

Пусть имеется `TotalPDisks` физических дисков ёмкостью `DriveSize` GB каждый. Тогда:

1. Общее количество слотов:

    ```text
    TotalSlots = TotalPDisks × ExpectedSlotCount
    ```

2. Количество слотов, доступных для групп хранения (за вычетом запаса на SelfHeal):

    ```text
    UsableSlots = TotalSlots - EmptySlots
    ```

    Здесь `EmptySlots` оценивается как сумма максимального количества слотов в одном домене отказа плюс 2.7% от `TotalSlots`.

3. Количество групп хранения:

    ```text
    TotalGroups = UsableSlots / VDisksInGroup
    ```

4. Полезная ёмкость распределённого хранилища с учётом порога заполнения 87%:

    ```text
    DistributedStorageCapacity = TotalGroups × VDisksInGroup × SlotSize × 0.87
    ```

5. Оценка полезного объёма данных:

    ```text
    TabletStorageCapacity = DistributedStorageCapacity / (RF × Overhead)
    ```

## Пример расчёта {#example}

Допустим, необходимо разместить 10 ТБ полезных данных в кластере с режимом отказоустойчивости `block-4-2`, используя SSD-диски ёмкостью 1 ТБ (1000 ГБ).

**Исходные параметры:**

| Параметр | Значение |
| --- | --- |
| Tablet Storage Used | 10 ТБ = 10 000 ГБ |
| Режим | `block-4-2` |
| RF | 1.5 |
| Overhead | 3 |
| DriveSize | 1000 ГБ |
| ExpectedSlotCount | 16 |
| VDisksInGroup | 8 |

**Расчёт:**

1. Distributed Storage Used = 10 000 × 1.5 × 3 = **45 000 ГБ**
2. SlotSize = (1000 − 28.4) / 16 ≈ **60.7 ГБ**
3. TotalGroups = 45 000 / (60.7 × 8 × 0.87) ≈ **107 групп хранения**
4. TotalSlots (без запаса) = 107 × 8 = **856 слотов**
5. TotalPDisks (без запаса) = 856 / 16 = **54 диска**
6. При 54 дисках и 8 стойках получается ~7 дисков на стойку, макс. слотов в одном домене отказа = 7 × 16 = **112 слотов**
7. EmptySlots ≈ 112 + 0.027 × 856 ≈ 112 + 23 = **135 слотов**
8. TotalSlots = 856 + 135 = **991 слот**
9. TotalPDisks = 991 / 16 ≈ **62 диска** (при 8 стойках — по 8 дисков на стойку)
10. Пересчёт EmptySlots: макс. слотов в домене отказа = 8 × 16 = 128, EmptySlots ≈ 128 + 0.027 × 991 ≈ 128 + 27 = **155 слотов**
11. TotalSlots = 856 + 155 = **1011 слотов**
12. TotalPDisks = 1011 / 16 ≈ **64 диска**

Таким образом, для хранения 10 ТБ полезных данных в режиме `block-4-2` потребуется около 64 SSD-дисков ёмкостью 1 ТБ, распределённых по 8 или более серверным стойкам.
