# Настройка аутентификации и авторизации узлов баз данных

Аутентификация узлов баз данных — это проверка подлинности при соединении узлов кластера друг с другом по протоколу gRPC.
Авторизация — это проверка, есть ли у узла нужные полномочия, в том числе для регистрации в кластере и доступа к управлению конфигурацией.

В совокупности эти механизмы нужны для того, чтобы гарантировать, что только доверенные члены кластера могут подключаться друг к другу.

Рекомендуется включать эти механизмы во всех кластерах {{ ydb-short-name }}, чтобы снизить риск несанкционированного доступа через подключение чужих узлов. Использование кластера без аутентификации и авторизации допустимо только в полностью изолированных тестовых окружениях.

## Как проходит проверка узла

Ниже — последовательность, в которой узел проходит аутентификацию и авторизацию:

1. Запускаемый узел базы данных открывает gRPC-подключение к одному из узлов хранения, указанных в опции `--node-broker`. Подключение выполняется по TLS, а в качестве клиентского сертификата используется сертификат запускаемого узла, заданный в опции `--grpc-cert`.
2. Узел хранения и узел базы данных выполняют взаимную TLS-проверку: проверяется цепочка доверия сертификата и соответствие имени хоста значению поля "Subject Name" сертификата.
3. Узел хранения проверяет поле "Subject" сертификата на соответствие требованиям, заданным в статической конфигурации в секции [client_certificate_authorization](../../../reference/configuration/client_certificate_authorization.md).
4. Если проверки успешны, подключение считается аутентифицированным, и со стороны узла хранения ему присваивается [SID](../../../concepts/glossary.md#access-sid), заданный в поле `member_groups` соответствующей записи `client_certificate_definitions`.
5. Узел базы данных использует это gRPC-подключение для регистрации в кластере и анонсирует свой сетевой адрес для связи с другими узлами.
6. Узел хранения проверяет, что SID, присвоенный подключению, содержится в списке `register_dynamic_node_allowed_sids`. Если проверка успешна, узел хранения регистрирует узел базы данных в кластере: назначает ему внутренний идентификатор (NodeId) и записывает соответствие между NodeId и сетевым адресом узла в реестр кластера.
7. Узел базы данных, подключаясь к другим узлам в кластере, передает свой SID, полученный ранее. Узлы кластера блокируют попытки подключения узлов с неизвестными сетевыми адресами или SID.

Режим аутентификации узлов при регистрации задаётся параметром [node_registration_token](../../../reference/configuration/auth_config.md#node-registration-token) в секции `auth_config`. По умолчанию используется значение `root@builtin` — встроенный токен, не требующий сертификатов. Пустая строка (`""`) переключает кластер на аутентификацию узлов по TLS-сертификатам. Режим с `root@builtin` допустим только в полностью изолированных тестовых окружениях.

## Настройка аутентификации и авторизации узлов баз данных

Чтобы включить аутентификацию и авторизацию узлов, перед изменением конфигурации кластера необходимо подготовить инфраструктуру сертификатов и настроить запуск узлов с использованием TLS-сертификатов.

1. Убедитесь, что в развернутом кластере {{ ydb-short-name }} [настроено шифрование gRPC-трафика](../../../reference/configuration/tls.md#grpc) с использованием TLS.
2. При выпуске сертификатов для узлов задайте единые правила заполнения поля "Subject", чтобы можно было отличать сертификаты узлов кластера. Подробности — в [документации по правилам проверки сертификатов](../../../reference/configuration/client_certificate_authorization.md).

    {% note info %}

    Пример [скрипта генерации самоподписанных сертификатов](https://github.com/ydb-platform/ydb/blob/main/ydb/deploy/tls_cert_gen/) для узлов {{ ydb-short-name }} заполняет поле "Subject" значением `O=YDB` для всех узлов. Примеры в этой статье рассчитаны именно на такой вариант заполнения поля "Subject".

    {% endnote %}

3. В параметры командной строки для [запуска узлов баз данных](./initial-deployment/deployment-configuration-v1.md#start-dynnode) добавьте опции с путями к сертификатам и ключу:

    | **Опция командной строки** | **Описание** |
    |----------------------------|--------------|
    | `--grpc-ca`                | Путь к файлу сертификатов доверенных центров сертификации `ca.crt`. |
    | `--grpc-cert`              | Путь к файлу сертификата узла `node.crt`. |
    | `--grpc-key`               | Путь к файлу секретного ключа узла `node.key`. |

    Пример запуска узла базы данных:

    ```bash
    /opt/ydb/bin/ydbd server --yaml-config  /opt/ydb/cfg/config.yaml --tenant /Root/testdb \
        --grpcs-port 2136 --grpc-ca /opt/ydb/certs/ca.crt \
        --grpc-cert /opt/ydb/certs/node.crt --grpc-key /opt/ydb/certs/node.key \
        --ic-port 19002 --ca /opt/ydb/certs/ca.crt \
        --mon-port 8766 --mon-cert /opt/ydb/certs/web.pem \
        --node-broker grpcs://<ydb1>:2135 \
        --node-broker grpcs://<ydb2>:2135 \
        --node-broker grpcs://<ydb3>:2135
    ```

## Включение режима аутентификации и авторизации узлов

Чтобы включить обязательную авторизацию узлов баз данных, обновите [статическую конфигурацию кластера](../../../reference/configuration/index.md):

1. На корневом уровне добавьте блок `client_certificate_authorization` с требованиями к полю "Subject" сертификатов подключаемых узлов. Пример:

    ```yaml
    client_certificate_authorization:
      request_client_certificate: true
      client_certificate_definitions:
        - member_groups: ["registerNode@cert"]
          subject_terms:
          - short_name: "O"
            values: ["YDB"]
    ```

    При необходимости добавьте дополнительные проверки сертификатов согласно [документации по `client_certificate_authorization`](../../../reference/configuration/client_certificate_authorization.md).

    В случае успешной проверки сертификата и соответствия компонентов его поля "Subject" требованиям, заданным в `subject_terms`, подключению будут присвоены субъекты доступа (SID), перечисленные в `member_groups`. Чтобы отделять такие субъекты от других групп и учётных записей, их принято именовать с суффиксом `@cert`.

2. В блоке `security_config` добавьте параметр `register_dynamic_node_allowed_sids` со списком SID, которым разрешена регистрация узлов баз данных. Пример:

    ```yaml
    domains_config:
        ...
        security_config:
            enforce_user_token_requirement: true
            ...
            register_dynamic_node_allowed_sids:
            - "root@builtin"
            - "registerNode@cert"
    ```

    Включение `root@builtin` в этот список обязательно по техническим причинам: некоторые внутренние процессы кластера используют его независимо от режима аутентификации узлов.

    Подробности о параметрах аутентификации кластера см. в [документации по `security_config`](../../../reference/configuration/security_config.md#security-access-levels).

3. В блоке `auth_config` присвойте параметру `node_registration_token` пустую строку:

    ```yaml
    auth_config:
      ...
      node_registration_token: ""
    ```

    Подробности о параметре `node_registration_token` см. в [документации по `auth_config`](../../../reference/configuration/auth_config.md#node-registration-token).

4. Обновите статическую конфигурацию на всех узлах кластера вручную или через [плейбук Ansible](../ansible/update-config.md).
5. Выполните поэтапный перезапуск узлов хранения [с помощью ydbops](../../../reference/ydbops/rolling-restart-scenario.md) или через [плейбук Ansible](../ansible/restart.md).
6. Выполните поэтапный перезапуск узлов баз данных кластера с помощью ydbops или плейбука Ansible.
