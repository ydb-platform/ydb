# Практические советы

Ниже кратко сформулированы типичные проблемы при работе с {{ ydb-short-name }} и рекомендуемые действия. Тексты основаны на внутренних заметках по эксплуатации и разработке с {{ ydb-short-name }}. Справа от формулировок даны перекрёстные ссылки на статьи документации по связанным понятиям.

## Балансировка и подключение {#balancer}

### Алгоритмы `PreferLocalDC` / `PreferNearestDC` {#tip-prefer-local-dc}

**Проблема.** Режимы «предпочитать локальный (или ближайший) ДЦ» концентрируют трафик и сессии в одном датацентре: неравномерная нагрузка, риск при rolling restart и учениях, а при «уезде» таблеток кросс-ДЦ задержки всё равно возрастают. Выигрыш по сети оправдан в основном при чтении с [реплик](../../concepts/datamodel/table.md#read_only_replicas) и специальных режимах запросов.

**Решение.** По умолчанию использовать [равномерную балансировку](../../recipes/ydb-sdk/balancing-random-choice.md) по узлам (`RandomChoice()` или эквивалент в SDK), если нет согласованной с командой {{ ydb-short-name }} стратегии с репликами. Общий обзор — в разделе [Балансировка](../../recipes/ydb-sdk/balancing.md); про режим [PreferLocalDC](../../recipes/ydb-sdk/balancing-prefer-local.md) — отдельно.

### Серверная балансировка сессий и `BAD_SESSION` {#tip-server-side-balancing}

**Проблема.** Игнорирование сигналов сервера (`shutdownHint`), долгое удержание одной сессии и отсутствие ретраев приводят к частым ошибкам [`BAD_SESSION`](../../reference/ydb-sdk/ydb-status-codes.md#bad-session) при перераспределении сессий.

**Решение.** Использовать официальные SDK: короткие операции через [пул сессий](../../recipes/ydb-sdk/session-pool-limit.md), обработка `shutdownHint` и ретраи `BAD_SESSION`, при необходимости — возможности клиента вроде `session-balancer`. См. также [клиентскую балансировку](../../recipes/ydb-sdk/balancing.md).

### Rolling restart кластера {#tip-rolling-restart}

**Проблема.** При поочерёдном перезапуске узлов возможны [`BAD_SESSION`](../../reference/ydb-sdk/ydb-status-codes.md#bad-session), рост задержек, сброс кэша планов на узлах и усугубление проблем из‑за неудачной клиентской балансировки.

**Решение.** Достаточная избыточность узлов по зонам (как правило, не меньше двух узлов на зону для стабильности), [равномерная балансировка](../../recipes/ydb-sdk/balancing-random-choice.md), [ретраи](../../recipes/ydb-sdk/retry.md) с учётом временных ошибок, разумные [таймауты](../../dev/timeouts.md) (в том числе относительно перцентилей времени ответа), при необходимости — прогрев сессий.

### Учения по отключению датацентра {#tip-drill-dc}

**Проблема.** При вводе и выводе из учений возможны всплески ошибок и ретраев, рост задержек, инвалидация сессий, «телепортация» клиентов между ДЦ при привязке к ближайшему ДЦ и переполнение пулов при каскаде повторов.

**Решение.** Идемпотентные операции помечать как идемпотентные ([политика повторов](../../recipes/ydb-sdk/retry.md)), настраивать [таймауты](../../dev/timeouts.md) и по возможности проброс дедлайнов, не держать сессии долго, при необходимости задавать предпочтение конкретного ДЦ вручную или временно отключать prefer-DC ([балансировка](../../recipes/ydb-sdk/balancing.md)), увеличивать пулы с запасом на пик.

### Ошибка подключения к IPv6 в Docker {#tip-docker-ipv6}

**Проблема.** Запуск контейнера {{ ydb-short-name }} без корректного hostname приводит к тому, что discovery отдаёт недоступный с клиента IPv6, и подключение падает с ошибкой вида `dial tcp: lookup [ipv6-address]`.

**Решение.** Задавать hostname контейнера (`docker run -h localhost` или осмысленное имя / IP), в Compose — поле `hostname`, чтобы в выдаче эндпойнтов были достижимые адреса. См. [конфигурацию Docker-образа](../../reference/docker/configuration.md) и [быстрый старт](../../quickstart.md).

## Ретраи и идемпотентность {#retry}

### Отказ от встроенных ретраев {#tip-no-retry}

**Проблема.** Без автоматических повторов временные сетевые и серверные сбои сразу «пробивают» приложение; ручная обработка всех кодов ошибок сложна и ненадёжна.

**Решение.** Использовать [политику повторов в SDK](../../recipes/ydb-sdk/retry.md), не ретраить заведомо неуспешные операции, не устраивать бесконечные и слишком частые циклы повторов. Классификация статусов — в [обработке ошибок](../../reference/ydb-sdk/error_handling.md).

### Самописные ретраеры {#tip-custom-retry}

**Проблема.** Самодельные циклы с фиксированной задержкой часто не различают типы ошибок, провоцируют перегрузку и не учитывают особенности {{ ydb-short-name }}.

**Решение.** Опираться на [ретраеры SDK](../../recipes/ydb-sdk/retry.md) с настраиваемой политикой.

### Бесконечный ретрай вручную {#tip-retry-all-errors}

**Проблема.** Оборачивание запроса в бесконечный цикл с `Sleep` маскирует ошибки, может создать лишнюю нагрузку и не использует встроенную классификацию ошибок.

**Решение.** Доверить повторы [SDK](../../recipes/ydb-sdk/retry.md) и ограничивать число попыток и backoff на уровне API драйвера.

### Временные объекты и замыкания в лямбде ретрая {#tip-closure-temp}

**Проблема.** Ретрай может выполнить лямбду несколько раз; результат запроса или накопленные срезы, вынесенные наружу, дают невалидные или продублированные данные.

**Решение.** Вся работа с `Result` и буферами — внутри лямбды; при необходимости сбрасывать накопители в начале каждой попытки. См. [повторы запросов](../../recipes/ydb-sdk/retry.md).

### Накопление результатов между попытками {#tip-no-cleanup}

**Проблема.** Глобальные или замыкаемые списки, в которые дописываются строки при каждом проходе лямбды, дублируют данные при ретрае.

**Решение.** Локальные структуры данных внутри попытки или явный сброс перед повторным выполнением ([ретраи](../../recipes/ydb-sdk/retry.md)).

### Стриминг и большие выборки {#tip-streaming}

**Проблема.** Повтор «с нуля» для стрима дорог; продолжить с места обрыва встроенно не всегда возможно.

**Решение.** Полагаться на API SDK для стримов, для больших объёмов — [постраничный вывод](../../dev/paging.md) по ключу или [паттерны потоковой обработки](../../dev/streaming-query/index.md).

### Флаг идемпотентности {#tip-idempotency}

**Проблема.** Без флага идемпотентности для безопасных операций повторы при `UNAVAILABLE` / дедлайнах могут не выполняться; ложная идемпотентность для записи приводит к дублям.

**Решение.** Помечать идемпотентными только действительно идемпотентные операции. Подробнее — [ретраи и флаг идемпотентности](../../recipes/ydb-sdk/retry.md).

### Вложенный `Query().Do` внутри `Query().Do` {#tip-retry-inside-retry}

**Проблема.** Второй вызов ретраера внутри лямбды может заблокировать пул сессий (`DEADLOCK` при ожидании сессии).

**Решение.** Внутри внешней лямбды вызывать только методы переданной сессии, без вложенных `Do`/`Retry` на тот же пул ([ретраи](../../recipes/ydb-sdk/retry.md), [пул сессий](../../recipes/ydb-sdk/session-pool-limit.md)).

## Таблицы, ключи и хранение {#tables}

### Монотонный `PRIMARY KEY` {#tip-monotonic-pk}

**Проблема.** Монотонно растущий ключ создаёт «горячую» последнюю партицию и узкое место по записи.

**Решение.** Избегать монотонии в первой компоненте ключа или добавлять префикс с хешем/равномерным распределением; для серийных типов — осознанное проектирование ключа. См. [проектирование первичного ключа](../../dev/primary-key/index.md) и [партиционирование](../../concepts/datamodel/table.md#partitioning_row_table).

### Настройки партиций по умолчанию {#tip-default-partitioning}

**Проблема.** Значения по умолчанию (в том числе без автосплита по нагрузке, один минимальный шард) при активной нагрузке и плохом ключе ведут к перегрузке шардов и частым split/merge.

**Решение.** Задавать осмысленные `AUTO_PARTITIONING_*`, минимальное число партиций больше одного при необходимости, включать [сплит по нагрузке](../../concepts/datamodel/table.md#auto_partitioning_by_load) для горячих таблиц. Описание параметров — в [партиционировании строковой таблицы](../../concepts/datamodel/table.md#partitioning_row_table).

### Автопартицирование по нагрузке {#tip-auto-load}

**Проблема.** Партиция использует ограниченно CPU; при перегрузке одного шарда масштабирование «вширь» без сплита не помогает.

**Решение.** Включать [`AUTO_PARTITIONING_BY_LOAD`](../../concepts/datamodel/table.md#auto_partitioning_by_load), задавать разумные min/max числа партиций и следить за метриками split/merge.

### Большие партиции {#tip-large-partitions}

**Проблема.** Слишком крупные шарды ухудшают latency и усложняют split; слишком частые split/merge дают фон операций.

**Решение.** Подбирать [`AUTO_PARTITIONING_PARTITION_SIZE_MB`](../../concepts/datamodel/table.md#auto_partitioning_partition_size_mb), согласовать min/max партиций и мониторить баланс ([партиционирование](../../concepts/datamodel/table.md#partitioning_row_table)).

### Отсутствие вторичных индексов {#tip-no-indexes}

**Проблема.** Фильтрация по неключевым полям без индекса приводит к полному сканированию таблицы.

**Решение.** Добавлять [вторичные индексы](../../concepts/query_execution/secondary_indexes.md) под частые предикаты, учитывая цену записи; практические шаги — в [работе с индексами](../../dev/secondary-indexes.md).

### Верхняя граница партиций у индексных таблиц {#tip-index-max-partitions}

**Проблема.** У индексов свой лимит партиций по умолчанию; при росте данных индекс перестаёт сплититься и становится узким местом.

**Решение.** Явно настраивать `AUTO_PARTITIONING_*` для индекса через `ALTER INDEX ... SET (...)` ([изменение таблицы и индексов](../../yql/reference/syntax/alter_table/index.md)).

### Компакшн после массовых удалений {#tip-compaction-delete}

**Проблема.** Удаление оставляет tombstone; при слабой фоновой компактификации растёт объём и страдает чтение.

**Решение.** Настраивать параметры background compaction под сценарий (частота, параллелизм, приоритет очередей удалений) и смотреть метрики. Удаление строк и [TTL](../../concepts/datamodel/table.md#ttl) описаны в модели таблицы.

### Избыточная нормализация {#tip-denormalization}

**Проблема.** Много `JOIN` в одной транзакции увеличивают конфликты и задержки.

**Решение.** Умеренная денормализация и таблицы «под запросы», чтобы сократить число соединений в горячих путях. См. [синтаксис JOIN](../../yql/reference/syntax/select/join.md) и [режимы транзакций](../../recipes/ydb-sdk/tx-control.md).

### Сжатие колонок {#tip-compression}

**Проблема.** Большие текстовые/JSON поля без сжатия быстро заполняют квоту хранения.

**Решение.** Использовать [группы колонок](../../concepts/datamodel/table.md#column-groups) с `COMPRESSION` (lz4, zstd и др.) по типу данных; синтаксис — в [семействах колонок](../../yql/reference/syntax/create_table/family.md).

## Запросы и YQL {#yql-tips}

### Огромные запросы от ORM {#tip-large-orm}

**Проблема.** Массовые `INSERT`/`IN` тысячами литералов упираются в [лимиты запроса](../../concepts/limits-ydb.md#query) и gRPC, долго компилируются.

**Решение.** Пакеты через параметры-списки (`AS_TABLE($rows)`), нативные bulk API интеграций; общие советы — в [загрузке данных](../../dev/batch-upload.md).

### Вставка при синхронных индексах {#tip-insert-sync-idx}

**Проблема.** [`BulkUpsert`](../../recipes/ydb-sdk/bulk-upsert.md) несовместим с синхронными индексами; тяжёлые транзакции чаще ловят TLI ([инвалидация блокировок](../../troubleshooting/performance/queries/transaction-lock-invalidation.md)).

**Решение.** Транзакционный `UPSERT` с параметром-таблицей, дробление порций, либо загрузка без индекса с добавлением индекса после ([вторичные индексы](../../dev/secondary-indexes.md)).

### SQL-инъекции {#tip-sql-injection}

**Проблема.** Конкатенация пользовательского ввода в текст YQL открывает обход фильтров и вредоносные запросы.

**Решение.** Только [параметризованные запросы](../../reference/ydb-cli/parameterized-queries-cli.md); значения передавать через параметры SDK.

### Нестабильный порядок строк без `ORDER BY` {#tip-order-by}

**Проблема.** Без `ORDER BY` порядок строк не гарантирован и может меняться после сплита/обновлений.

**Решение.** Явный `ORDER BY`, когда порядок важен для UI или логики ([сортировка в SELECT](../../yql/reference/syntax/select/index.md)).

### Устаревший Scripting Service {#tip-scripting}

**Проблема.** Scripting Service не развивается, хуже по диагностике и семантике DDL.

**Решение.** Query Service: в CLI — [`ydb sql`](../../reference/ydb-cli/sql.md), в SDK — клиенты `Query` вместо `Scripting` ([устаревшая `scripting yql`](../../reference/ydb-cli/scripting-yql.md)).

### Лимит 1000 строк у Table API {#tip-1000-rows}

**Проблема.** У `ExecuteDataQuery` ограничение на размер ответа ([строки и Table Service](../../concepts/limits-ydb.md#query)).

**Решение.** Переход на Query Service / стриминг `ExecuteQuery` ([`ydb sql`](../../reference/ydb-cli/sql.md)), [постраничный вывод](../../dev/paging.md) или обходные режимы чтения больших выборок по задаче.

### Параметризованные запросы {#tip-param-queries}

**Проблема.** Подстановка констант в текст даёт уникальный запрос на каждый набор значений и лишние компиляции.

**Решение.** Один текст запроса и параметры `$name` — для кэша планов и меньшей нагрузки на CPU ([параметризованные запросы в CLI](../../reference/ydb-cli/parameterized-queries-cli.md), [пример приложения на Go](../../dev/example-app/go/index.md)).

### `KeepInCache` и Table Service {#tip-keep-in-cache}

**Проблема.** У Table Service кэш планов по умолчанию выключен — повторные одинаковые запросы снова компилируются ([ограничения запросов](../../concepts/limits-ydb.md#query)).

**Решение.** Перейти на Query Service ([`ydb sql`](../../reference/ydb-cli/sql.md)) или включать `KeepInCache` и параметризацию на Table API.

### Покрывающие (covering) индексы {#tip-covering-index}

**Проблема.** По вторичному индексу находятся ключи, затем выполняется lookup в основную таблицу — лишние чтения.

**Решение.** Проектировать индекс так, чтобы нужные колонки были в индексе, если это оправдано по записи ([GLOBAL индексы](../../concepts/query_execution/secondary_indexes.md)).

### `LIKE` против поиска по префиксу {#tip-startswith}

**Проблема.** Даже `LIKE 'prefix%'` может увести оптимизатор в FullScan.

**Решение.** Для префикса — [`StartsWith`](../../yql/reference/builtins/basic.md#starts_ends_with) / сравнение диапазона, чтобы задействовать индекс.

### FullScan на растущих таблицах {#tip-fullscan-grow}

**Проблема.** С ростом таблицы полное сканирование линейно замедляется.

**Решение.** [Индексы](../../concepts/query_execution/secondary_indexes.md) под фильтры; при необходимости — `VIEW` нужного индекса ([`VIEW` и вторичный индекс](../../yql/reference/syntax/select/secondary_index.md)).

### Планы после изменения схемы {#tip-explain-schema}

**Проблема.** После добавления индекса оптимизатор может выбрать другой план — не всегда лучший.

**Решение.** Проверять план критичных запросов ([оптимизация по планам](../../dev/query-plans-optimization.md), [EXPLAIN в CLI](../../reference/ydb-cli/commands/explain-plan.md)) и закреплять ожидаемый план в тестах.

### `VIEW PRIMARY KEY` {#tip-view-pk}

**Проблема.** Rule-based оптимизатор иногда выбирает не тот индекс в сложных join.

**Решение.** Явно указывать `VIEW` для вторичного индекса или `VIEW PRIMARY KEY` для чтения по первичному ключу, когда выбор проверен ([вторичные индексы и автоматический выбор](../../dev/secondary-indexes.md)).

### Размер батча в `IN` {#tip-in-batch}

**Проблема.** Очень длинный `IN` может перейти в FullScan или упереться в [лимит ответа](../../concepts/limits-ydb.md#query).

**Решение.** Дробить списки (например, порции по тысяче), смотреть план, использовать адаптивный batch в CLI при необходимости ([планы запросов](../../dev/query-plans-optimization.md)).

### Несколько запросов подряд {#tip-multiple-queries}

**Проблема.** Каждый round-trip добавляет сетевую задержку к общему времени.

**Решение.** Объединять независимые операции в один YQL-запрос, если не нужна логика между шагами на клиенте ([несколько операций в скрипте](../../reference/ydb-cli/yql.md)).

### `DECLARE` и пустой список параметров {#tip-declare-empty}

**Проблема.** Явный `DECLARE $ids AS List<Uint64>` при фактическом `EmptyList` может давать ошибку типов.

**Решение.** Убрать избыточный `DECLARE`, если тип выводится из переданных параметров (актуально для новых версий сервера). См. [параметры в YQL](../../yql/reference/syntax/select/index.md).

### `INSERT` и `UPSERT` {#tip-insert-upsert}

**Проблема.** `UPSERT` без чтения быстрее на таблицах без индексов; с несколькими синхронными индексами выигрыш может быть у `INSERT`; семантика «вставить ровно один раз» требует `INSERT`.

**Решение.** Выбирать оператор по требованиям целостности и схеме ([вставка в YQL](../../dev/yql-tutorial/insert_into.md), [UPSERT](../../recipes/ydb-sdk/upsert.md)).

### Параметризованный запрос вместо Prepare {#tip-prepared-vs-param}

**Проблема.** Prepared statements привязаны к сессии, ломаются при балансировке и усложняют ретраи; все execute уходят на один узел.

**Решение.** Параметризованный `Execute` с серверным кэшем планов (`KeepInCache` / Query Service), без отдельного `Prepare` в долгоживущем виде ([ограничения и Query Service](../../concepts/limits-ydb.md#query), [`ydb sql`](../../reference/ydb-cli/sql.md)).

### kNN и эмбеддинги {#tip-knn-bytes}

**Проблема.** Передача вектора как `List<Float>` даёт лишние сериализации на пути в `Bytes`.

**Решение.** Передавать эмбеддинг сразу как `Bytes` в целевом формате ([векторные индексы](../../dev/vector-indexes.md)).

### `BulkUpsert` и синхронные индексы {#tip-bulk-upsert-sync}

**Проблема.** `BulkUpsert` не поддерживается при синхронных вторичных индексах.

**Решение.** `UPSERT ... SELECT FROM AS_TABLE($input)`, асинхронные индексы, отложенное создание индекса или загрузка без индекса с последующим построением ([пакетная вставка](../../recipes/ydb-sdk/bulk-upsert.md), [индексы](../../dev/secondary-indexes.md)).

### Параллельные запросы на одной сессии {#tip-session-busy}

**Проблема.** Сессия однопоточна; параллельные запросы или непрочитанный результат приводят к [`SESSION_BUSY`](../../reference/ydb-sdk/ydb-status-codes.md#session-busy) и лишним пересозданиям сессий.

**Решение.** Сериализовать работу на сессии и полностью читать поток результата ([пул сессий](../../recipes/ydb-sdk/session-pool-limit.md)).

## Транзакции {#tx}

### Явный `Begin` без нужды {#tip-begin-tx}

**Проблема.** Отдельный вызов начала транзакции добавляет лишний round-trip.

**Решение.** Начинать транзакцию первым запросом с нужным режимом ([управление транзакциями](../../recipes/ydb-sdk/tx-control.md)).

### Отдельный `Commit` в конце {#tip-commit-tx}

**Проблема.** Явный коммит после последнего запроса — лишний сетевой ход.

**Решение.** Коммит в последнем запросе транзакции там, где API это поддерживает ([транзакции в SDK](../../recipes/ydb-sdk/tx-control.md)).

### Долгие транзакции {#tip-long-tx}

**Проблема.** Длинные интерактивные транзакции повышают риск TLI и конфликтов ([оптимистичные блокировки](../../concepts/glossary.md#optimistic-locking)).

**Решение.** Укорачивать транзакции, не выполнять тяжёлую бизнес-логику внутри, коммитить последним запросом ([TLI](../../troubleshooting/performance/queries/transaction-lock-invalidation.md)).

### Много чтений в read-write транзакции {#tip-many-reads}

**Проблема.** Большие чтения в RW-транзакции тянут много блокировок и TLI.

**Решение.** Вынести широкое чтение в read-only снимок или разбить работу на меньшие транзакции ([режимы транзакций](../../recipes/ydb-sdk/tx-control.md), [TLI](../../troubleshooting/performance/queries/transaction-lock-invalidation.md)).

### Много записей в одной транзакции {#tip-many-writes}

**Проблема.** Большой объём записи в строковые таблицы может дать `Out of buffer memory` и тяжёлые распределённые транзакции.

**Решение.** Дробить порции, по возможности держать одну транзакцию на одном шарде, для массовой заливки — [`BulkUpsert`](../../recipes/ydb-sdk/bulk-upsert.md) где допустимо ([загрузка данных](../../dev/batch-upload.md)).

### TLI на идемпотентных операциях {#tip-tli-idempotent}

**Проблема.** Оптимистичные конфликты дают `Transaction Locks Invalidated`; без ретрая идемпотентная операция может «промахнуться».

**Решение.** Ретраи с backoff для идемпотентных путей ([ретраи](../../recipes/ydb-sdk/retry.md)), уменьшение объёма конфликтующих чтений, дробление транзакций ([статья про TLI](../../troubleshooting/performance/queries/transaction-lock-invalidation.md)).

## Топики {#topics}

### Лаг чтения {#tip-read-lag}

**Проблема.** Без мониторинга лага возможна потеря по retention, переполнение при важном потребителе и внезапная догрузка после простоя.

**Решение.** Алерты по метрикам лага и простоя чтения по партициям, профилактика ещё до критического отставания ([топики](../../concepts/datamodel/topic.md), [чтение в CLI](../../reference/ydb-cli/topic-read.md)).

### Порядок событий с одним ключом {#tip-keyed-write}

**Проблема.** Нужна строгая упорядоченность по бизнес-ключу (например, один счёт).

**Решение.** Стабильное сопоставление ключа партиции (пул писателей по числу партиций, хеш ключа, неизменное число партиций); избегать одного writer на все ключи ([запись в топик](../../reference/ydb-cli/topic-write.md)).

### Случайный `producerId` {#tip-producer-id}

**Проблема.** Новый случайный идентификатор на каждую отправку раздувает учёт на кластере, усложняет отладку и может конфликтовать с владением сессией.

**Решение.** Стабильные осмысленные `producerId` на инстанс/партицию, без лишнего размножения уникальных имён ([обзор топиков](../../reference/ydb-cli/topic-overview.md)).

## Общие рекомендации {#common-tips}

### gRPC keep-alive {#tip-grpc-keepalive}

**Проблема.** Слишком частые ping’и дают `too_many_pings` и обрывы; экстремальные настройки ухудшают старт и стабильность.

**Решение.** Оставлять дефолты SDK, менять только при измеренной необходимости; для serverless — согласовать интервалы ([обработка ошибок и транспорт](../../reference/ydb-sdk/error_handling.md)).

### Интеграционные тесты {#tip-integration-tests}

**Проблема.** Обновление {{ ydb-short-name }} может незаметно поменять поведение и перформанс.

**Решение.** Контейнерные тесты на критичных сценариях, при необходимости — несколько версий, изолированные схемы, быстрый CI ([пример приложения](../../dev/example-app/index.md)).

### Репродьюсер бага {#tip-repro}

**Проблема.** Тикет без воспроизведения растягивает диагностику.

**Решение.** Минимальный самодостаточный пример (код или тест), стабильно воспроизводящий проблему ([сообщения об ошибках](../../reference/ydb-sdk/error_handling.md)).

## YDB CLI {#cli}

### Интерактивный режим {#tip-cli-interactive}

**Проблема.** Запуск CLI на каждую команду теряет историю и автодополнение.

**Решение.** [Интерактивная сессия](../../reference/ydb-cli/interactive-cli.md) `ydb` с профилем или параметрами подключения.

### Профили подключения {#tip-cli-profiles}

**Проблема.** Длинные команды с endpoint, БД и токеном каждый раз — ошибки и утомление.

**Решение.** `ydb config profile create`, [активация профиля](../../reference/ydb-cli/profile/activate.md), переменные окружения для секретов ([профили](../../reference/ydb-cli/profile/index.md)).

### Команда `ydb yql` {#tip-ydb-sql}

**Проблема.** Устаревший путь вызывает старый API.

**Решение.** Использовать [`ydb sql`](../../reference/ydb-cli/sql.md) (Query Service); см. [устаревшие команды](../../reference/ydb-cli/scripting-yql.md).

## Конфигурация и эксплуатация {#configuration}

### Размер серверного кэша планов {#tip-query-cache}

**Проблема.** Слишком маленький кэш на узле вымывает планы и увеличивает CPU на компиляции.

**Решение.** Подстроить лимит кэша запросов в конфигурации узла под профиль нагрузки ([конфигурация](../../reference/configuration/index.md)).

### Уровень логирования в проде {#tip-logging}

**Проблема.** `DEBUG`/`TRACE` на постоянной основе перегружают CPU, диск и задержки.

**Решение.** Уровень порядка `NOTICE` для продакшена, точечное включение детализации, семплирование трассировки ([конфигурация логирования](../../reference/configuration/log_config.md)).

### TTL метрик мониторинга {#tip-metrics-ttl}

**Проблема.** Без TTL метрики бесконтрольно растут и раздувают шард мониторинга.

**Решение.** Настроить TTL в системе мониторинга, иерархию сервис → кластер → шард ([метрики](../../reference/observability/metrics/index.md)).

### Оверкоммит групп хранения {#tip-storage-overcommit}

**Проблема.** Неправильный `storage_overcommit` ломает балансировку каналов и приводит к недогрузке или перегрузу групп.

**Решение.** Начинать с `1.0`, повышать осторожно при пиках, мониторить [Hive](../../concepts/glossary.md#hive) и диски ([группы хранения](../../concepts/glossary.md#storage-group)).

### Интервал запуска TTL {#tip-ttl-interval}

**Проблема.** Слишком редкий запуск TTL откладывает физическое удаление; избыточно частый — лишняя нагрузка без выгоды.

**Решение.** Подбирать run-interval под объём и SLA на удаление данных ([TTL](../../concepts/ttl.md), [TTL таблицы в CLI](../../reference/ydb-cli/table-ttl-set.md)).

### Расхождение времени на узлах {{ ydb-short-name }} {#tip-time-ydb}

**Проблема.** Рассинхрон часов замедляет распределённые транзакции; при большом расхождении транзакции могут перестать выполняться.

**Решение.** NTP/chrony на всех узлах, healthcheck `NODES_TIME_DIFFERENCE`, держать расхождение в допустимых пределах ([проверка здоровья](../../reference/ydb-cli/commands/monitoring-healthcheck.md)).

### Расхождение времени на клиентах {#tip-time-client}

**Проблема.** Сдвиг часов на приложениях ломает JWT, дедлайны и логику таймстемпов.

**Решение.** Синхронизация времени на клиентах, согласование с временем кластера при выдаче токенов ([аутентификация](../../reference/ydb-sdk/auth.md)).

### Энергосбережение CPU на серверах {#tip-cpu-power}

**Проблема.** Снижение частоты и глубокие C-state дают джиттер latency и нестабильный throughput.

**Решение.** Режим performance в BIOS/ОС, отключение агрессивного энергосбережения для серверов с {{ ydb-short-name }} ([производительность](../../troubleshooting/performance/index.md)).
