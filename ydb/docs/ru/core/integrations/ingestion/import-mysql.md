# Импорт структуры таблиц и данных из MySQL

## Введение {#intro}

Задача переноса схемы и данных из **MySQL** в {{ ydb-short-name }} для **больших таблиц** (когда объём данных превышает доступную оперативную память) может быть решена с помощью [утилиты mysql2ydb](https://github.com/ydb-platform/mysql-ydb-importer).

**Цель утилиты — получить копию исходной базы данных один к одному.** Имена таблиц и колонок сохраняются как в MySQL; типы данных сопоставляются с ближайшими эквивалентами {{ ydb-short-name }} (`INT` → `Int32`, `VARCHAR` → `Text`, `AUTO_INCREMENT` → `BigSerial` и т. д.). В результате должна получиться база {{ ydb-short-name }}, повторяющая исходную структуру и позволяющая выполнять запросы с минимальным «переводом» с MySQL.

Каждый фрагмент данных читается из MySQL одним ограниченным запросом `SELECT`, записывается в {{ ydb-short-name }} через идемпотентный `BulkUpsert`, после чего освобождается из памяти. Процесс повторяется, пока не будет обработана вся таблица — в памяти одновременно хранится не больше одной порции (при конвейерном чтении — не больше двух).

Импорт выполняется в следующем порядке:

1. Утилита подключается к MySQL и по метаданным `information_schema` определяет состав таблиц и их структуру.
2. В {{ ydb-short-name }} создаются целевые таблицы с сопоставленными типами данных.
3. Данные переносятся порциями: чтение из MySQL и запись в {{ ydb-short-name }} через `BulkUpsert`.
4. После успешной записи каждой порции в служебную таблицу состояния сохраняется позиция курсора — при сбое миграцию можно продолжить с последнего успешного фрагмента.

## Зачем отдельная утилита? {#why-separate}

[YDB Importer](import-jdbc.md) — универсальный **JDBC**-инструмент: PostgreSQL, Oracle, SQL Server, Db2, Informix и другие источники. Он удобен, когда нужен один импортёр для многих СУБД, параллельный импорт таблиц, дополнительные таблицы для BLOB/CLOB, XML-конфигурация сопоставления таблиц и настройка партиционирования {{ ydb-short-name }}.

**mysql2ydb** решает более узкую задачу: перенести базу MySQL в {{ ydb-short-name }} одним бинарным файлом, без JVM и XML-конфигурации.

| | [ydb-importer](import-jdbc.md) | **mysql2ydb** |
|---|---|---|
| Среда выполнения | Java + JDBC-драйверы + XML-конфигурация | Один бинарный файл на Go, нативный протокол MySQL |
| Область применения | Множество JDBC-источников | Только MySQL |
| Размер порции по умолчанию | 1 000 строк (`max-batch-rows`) | 10 000 строк (`-batch-size`), с автонастройкой |
| Большие таблицы | Параллельные диапазоны, буферы партиций | Постраничное чтение по курсору, размер порции с учётом RAM и сети |
| Возобновление после сбоя | Повторный запуск импорта | Контрольная точка на каждую порцию в служебной таблице {{ ydb-short-name }} |
| Настройка | JAR-файлы JDBC, XML-конфигурация | `~/.my.cnf` и два флага |

Для **миграции только из MySQL** с **многогигабайтными таблицами** специализированный инструмент удобнее универсального JDBC-импортёра: достаточно `~/.my.cnf` и флага `-ydb`, прогресс сохраняется в служебной таблице {{ ydb-short-name }}, размер каждой порции ограничен доступной RAM.

Подробнее о различиях в схеме данных — в разделе [Соответствие схемы MySQL](#schema-fidelity).

## Соответствие схемы MySQL {#schema-fidelity}

При переносе схемы утилита стремится к **структурному соответствию**: те же имена таблиц и колонок, индексы там, где {{ ydb-short-name }} их поддерживает, типы, при которых значения сохраняются без потерь (беззнаковые целые остаются беззнаковыми, логические типы — логическими и т. д.).

[YDB Importer](import-jdbc.md) решает другую задачу схемы: он рассчитан на импорт из многих JDBC-источников в **настраиваемую** структуру {{ ydb-short-name }} (`table-name-format`, `blob-name-format`, режимы преобразования дат, опциональные синтетические ключи). Такая гибкость меняет результат — это не зеркальная копия каталога MySQL.

mysql2ydb читает метаданные MySQL из `information_schema` и формирует DDL {{ ydb-short-name }} напрямую. Ниже — практические отличия при импорте одной и той же базы MySQL двумя инструментами:

| Возможность MySQL | mysql2ydb (копия один к одному) | ydb-importer |
|---|---|---|
| **Имена таблиц** | Как в MySQL (`users`, `orders`, …) | Переименование по шаблону, например `mysql1/${schema}/${table}` → `mysql1/mydb/users` ([пример конфигурации](https://github.com/ydb-platform/ydb-importer/blob/main/scripts/sample-mysql.xml)) |
| **Имена колонок** | Как в MySQL | Санитизация: пробелы, `.`, `/`, `` ` `` → `_` |
| `AUTO_INCREMENT` | `BigSerial` + `ALTER SEQUENCE … START WITH` из `TABLES.AUTO_INCREMENT` | Обычный `Int32`/`Int64`; без `BigSerial` и сброса последовательности |
| `INT UNSIGNED`, `BIGINT UNSIGNED`, … | `Uint32` / `Uint64` | JDBC сопоставляет целые со знаковыми `Int32`/`Int64` |
| `TINYINT(1)` | `Bool` | `TINYINT` → `Int32` (`BOOLEAN` → `Bool`) |
| `BIT` | `Uint64` | `Bool` (JDBC `Types.BIT`) |
| Вторичные `KEY` / `UNIQUE KEY` | `INDEX … GLOBAL ASYNC` / `GLOBAL UNIQUE SYNC` в `CREATE TABLE` | В DDL только `PRIMARY KEY`; вторичные индексы не создаются |
| `ENUM`, `SET`, `JSON` | `Text` | `Text` (аналогично) |
| `BLOB` / `BINARY` | Та же колонка, значение inline в `String` (байты) | Колонка становится `Int64` (идентификатор blob); данные выносятся в **отдельную таблицу** `${schema}/${table}_${field}` со строками `(id, pos, val)` блоками по 64 КБ |
| `TEXT` / `CLOB` (большой текст) | Inline `Text` в основной таблице | Опционально отдельная CLOB-таблица (блоки по 32 К символов) или inline `Text` в зависимости от настроек |
| Таблица **без первичного ключа** | Схема без изменений; данные читаются через `LIMIT/OFFSET` | Добавляется колонка `ydb_synth_key Text` как PK (SHA-256 по строке); **дубликаты схлопываются** в одну строку |
| `DATE` | `Date` | `Date32` по умолчанию (`conv-date=DATE_NEW`) |
| `DATETIME` / `TIMESTAMP` | `Timestamp` | `Datetime64` / `Timestamp64` по умолчанию |
| `TIME` | `Text` (запасной вариант) | `Int32` (секунды с полуночи) |
| `DECIMAL(p,s)` | `Decimal(22, 9)` для всех | `Decimal(p,s)` при `allow-custom-decimal=true` |
| `YEAR` | `Uint16` | Не покрыто в тестах типов MySQL |
| Партиционирование | Только `AUTO_PARTITIONING_BY_LOAD` | `PARTITION_AT_KEYS`, разбиение по партициям источника, `HASH` для column store и др. |

### Примеры структурных отличий {#schema-examples}

**`AUTO_INCREMENT`** — колонка MySQL `id BIGINT AUTO_INCREMENT` превращается в:

```sql
`id` BigSerial NOT NULL,
PRIMARY KEY (`id`)
```

После `CREATE TABLE`, если известно значение `information_schema.TABLES.AUTO_INCREMENT`:

```sql
ALTER SEQUENCE `<db>/<table>/_serial_column_id` START WITH <next_value> RESTART
```

**`BLOB`** — таблица MySQL `attachments (id INT, data MEDIUMBLOB)`:

| | Колонка `data` в основной таблице | Где лежат байты |
|---|---|---|
| **mysql2ydb** | `data String` | В той же строке |
| **ydb-importer** | `data Int64` (ссылка) | Отдельная таблица `…/attachments_data` с порционными строками |

С mysql2ydb можно выполнить `SELECT data FROM attachments WHERE id = 1` так же, как в MySQL. С ydb-importer потребуется соединение основной таблицы с дополнительной blob-таблицей.

**Таблицы без PK** — mysql2ydb не добавляет колонок; ydb-importer вводит `ydb_synth_key` и меняет количество строк при полных дубликатах.

**Вторичные индексы** — неуникальные ключи становятся `GLOBAL ASYNC`, уникальные (кроме PK) — `GLOBAL UNIQUE SYNC`. Таблицы с синхронными уникальными индексами при загрузке данных автоматически переключаются на транзакционный `UPSERT`, поскольку `BulkUpsert` поддерживает только асинхронные индексы.

## Эффективный доступ к MySQL {#efficient-access}

Утилита минимизирует обращения к MySQL и ограничивает объём каждого запроса:

1. **Постраничное чтение по первичному ключу** — `WHERE (pk…) > (?) ORDER BY pk LIMIT n`. Без полного сканирования таблицы в память и без деградирующего `OFFSET` на больших объёмах.
2. **Конвейер чтение → запись** — одна горутина читает следующую порцию из MySQL, пока другая записывает предыдущую в {{ ydb-short-name }}. На таблицу одновременно в полёте не более двух порций.
3. **Адаптивный размер порции** — число строк ограничивается доступной RAM, средним размером строки из `information_schema` и сетевым бюджетом (≤ 125 МБ на выборку ≈ 10 с при 100 Мбит/с).
4. **Контрольная точка прогресса** — после каждого успешного `BulkUpsert` позиция курсора и счётчик строк сохраняются в служебной таблице {{ ydb-short-name }}; перезапуск продолжает с последней порции.

```text
MySQL: SELECT порция N   →  BulkUpsert(порция N)   →  YDB
MySQL: SELECT порция N+1 →  BulkUpsert(порция N+1) →  YDB
...
```

## Возможности {#features}

- **Копия схемы один к одному** — исходные имена таблиц и колонок; сопоставление типов MySQL с ближайшими типами {{ ydb-short-name }} (`AUTO_INCREMENT` → `BigSerial`, `UNSIGNED` → `Uint32`/`Uint64`, вторичные индексы, `TINYINT(1)` → `Bool`; см. раздел [Соответствие схемы MySQL](#schema-fidelity)).
- **Создание схемы** — таблицы в {{ ydb-short-name }} создаются по метаданным MySQL (`information_schema`).
- **Пофрагментная миграция данных** — таблица обрабатывается порциями фиксированного размера; объём данных может превышать доступную RAM.
- **Идемпотентная запись** — `BulkUpsert` с `table.WithIdempotent()`; повторный запуск не создаёт дубликаты.
- **Возобновление** — прогресс по каждой таблице хранится в {{ ydb-short-name }}; прерванную миграцию можно продолжить с последней порции.

## Установка {#install}

Исходный код и инструкции по сборке — в [репозитории mysql-ydb-importer](https://github.com/ydb-platform/mysql-ydb-importer) на GitHub.

Для сборки требуется [Go](https://go.dev/) (версия, указанная в `go.mod` репозитория):

```bash
git clone https://github.com/ydb-platform/mysql-ydb-importer.git
cd mysql-ydb-importer
go build -o mysql2ydb ./cmd/mysql2ydb
```

## Порядок использования {#use}

Параметры подключения к MySQL по умолчанию читаются из файла **~/.my.cnf** (как у клиента `mysql`), секция `[client]`. Флаг `-mysql` переопределяет конфигурационный файл.

Пример `~/.my.cnf`:

```ini
[client]
user = myuser
password = mypass
host = localhost
port = 3306
database = mydb
```

Если сервер требует защищённое соединение (`require_secure_transport=ON`), добавьте в `[client]`:

```ini
ssl-mode = REQUIRED
```

Для самоподписанного сертификата (без проверки):

```ini
ssl-mode = REQUIRED
ssl-verify = 0
```

(или `ssl=1` и `ssl-verify=0`).

Минимальный запуск при наличии `~/.my.cnf` (обязателен только `-ydb`):

```bash
./mysql2ydb -ydb "grpc://localhost:2136"
```

С явным DSN MySQL (игнорирует `~/.my.cnf`):

```bash
./mysql2ydb -mysql "user:password@tcp(localhost:3306)/mydb" -ydb "grpc://localhost:2136" -batch-size 10000
```

### Флаги командной строки {#flags}

| Флаг | Описание |
|------|----------|
| `-mysql` | DSN MySQL (если задан — переопределяет `~/.my.cnf`) |
| `-ydb` | Конечная точка {{ ydb-short-name }} (обязательный) |
| `-ydb-database` | Путь к базе данных {{ ydb-short-name }} (по умолчанию `local`) |
| `-schema-only` | Только создать схему, без переноса данных |
| `-data-only` | Только перенести данные (схема должна уже существовать) |
| `-batch-size` | Целевой размер порции в строках (по умолчанию 10 000) |
| `-max-chunk-rows` | Жёсткий верхний предел строк в порции (по умолчанию 25 000) |
| `-parallel-tables` | Число таблиц для параллельного переноса (по умолчанию 1) |
| `-tables` | Список таблиц через запятую (по умолчанию — все) |
| `-force` | Перенести все таблицы заново, игнорируя сохранённое состояние |
| `-force-recreate` | Удалить все таблицы в {{ ydb-short-name }} и пересоздать схему с нуля |

### Примеры {#examples}

Только схема:

```bash
./mysql2ydb -mysql "..." -ydb "grpc://localhost:2136" -schema-only
```

Только данные (после создания схемы):

```bash
./mysql2ydb -mysql "..." -ydb "grpc://localhost:2136" -data-only -batch-size 5000
```

Выбранные таблицы:

```bash
./mysql2ydb -mysql "..." -ydb "grpc://localhost:2136" -tables "users,orders"
```

## Пофрагментное чтение {#chunked-reading}

- Для таблиц с **первичным ключом** используется постраничное чтение по курсору: `WHERE (pk > ?) ORDER BY pk LIMIT batch_size`. В памяти одновременно находится не больше одной порции; смещение `OFFSET` не растёт с номером страницы.
- Для таблиц без подходящего ключа применяется `LIMIT batch_size OFFSET offset` (очень большие смещения могут замедлять работу).

## Идемпотентность BulkUpsert {#bulkupsert}

Запись в {{ ydb-short-name }} выполняется через `BulkUpsert` с опцией **WithIdempotent()**:

- повторная отправка той же порции не создаёт дублирующие строки;
- удобно при перезапуске миграции или повторной загрузке после сбоя.
