При подключении к [эндпоинту](../../../../concepts/connect.md#endpoint) по gRPCs (с шифрованием) клиент проверяет серверный сертификат по [цепочке доверия](../../../../concepts/connect.md#tls-cert). Если сервер использует корневой CA, которого нет в стандартном наборе доверенных сертификатов CLI, то путь к нему указывается в параметре:

- `--ca-file <filename>` : файл корневого PEM-сертификата (CA), которым подписан серверный сертификат

Если в конфигурации кластера в секции `client_certificate_authorization` включён параметр [`request_client_certificate`](../../../configuration/client_certificate_authorization.md), сервер также запрашивает клиентский сертификат при TLS-handshake. В этом случае дополнительно указывается:

- `--client-cert-file <filename>` : файл клиентского сертификата (PEM или PKCS#12)
- `--client-cert-key-file <filename>` : файл закрытого ключа клиентского сертификата (указывается, если сертификат и ключ содержатся в отдельных файлах)
- `--client-cert-key-password-file <filename>` : файл с паролем к закрытому ключу, если ключ зашифрован; если ключ зашифрован, а опция не указана, пароль будет запрошен интерактивно

Для [аутентификации по клиентскому сертификату](../../../../security/authentication.md#client-certificate) на стороне сервера должен быть включён параметр `request_client_certificate: true`, а на клиенте не должны быть заданы другие опции [аутентификации](#authentication). Если одновременно указаны `--client-cert-file` и другой режим аутентификации, сертификат по-прежнему передаётся при TLS-handshake, но для идентификации запросов используется [аутентификационный токен](../../../../concepts/glossary.md#auth-token), а не сертификат.

В итоге успешной проверки клиенту назначается идентификатор защиты [SID](../../../../concepts/glossary.md#access-sid), который обладает всеми назначенными соответствующему идентификатору [правами](../../../../concepts/glossary.md#access-right). Права на схему и данные не выдаются автоматически — см. [авторизацию](../../../../security/authorization.md).
