# Серверная балансировка сессий {{ ydb-short-name }} и обработка ошибок `BAD_SESSION`

Серверная балансировка сессий — это механизм {{ ydb-short-name }} для равномерного распределения нагрузки между узлами кластера. В отличие от клиентской балансировки, которая выбирает ноду для создания сессии, серверная балансировка работает на уровне уже созданных сессий и позволяет "перемещать" их между узлами для выравнивания нагрузки.

## Как работает серверная балансировка

Когда сервер {{ ydb-short-name }} обнаруживает дисбаланс нагрузки (например, на одной ноде слишком много активных сессий), он инициирует процесс graceful shutdown для части сессий. Вместо немедленного закрытия сессии сервер:

1. Отправляет клиенту `shutdownHint` в метаданных ответа
2. Дает клиенту время (таймаут) для корректного завершения работы с сессией
3. Если клиент не реагирует на сигнал, сервер самостоятельно закрывает сессию с ошибкой `BAD_SESSION`

Правильная обработка `shutdownHint` клиентом обеспечивает бесшовное переключение нагрузки без ошибок для пользователя.

## Проблема

Самостоятельный менеджмент сессий на клиенте без учета серверных сигналов приводит к повышенному фону ошибок `BAD_SESSION`. Основные причины:

- Клиент игнорирует `shutdownHint` и продолжает использовать сессию
- Долгосрочное удержание сессии без возврата в пул
- Отсутствие автоматического ретрая при получении `BAD_SESSION`

## Решение

Использовать официальные SDK {{ ydb-short-name }}, которые содержат отлаженные механизмы работы с сессиями:

- Автоматическое распознавание `shutdownHint` и graceful закрытие сессий
- Пул сессий с автоматическим удалением "плохих" сессий
- Встроенные ретраи для `BAD_SESSION` ошибок
- Поддержка серверной балансировки через заголовок `x-ydb-client-capabilities: session-balancer`

## Примеры кода


### ❌ Плохой пример: Самостоятельный менеджмент сессий

```go
// Плохо: клиент сам управляет сессией без учета серверных сигналов
type BadClient struct {
    session *table.Session
    mutex   sync.Mutex
}

func (c *BadClient) ExecuteQuery(ctx context.Context, query string) error {
    c.mutex.Lock()
    defer c.mutex.Unlock()
    
    // Игнорируем возможные shutdownHint от сервера
    result, err := c.session.Execute(ctx, query)
    if err != nil {
        // Просто возвращаем ошибку без ретрая
        return err
    }
    // Работаем с результатом...
    return nil
}
```

### ✅ Хороший пример: Использование официального SDK

```go
// Хорошо: используем пул сессий из официального SDK
func GoodExample(db *ydb.Driver) error {
    // Создаем табличного клиента с поддержкой серверной балансировки
    tableClient := db.Table()
    
    // Выполняем запрос через пул сессий
    err := tableClient.Do(ctx, func(ctx context.Context, s table.Session) error {
        // SDK автоматически обрабатывает shutdownHint и BAD_SESSION
        result, err := s.Execute(ctx, query)
        if err != nil {
            // SDK автоматически ретраит BAD_SESSION на другой сессии
            return err
        }
        // Работаем с результатом...
        return nil
    }, table.WithIdempotent())
    
    return err
}
```

### ✅ Настройка клиента с поддержкой серверной балансировки

```go
// Настройка драйвера с поддержкой всех возможностей
func CreateBalancedDriver() (*ydb.Driver, error) {
    return ydb.Open(ctx,
        "grpcs://ydb.server.yandex.net:2135/?database=/ru/prod/db",
        ydb.WithSessionPoolSize(50),
        ydb.WithSessionPoolIdleThreshold(5*time.Minute),
        ydb.WithBalancingPolicy(ydb.BalancingPolicyUseAllNodes),
        ydb.WithClientCapabilities("session-balancer"), // Включаем серверную балансировку
    )
}
```

### ❌ Распространенная ошибка: Долгосрочное удержание сессии

```go
// Плохо: берем сессию надолго, не давая SDK реагировать на балансировку
func LongRunningProcess(db *ydb.Driver) error {
    session, err := db.Table().CreateSession(ctx)
    if err != nil {
        return err
    }
    defer session.Close() // Закрываем только в конце
    
    // Много операций на одной сессии
    for i := 0; i < 1000; i++ {
        _, err := session.Execute(ctx, fmt.Sprintf("INSERT ... VALUES (%d)", i))
        if err != nil {
            // Если сервер начал shutdown этой сессии, все последующие запросы упадут
            return err
        }
    }
    return nil
}
```

### ✅ Правильный подход: Короткие операции с возвратом в пул

```go
// Хорошо: каждая операция в отдельной сессии из пула
func ShortOperations(db *ydb.Driver) error {
    for i := 0; i < 1000; i++ {
        err := db.Table().Do(ctx, func(ctx context.Context, s table.Session) error {
            _, err := s.Execute(ctx, fmt.Sprintf("INSERT ... VALUES (%d)", i))
            return err
        })
        if err != nil {
            return err
        }
    }
    return nil
}
```


## Рекомендации

1. **Всегда используйте официальные SDK** — они содержат отлаженную логику обработки серверных сигналов
2. **Не удерживайте сессии долго** — возвращайте их в пул после каждой логической операции
3. **Включайте поддержку серверной балансировки** через соответствующие заголовки
4. **Настраивайте разумные лимиты пула** в соответствии с нагрузкой
5. **Мониторьте метрики BAD_SESSION** для своевременного обнаружения проблем

Правильная работа с серверной балансировкой позволяет {{ ydb-short-name }} эффективно распределять нагрузку и обеспечивать высокую доступность сервиса даже при плановых работах и обновлениях кластера.
