# Как клиентским приложениям переживать rolling restart {{ ydb-short-name }}

## Проблема

При обновлении кластера {{ ydb-short-name }} или проведении плановых работ администраторы выполняют rolling restart - последовательный перезапуск узлов. Хотя данные остаются доступными, клиентские приложения могут столкнуться с проблемами:

1. **Ошибки BAD_SESSION** - подключения к узлам, которые находятся в процессе остановки
2. **Временное увеличение латентности** - перемещение таблеток между узлами требует времени
3. **Потеря кэша запросов** - перезапуск узлов сбрасывает кэш откомпилированных запросов
4. **Нарушение балансировки** - неправильная настройка клиентской балансировки усугубляет проблемы

**Ключевой инсайт**: Проблемы возникают не из-за недоступности данных, а из-за того, что клиенты подключаются к неправильным узлам в неподходящее время.

## Практические рекомендации для клиентов

### 1. Архитектура кластера: обеспечьте избыточность в каждой зоне

**Проблема**: Клиенты используют алгоритм "предпочитать ближайший дата-центр" в кластерах с 1 узлом на ДЦ.

**Решение**: Размещайте минимум 2 узла в каждой зоне доступности. Это позволяет:
- Сохранять локальность подключений
- Обеспечивать отказоустойчивость при перезапуске одного узла
- Избегать ошибок BAD_SESSION

**Рекомендуемая конфигурация**:
- Для production: минимум 6 узлов, распределенных по разным зонам доступности
- Для high-availability: 9+ узлов (2+ на зону)

### 3. Настройка пула сессий и повторных попыток

**Прогрев подключений** перед началом высокой нагрузки:
```go
func warmupConnection(db *ydb.Driver, warmupQueries []string) error {
    ctx := context.Background()
    
    // Создаем сессии параллельно для распределения по разным узлам
    var wg sync.WaitGroup
    sessions := make([]query.Session, 10) // Количество зависит от ожидаемой параллельности
    
    for i := range sessions {
        wg.Add(1)
        go func(idx int) {
            defer wg.Done()
            sess, err := db.Query().CreateSession(ctx)
            if err == nil {
                sessions[idx] = sess
                // Прогреваем кэш запросов на каждом узле
                for _, q := range warmupQueries {
                    sess.Execute(ctx, q) // Игнорируем ошибки при прогреве
                }
            }
        }(i)
    }
    wg.Wait()
    
    // Аккуратно закрываем сессии
    for _, sess := range sessions {
        if sess != nil {
            sess.Close(ctx)
        }
    }
    return nil
}
```

**Корректная обработка ошибок BAD_SESSION**:
```go
func executeWithRetry(ctx context.Context, db *ydb.Driver, query string, params ...interface{}) error {
    retrySettings := ydb.RetrySettings{
        MaxRetries: 5, // Увеличиваем количество попыток
        Backoff:    ydb.ExponentialBackoff{
            Min: 100 * time.Millisecond,
            Max: 5 * time.Second, // Увеличиваем максимальную задержку
        },
    }
    
    return ydb.Retry(ctx, retrySettings, func(ctx context.Context) error {
        return db.Query().Do(ctx, func(ctx context.Context, s query.Session) error {
            _, err := s.Execute(ctx, query, params...)
            if ydb.IsBadSession(err) {
                // Сессия невалидна - нужна новая, это нормально при rolling restart
                return ydb.RetryableError(err)
            }
            // Другие временные ошибки тоже обрабатываем как повторяемые
            if ydb.IsTransportError(err) || ydb.IsTimeoutError(err) {
                return ydb.RetryableError(err)
            }
            return err
        })
    })
}
```

### 4. Мониторинг и адаптация нагрузки

**Что мониторить во время rolling restart**:
- **Количество ошибок BAD_SESSION** - резкий рост указывает на проблемы
- **Латентность запросов (p95, p99)** - временное увеличение нормально
- **Количество активных сессий** - должно оставаться стабильным

**Адаптация нагрузки**:
```go
// Умный throttling при обнаружении проблем
func adaptiveExecute(ctx context.Context, db *ydb.Driver, query string) error {
    // Мониторим ошибки в реальном времени
    errorRate := getCurrentErrorRate()
    
    if errorRate > 0.1 { // Если более 10% ошибок
        // Увеличиваем таймауты и уменьшаем параллелизм
        ctx, cancel := context.WithTimeout(ctx, 30*time.Second)
        defer cancel()
        
        // Добавляем дополнительную задержку
        time.Sleep(100 * time.Millisecond)
    }
    
    return executeWithRetry(ctx, db, query)
}
```

### 5. Слишком большие и слишком маленькие timeout'ы
Таймауты на запросы – по крайней мере, наиболее востребованные в приложении – лучше выставлять исходя из характерного и ожидаемого времени ответа.
Например, если запрос _почти всегда_ (например, в 99 перцентиле) обрабатывается за 100ms, то одинаково плохой идеей будет как таймаут в 100ms, так и в 1000ms.
Слишком маленький таймаут приведёт к тому, что во время стрессовой для базы ситуации (к таковой относится не только роллинг-рестарт, но и, например, перезгрузка) запросы перестанут успевать исполниться и начнут копиться.
Слишком большой же может привести к тому, что база будет давать ответ, который на самом деле уже не нужен тому, кто его запросил, и ответ будет отброшен или клиентом SDK, или где-то ещё выше по стеку.

В идеальном мире правильно было бы пробрасывать оставшееся на исполнение запроса время (deadline propagation). Но если это недоступно - то хотя бы выставлять время на исполнение запроса по формуле минимума из таймаута клиента, который идёт в приложение, и удвоенного 99-го перцентиля времени ответа на запрос в нормальной ситуации.


## Заключение

Успешное переживание rolling restart {{ ydb-short-name }} требует правильной архитектуры и настройки клиентских приложений. Ключевые принципы:

1. **Обеспечьте избыточность** - минимум 2 узла в каждой зоне доступности
2. **Используйте равномерную балансировку** - избегайте "предпочитать ближайший ДЦ"
3. **Настройте корректные повторные попытки** - обрабатывайте BAD_SESSION как временную ошибку
4. **Прогревайте подключения** - создавайте сессии заранее и распределяйте по узлам
5. **Мониторьте и адаптируйтесь** - отслеживайте метрики и регулируйте нагрузку

Следуя этим рекомендациям, ваше приложение сможет беспрепятственно переживать плановые обновления кластера {{ ydb-short-name }} без существенного влияния на пользователей.
