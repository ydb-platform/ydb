# Учения по закрытию датацентров

Учения по закрытию ДЦ призваны проверить, как сервисы Яндекса способны пережить работу без одного из наших ДЦ (и иногда дать возможность закрытый ДЦ обслужить).
В общем виде отключение ДЦ со стороны {{ ydb-short-name }} выглядит так:
1. Отключаем автопочинку групп хранения
2. Таблетки (это такие сущности (меньшие, чем процессы), обрабатывающие определённые партиции строковых (даташарды) и колоночных (колумншарды) таблиц или топиков (PQ-таблетки)) получают запрет на запуск на узлах закрываемого ДЦ
3. Узлы закрываемого ДЦ с некоторым окном получают команду на освобождение, вынуждающую таблетки переехать на другие узлы (узлы закрываемого ДЦ были запрещены ранее)
4. Останавливаем вычислительные узлы в закрываемом ДЦ
5. Останавливаем узлы хранения в закрываемом ДЦ
6. <Где-то в этот момент снимается анонс с L3-балансера {{ ydb-short-name }} в локации, когда гаснет последний узел хранения>
7. Закрываем файрволлом все узлы в закрываемом ДЦ

## Проблемы

Важно разделить проблемы на разные категории:
- Проблемы при входе в учения и выходе из них
- Проблемы во время учений
- Проблемы после возвращения из учений

### Вход в учения 

Вход в учения  - своего рода "стресс" для базы данных: нужно перенести множество даташардов из закрывающегося датацентра в оставшиеся.

В это время часть транзакций будет завершена со статусом UNKNOWN, потому что потерялась информация об их статусе на переехавших даташардах, часть - займёт больше времени, чем обычно, потому что будут работать внутренние ретраи. Может так случиться, что запрос и вовсе зависнет до истечения таймаута.

Также, часть сессий будет инвалидирована со стороны сервера. Если вы пользуетесь актуальными версиями SDK, то они уже умеют обрабатывать shutdownHint, но стоит избегать прямой работы с сессиями: долгоживующие сессии всё же могут завершаться ошибками (https://stackoverflow.yandex-team.ru/questions/5303).

Такие всплески могут привести к росту числа ошибок, возвращаемых клиентам, или к росту числа ретраев запросов в базу. А ретраи запросов в базу, в свою очередь, потребляют ресурсы, которые могут быть ограничены: сессии из пула на клиенте, сессии из пула на сервере, машинное время на обработку самих сессий, RAM с данными для сессий, машинное время, RAM и IO на обработку запросов. И ресурсов, которых было в избытке при 3 ДЦ в 2 вдруг уже перестаёт быть достаточно, что порождает худшее замедление уже из-за дефицита ресурсов, и новые ретраи и сессии и так далее. Цикл с положительной обратной связью, в общем.  

Чтобы снизить риски от таких всплесков - важно маркировать идемпотентные запросы как, собственно, идемпотентные, чтобы SDK мог их поретраить при возможности. 

Также стОит ответственно использовать ретраи: выставлять таймауты не слишком большими, но и не слишком маленькими (что получится сделать только если выставлять их для каждого или почти каждого запроса индивидуально, что SDK благо позволяет). Хорошей отправной точкой будет p99 для конкретно этого запроса, увеличенный в 2-3 раза. 
Стоит учитывать, что у наших SDK есть несколько таймаутов, и нужно выставлять все: https://ydb.tech/docs/ru/dev/timeouts

Если архитектура клиента позволяет, то стоит учитывать в таймауте "оставшееся" время с учётом переданного с предыдущих слоёв обработки оставшегося времени (это часто называют Deadline Propagation). 
Если ответ на запрос больше не нужен, то хорошо бы инвалидировать сессию и разорвать соединение – это работает в качестве отмены запроса в {{ ydb-short-name }}. Это хуже, чем аккуратные таймауты и Deadline Propagation, но лучше, чем оставлять запрос висеть.

Переезды даташардов при выходе из учений будут плавными: {{ ydb-short-name }} обычно "неохотно" переселяет партиции, чтобы снизить влияние от переезда на пользователей. 

Размер пула сессий на клиенте стоит подобрать исходя хотя бы из обычного потока запросов (сколько обычно запросов исполняется одновременно) с поправкой на то, что во время проблем будут и обычные запросы исполняться дольше, и клиенты уже вашего приложения будут ретраить запросы к вам, то есть нагрузка вырастет. И ещё у вас нагрузка базово вырастет в полтора раза, потому что треть забрали.

Если ваше приложение умеет перечитывать всю или часть конфигурации без останова (обычно внутри Яндекса для такого оперативного управления конфигами используют сервис Molniya), то размер пула и предпочтение локального ДЦ - хорошие настройки для того, чтобы вынести их наружу; клиент {{ ydb-short-name }} перезапускается дёшево.


Кроме того, если вы используете предпочтение локального ДЦ – то клиенты во время закрытия могут начать "создавать давку" на оставшихся узлах: ведь клиент сейчас во время discovery не оценивает, что "узлов осталось всего 10%" - он просто получает информацию о том, что "вот есть два узла, ни в чём себе не отказывай". Узлы из закрывающегося ДЦ сейчас не одновременно выводятся (и возвращаются тоже не одновременно, так что эта история ждёт нас в третьей категории).

Той же проблеме подвержены фолловеры (рид-реплики): если "обычные" даташарды из закрывающегося ДЦ уезжают, то рид-реплики могут быть настроены так, чтобы в каждом ДЦ ихз было определённое количество. При закрытии и открытии ДЦ мы обращаем на них особое внимание и в целом умеем подобную давку предотвращать.

Также, в зависимости от способа определения ближайшего ДЦ клиент подвержен риску телепортации. Если клиент опирается на ответ L3-балансера ServiceDiscovery для определния своей локации (так делает Python SDK и C++ SDK), то в момент, когда {{ ydb-short-name }} в процессе подготовки к учениям перестаёт анонсировать свои IP в закрываемом ДЦ - клиенты начинают присваивать себе "соседний" ДЦ и направлять трафик в его узлы.

Предельное число сессий на один узел базы - 1000, при превышении в начнёте получать ошибки Overloaded, поэтому ограничение сверху для пула на клиенте можно получить, поделив 1000 * <число узлов базы> * 2/3 / <число подов клиентов>.

Чтобы избежать телепортации клиентов - можно вместо режима "предпочтения ближайшего ДЦ" использовать "предпочтение конкретного ДЦ" и задачу определения ДЦ выполнять вручную (например, задавать в конфиге или получать из переменных окружения в поде, Деплой такие выдаёт).

Чтобы избежать давки при предпочтении локального ДЦ (если без такого предпочтения не обойтись) придётся повозиться побольше.

В качестве самого простого решения, конечно же, можно предложить «закрываться в локации раньше, чем это сделает {{ ydb-short-name }}». Но на практике это не всегда удобно: обычно закрытие локации начинают за несколько часов до учений, нередко ещё вечером предыдущего дня, чтобы партиции переезжали медленнее и влияние на пользователей было минимальным. Поэтому стратегия «закрывать раньше {{ ydb-short-name }}» часто оказывается неудобной.

Что ещё можно сделать? Выше уже упоминалось использование REX для управления клиентскими конфигами. С его помощью закрываемый ДЦ можно переключить в режим «используй все узлы». Это немного ухудшит задержки в самом ДЦ и увеличит нагрузку на оставшиеся ДЦ {{ ydb-short-name }}, но поможет избежать «давки» как при закрытии, так и при открытии.

Более сложный вариант — реализовать режим деградации на клиенте. Например, из SDK можно получать количество эндпойнтов, которые он видит. Если сравнивать это значение с числом из конфига или с заранее сохранённым «нормальным» значением, можно автоматически понимать, что в ДЦ осталось слишком мало узлов, и переключаться в режим работы со всеми узлами или вовсе переставать сообщать балансировщику о готовности обслуживать трафик. Если над сервисом стоит awacs, можно настроить двухуровневую балансировку так, чтобы ДЦ автоматически снимался с балансировки, когда в нём достаточно подов сообщают о неготовности принимать трафик.

### Во время учений

Во время учений на каждый отдельный узел {{ ydb-short-name }} приходится бОльшая нагрузка, что в целом ожидаемо.

Но есть и менее ожидаемые вещи. Так, при недоступности 1 из 3 ДЦ пишется не три копии данных, а 4: по 2 копии на два диска, то есть объёмы записи растут примерно на треть.

Кроме того, тредпулы в {{ ydb-short-name }} заданы не жестко, и треды из "незанятого" пула могут быть переданы в пользу "голодающего", что делает оценку нагрузки в пуле более сложным занятием; лучше смотреть на общую нагрузку в {{ ydb-short-name }}, а нагрузку в каждом отдельном пуле для предсказаний не использовать. 

В {{ ydb-short-name }} используется актор-система, поэтому обработка запросов асинхронна. Это позволяет эффективно утилизировать CPU, но при высокой нагрузке даёт побочные эффекты. Каждый запрос требует взаимодействия множества акторов на множестве узлов и пересылки сообщений между ними. Поэтому высокая загрузка тредпула может привести к тому, что нужные сообщения начнут обрабатываться позже, а время выполнения запроса резко вырастет. Характерная загрузка, например, User- или IC-пула на отдельном узле, при которой такого рода проблемы становятся заметны, составляет около 80%. Теоретически рост задержек может начинаться и раньше, но 80% — это практический ориентир, который мы видим в обращениях в поддержку.

Соответственно, с ростом времени ответа мы рискуем получить вышеописанные проблемы со взрывами ретраев, как при входе в учения. Но вероятность такого развития событий ниже: у нас есть алерты сервис-провайдера на достаточность ресурсов, и мы перед учениями стараемся дойти до всех, у кого ресурсов может не хватить, и немножко понудеть об этом. Основное, что стоит учитывать относительно учений в теме с ретраями - это их влияние на время ответа: время записи во время учений прям всегда будет больше. Разницу между режимами "во время учений" и "обычный" можно сгладить, только увеличивая время записи в "обычном" режиме (включив Enable3x3RequestsForMirror3DCMinLatencyPut).


Итак, что плохого мы можем получить во время учений:
| Проблема | Что делать |
| -------- | ---------- |
| Рост времён ответов в верхних перцентилях для запросов, пишущих данные | Можно замедлить запись вне учений, чтобы роста не было;<br/>Контролировать наличие запаса ресурсов для закрытия ДЦ |
| Рост времён ответов в верхних перцентилях для запросов, только читающих данные | Контролировать наличие запаса ресурсов для закрытия ДЦ; |
| Рост числа ошибок TIMEOUT, ABORTED из-за роста времён ответов | Контролировать наличие запаса ресурсов для закрытия ДЦ; |
| Переполнение ретраями пула сессий на клиенте | Подбор размера пула на клиенте;<br/>Подбор таймаутов;<br/>Короткоживущие сессии; |
| Переполнение ретраями пула сессий на сервере | Подбор размера пула на клиенте;<br/>Подбор таймаутов;<br/>Deadline propagation;<br/>Отмена ненужных запросов; |


### После учений

После учений система вернётся к работе в обычном режиме не сразу. Данные, которые были записаны во время учений, не представлены на дисках открывшегося ДЦ, и чтобы их прочитать – нужны будут кросс-ДЦ походы. На времени обработки запросов это скажется не слишком хорошо - но, скорее всего, запросов, которые в таких данных нуждаются, будет мало. Таблетки, обслуживающие партиции, будут переезжать в открывшийся ДЦ постепенно. Но это не касается фолловеров - вот они (в случае настройки "по одному на ДЦ", "PER_AZ:1") как раз будут запущены ещё во время открытия ДЦ, и вот запросы к ним (то есть одношардовые запросы в режиме StaleRO), относящиеся к данным, записанным во время учений, как раз и пострадают. 

Итак, что плохого мы можем получить после учений:
| Проблема | Что делать |
| -------- | ---------- |
| Рост времён ответов в верхних перцентилях в режимах транзакций SerializableRW, SnapshotRO, OnlineRO | Дождаться синхронизации дисков в открывшийся ДЦ |
| Рост числа ошибок TIMEOUT, ABORTED из-за роста времён ответов в режимах транзакций SerializableRW, SnapshotRO, OnlineRO | Дождаться синхронизации дисков в открывшийся ДЦ |
| Рост времён ответов в верхних перцентилях в режимах транзакций StaleRO | Дождаться синхронизации дисков в открывшийся ДЦ;<br/>Отключать предпочтение ближайшего ДЦ в закрываемом ДЦ на время учений и ещё какое-то время; |
| Рост числа ошибок TIMEOUT, ABORTED из-за роста времён ответов  в режимах транзакций StaleRO | Дождаться синхронизации дисков в открывшийся ДЦ;<br/>Отключать предпочтение ближайшего ДЦ в закрываемом ДЦ на время учений и ещё какое-то время; |
| Переполнение ретраями пула сессий на клиенте | Подбор размера пула на клиенте;<br/>Подбор таймаутов;<br/>Короткоживущие сессии; |
| Переполнение ретраями пула сессий на сервере | Подбор размера пула на клиенте;<br/>Подбор таймаутов;<br/>Deadline propagation;<br/>Отмена ненужных запросов; |

Худшие latency в запросах, которые обрабатываются даташардами из "отучившегося" ДЦ.

## Примеры кода

### Плохая практика: Долгоживущие сессии

```typescript
// ❌ ПЛОХО: Одна сессия используется для всех операций
class BadSessionManager {
    private session: Session;
    
    async init() {
        this.session = await driver.tableClient.withSession(async (session) => session);
    }
    
    async query(data: string) {
        // Сессия может быть уже закрыта сервером
        return await this.session.executeQuery(`SELECT * FROM table WHERE data = "${data}"`);
    }
}
```

### Хорошая практика: Пул сессий

```typescript
// ✅ ХОРОШО: Использование пула сессий
class GoodSessionManager {
    async query(data: string) {
        // Каждая операция получает свежую сессию из пула
        return await driver.tableClient.withSession(async (session) => {
            return await session.executeQuery(`SELECT * FROM table WHERE data = "${data}"`);
        });
    }
}
```

## Дополнительные рекомендации

1. **Используйте встроенные ретраеры SDK** - они уже содержат оптимизированную логику обработки ошибок
2. **Настройте адекватные таймауты** - увеличивайте таймауты во время учений
3. **Мониторьте метрики** - отслеживайте latency и количество ошибок
4. **Тестируйте заранее** - проводите нагрузочное тестирование с имитацией закрытия ДЦ
