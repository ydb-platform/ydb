# Закрепление ожиданий в интеграционных тестах

## Проблема

При обновлении версии {{ ydb-short-name }} или изменении конфигурации кластера могут возникать неожиданные изменения в поведении системы. Без надлежащего тестирования такие изменения могут остаться незамеченными до продакшена, что приведет к инцидентам и простоям.

**Основные риски:**
- Изменение производительности запросов
- Ошибки в обработке граничных случаев
- Изменение недокументированных деталей поведения

## Решение

Создание и поддержание набора интеграционных тестов, которые закрепляют ожидаемое поведение {{ ydb-short-name }} в различных сценариях. Тесты должны проверять как успешные операции, так и обработку ошибок.

### Лучшие практики

1. **Использование тестового контейнера {{ ydb-short-name }}**
   - Запуск одноузлового {{ ydb-short-name }} в CI/CD окружении через Docker
   - Тестирование с несколькими версиями {{ ydb-short-name }} (текущей, которая используется у вас в production и более новыми: для тестирования изменений).

2. **Проверка всех критических сценариев**
   - Базовые CRUD операции
   - Обработка ошибок и исключительных ситуаций
   - Производительность ключевых запросов
   - Поведение сервера, на которое полагается программа, но оно не описано как гарантия (Например недокументированный низкоуровневый код ошибки).

3. **Изоляция и чистота тестов**
   - Каждый тест начинается с чистого состояния
   - Каждый тест самодостаточен и не полагается на результаты от результатов выполнения других тестов: в т.ч. не полагается на оставшиеся таблицы, данные и т.п.

4. **Производительность тестов**
   - Тесты должны быть достаточно быстрыми, чтобы не мешать разработке
   - Можно запускать несколько тестов параллельно
   - Параллельно выполняющиеся тесты можно запускать в одной базе, если они работают в разных папках
   - Подготовка структуры базы (удаление и пересоздание таблиц, топиков и т.п.) делать в начале теста: В CI база удаляется целиком и не нужно тратить время на её очистку, при ручных запусках можно видеть результаты работы теста 

### Рекомендации по внедрению

1. **Интеграция в CI/CD** - автоматический запуск тестов при каждом коммите
2. **Тестирование нескольких версий {{ ydb-short-name }}** - для раннего выявления изменений
3. **Мониторинг производительности** - закрепление базовых метрик производительности
4. **Регулярное обновление тестов** - актуализация при изменении бизнес-логики

Такие тесты становятся "живой документацией" ожидаемого поведения системы и надежной защитой от регрессий при обновлениях инфраструктуры.

### Примеры тестов
Подходы едины во всех языках, а нюансы могут отличаться, поэтому приведу примеры на псевдокоде

#### ❌ Плохой пример - медленно, ненадёжно и неудообно

```
CreateDbInCloud() # Долгое облачной базы, нельзя выбрать версию
...
выполнять один тест за другим, т.к. тесты будут мешать друг другу в одной схеме - увеличивает время прохождения тестов

function TestMyFunc1:
    db = connect_to_external_ydb(...)

    # при повторном запуске теста без очистки код упадёт
    db.exec("CREATE TABLE t (id Int64, val Int64, PRIMARY KEY (id))") 
    ...
    db.exec("DROP TABLE t)
    код теста

function TestMyFunc2:
    db = connect_to_external_ydb(...)
    db.exec("CREATE TABLE t (id2 Int64, val2 Int64, PRIMARY KEY (id2))") 
    ...
    db.exec("DROP TABLE t)
    код теста

DeleteDbInCloud() # при ошибках в тестах/падении и т.п. база может и не удалиться.
```

#### ✅ Хороший пример - комплексное тестирование, быстрые прогоны, удобно смотреть что получилось не так

```
# Через тег можно выбрать конкретную версию, можно прогонять тесты 2/3 раза в разными версиями и тестировать правильную работу кода с новыми/старыми версиями YDB
# контейнер запускается быстро
# Подробнее: [quickstart](../../quickstart?tabs=defaultTabsGroup-d2n2xrov_docker%2520x86_64)
EnsureRunYDBContainer(image=ydbplatform/local-ydb:latest) 

# Выполнять тесты параллельно, каждый в своём фолдере, тогда не будет пересечения по схемам и данным

function TestMyFunc1:
    folder_name = testName()
    db = connect_to_external_ydb(...)
    # если таблица осталась с прошлого запуска - она удаляется
    # а после выполнения теста можно при необходимости посмотреть что там получилось - бывает полезно при отладке
    db.exec("
       Pragma TablePathPrefix({folder_name});
       DROP TABLE IF EXISTS t;
    ") 
    db.exec("
       Pragma TablePathPrefix({folder_name});
       CREATE TABLE t (id Int64, val Int64, PRIMARY KEY (id));
    ") # при повторном запуске теста без очистки код упадёт
    ...
    код теста

function TestMyFunc2:
    folder_name = testName()
    db = connect_to_external_ydb(...)
    # если таблица осталась с прошлого запуска - она удаляется
    # а после выполнения теста можно при необходимости посмотреть что там получилось - бывает полезно при отладке
    db.exec("
      Pragma TablePathPrefix({folder_name});
      DROP TABLE IF EXISTS t;
    ") 
    db.exec("
       Pragma TablePathPrefix({folder_name});
       CREATE TABLE t (id2 Int64, val2 Int64, PRIMARY KEY (id2))
    ")
    ...
    код теста

# опционально - если в CI выполняется полная очистка тестового окружения
# например удаление контейнера или виртуальной машины - то базу можно отдельно не чистить и не выключать - это быстрее
# При ручных прогонах теста тоже удобнее, чтобы она осталась работать - тогда не надо будет запускать контейнер на следующей итерации
# и можно посмотреть что в базе осталось
StopDockerContainer() 
```
