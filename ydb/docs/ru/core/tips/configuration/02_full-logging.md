# Оптимальный уровень логирования для продакшена

## Проблема

Настройка оптимального уровня логирования для продакшена помогает избежать излишнего расхода CPU и дополнительной latency от полного логирования и трассировки. Чрезмерно детальное логирование может:

- **Увеличить нагрузку на CPU** на 10-30% из-за генерации и записи большого объема данных
- **Добавить latency** к запросам из-за синхронной записи логов
- **Заполнить дисковое пространство** большими лог-файлами
- **Усложнить анализ** из-за информационного шума

## Решение

### Уровни логирования в {{ ydb-short-name }}

{{ ydb-short-name }} использует 9 уровней логирования (от наиболее критичных к наименее):

| Уровень | Числовое значение | Описание |
|---------|------------------|----------|
| `EMERG` | 0 | Возможен сбой системы (например, отказ кластера) |
| `ALERT` | 1 | Возможна деградация системы |
| `CRIT` | 2 | Критическое состояние |
| `ERROR` | 3 | Некритическая ошибка |
| `WARN` | 4 | Предупреждение, требующее реакции |
| `NOTICE` | 5 | **Рекомендуемый для продакшена** - существенные события |
| `INFO` | 6 | Отладочная информация для статистики |
| `DEBUG` | 7 | Отладочная информация для разработчиков |
| `TRACE` | 8 | Очень подробная отладочная информация |

### Рекомендуемая конфигурация для продакшена

```yaml
# ХОРОШО: Оптимальная конфигурация для продакшена
log_config:
  default_level: 5  # NOTICE - основной уровень для продакшена
  default_sampling_level: 7  # DEBUG - уровень для семплирования
  default_sampling_rate: 10  # Логировать каждое 10-е DEBUG сообщение
  
  # Настройка для конкретных компонентов
  entry:
    - component: "BLOBSTORAGE"
      level: 5  # NOTICE
      sampling_level: 8  # TRACE
      sampling_rate: 100  # Логировать каждое 100-е TRACE сообщение
    
    - component: "KQP"
      level: 5  # NOTICE
      sampling_level: 7  # DEBUG
      sampling_rate: 50  # Логировать каждое 50-е DEBUG сообщение
  
  # Оптимизация производительности
  allow_drop_entries: true  # Разрешить отбрасывание логов при перегрузке
  time_threshold_ms: 1000   # Период сброса буфера логов
  
  # Формат вывода
  format: "json"  # Для удобства парсинга
  cluster_name: "production-cluster"
```

### Примеры плохого и хорошего кода

#### ❌ Плохо: Полное логирование в продакшене

```yaml
# ПЛОХО: Слишком детальное логирование для продакшена
log_config:
  default_level: 8  # TRACE - слишком детально
  default_sampling_level: 8  # TRACE
  default_sampling_rate: 1   # Логировать ВСЕ сообщения
  
  entry:
    - component: "*"  # Все компоненты
      level: 8        # TRACE для всего
      sampling_level: 8
      sampling_rate: 1
```

**Проблемы:**
- Генерирует огромный объем логов
- Значительно увеличивает нагрузку на CPU
- Затрудняет поиск реальных проблем
- Быстро заполняет дисковое пространство

#### ✅ Хорошо: Сбалансированная настройка

```yaml
# ХОРОШО: Сбалансированная конфигурация
log_config:
  default_level: 5  # NOTICE - только важные события
  default_sampling_level: 7  # DEBUG - для отладки
  default_sampling_rate: 20  # 5% DEBUG сообщений
  
  # Компоненты с разной детализацией
  entry:
    - component: "SCHEMESHARD"
      level: 5  # NOTICE
    - component: "TABLET_MAIN"
      level: 6  # INFO - более детально для табличных операций
    - component: "BLOBSTORAGE"
      level: 4  # WARN - только предупреждения и ошибки
      sampling_level: 8
      sampling_rate: 100
  
  allow_drop_entries: true
  time_threshold_ms: 500  # Более частая запись для снижения потерь
```

### Настройка трассировки

Для трассировки используйте семплирование вместо полного трейсинга:

```yaml
# ХОРОШО: Семплирование трассировки
tracing_config:
  sampling:
    - level: 10  # Detailed - достаточно для диагностики
      fraction: 0.01  # 1% запросов
    - level: 14  # Diagnostic - для глубокой отладки
      fraction: 0.001 # 0.1% запросов
```

### Мониторинг и динамическое изменение

Используйте встроенный UI или API для мониторинга нагрузки от логирования:

```bash
# Проверка текущих настроек логирования
ydb --endpoint <endpoint> admin blobstorage config show --json

# Временное увеличение детализации для отладки
ydb --endpoint <endpoint> admin log set-level --component KQP --level DEBUG --duration 1h
```

### Мониторинг и алертинг

#### Дашборды для мониторинга логирования

Создайте дашборды для отслеживания:

1. **Объем логов в секунду** по компонентам
2. **Распределение по уровням** логирования
3. **Размер лог-файлов** на диске
4. **CPU нагрузка** от логирования

#### Пример алерта для Prometheus

```yaml
# Alert при чрезмерном объеме логов
groups:
- name: ydb_logging
  rules:
  - alert: YDBLogVolumeHigh
    expr: rate(ydb_log_messages_total[5m]) > 10000
    for: 5m
    labels:
      severity: warning
    annotations:
      summary: "Высокий объем логов YDB"
      description: "Объем логов превышает 10к сообщений в секунду"

  - alert: YDBLogDiskUsageHigh
    expr: (disk_used_bytes{device=~".*ydb.*"} / disk_total_bytes{device=~".*ydb.*"}) > 0.8
    for: 10m
    labels:
      severity: critical
    annotations:
      summary: "Высокое использование диска логами YDB"
      description: "Использование диска для логов превышает 80%"
```

### Динамическое управление логированием

Используйте API для временного увеличения детализации при проблемах:

```bash
# Временное включение DEBUG для диагностики
ydb admin log set-level --component KQP --level DEBUG --duration 30m

# Возврат к нормальным настройкам
ydb admin log set-level --component KQP --level NOTICE

# Просмотр текущих настроек
ydb admin log get-level --component KQP
```

### Практические рекомендации

1. **Начинайте с NOTICE (5)** как основного уровня
2. **Используйте семплирование** для DEBUG/TRACE уровней (5-20% для DEBUG, 0.1-1% для TRACE)
3. **Настройте ротацию логов** с ограничением по размеру (например, 1GB) и времени (7 дней)
4. **Мониторьте размер лог-файлов** и нагрузку на CPU от логирования
5. **Используйте разные уровни для разных компонентов**:
   - `SCHEMESHARD`, `KQP`: NOTICE/INFO
   - `BLOBSTORAGE`: WARN/NOTICE с семплированием TRACE
   - `TABLET_MAIN`: INFO с семплированием DEBUG
6. **Тестируйте настройки** на staging-окружении перед применением в продакшене
7. **Используйте структурированное логирование** (JSON формат) для удобства анализа

### Резюме по производительности

| Настройка | Влияние на CPU | Влияние на Latency | Объем логов |
|-----------|----------------|-------------------|-------------|
| TRACE уровень | +20-30% | +5-10ms | Очень высокий |
| DEBUG уровень | +10-15% | +2-5ms | Высокий |
| NOTICE уровень | +1-3% | +0.1-0.5ms | Умеренный |
| Семплирование 10% | +2-5% | +0.5-1ms | Контролируемый |

Такая настройка обеспечивает баланс между наблюдаемостью и производительностью в продакшен-среде, позволяя эффективно диагностировать проблемы без значительного влияния на производительность системы.
