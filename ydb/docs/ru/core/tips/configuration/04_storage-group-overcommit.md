# Учет оверкоммита групп хранения при планировании

## Проблема

Коэффициент переподписки (`storage_overcommit`) в {{ ydb-short-name }} позволяет системе учитывать, насколько можно перераспределить нагрузку между группами хранения при автобалансировке каналов. Однако неправильная настройка этого параметра может привести к серьезным проблемам:

1. **Недогрузка или перегрузка групп хранения** - при слишком низком коэффициенте система не использует группы хранения полностью, что приводит к неэффективному распределению ресурсов. При слишком высоком коэффициенте возникает перегрузка групп хранения, увеличиваются задержки и снижается производительность.

2. **Нарушение балансировки** - неправильная настройка нарушает механизм автобалансировки, что приводит к неравномерному распределению нагрузки и снижению отказоустойчивости.

3. **Проблемы с производительностью** - неоптимальная настройка приводит к неэффективному использованию ресурсов, увеличению времени ответа на запросы и снижению общей эффективности базы данных.

## Решение

### Понимание параметра storage_overcommit

Параметр `storage_overcommit` в конфигурации Hive определяет коэффициент переподписки на ресурсы групп хранения. Значение по умолчанию - `1.0`, что означает отсутствие переподписки.

### Рекомендации по настройке

1. **Для стабильных рабочих нагрузок** используйте значение по умолчанию `1.0`
2. **Для рабочих нагрузок с пиками** можно установить значение `1.1-1.3` для лучшего использования ресурсов
3. **Избегайте значений выше 1.5** без тщательного мониторинга и тестирования
4. **Регулярно мониторьте** метрики использования групп хранения через Hive web-viewer

### Мониторинг и диагностика

Используйте следующие метрики для контроля состояния групп хранения:
- Потребление места на дисках
- IOPS и пропускная способность
- Статус балансировки каналов
- Количество перераспределений каналов


### Пример плохой конфигурации

```yaml
# config.yaml - НЕПРАВИЛЬНО
hive_config:
  storage_overcommit: 2.5  # Слишком высокий коэффициент
  storage_safe_mode: false # Отключена защита от перегрузки
  storage_balance_strategy: "HIVE_STORAGE_BALANCE_STRATEGY_SIZE"
```

**Проблемы:**
- Высокий коэффициент переподписки (2.5) может привести к перегрузке групп хранения
- Отключенный безопасный режим позволяет превышать лимиты ресурсов
- Риск деградации производительности при росте нагрузки

### Пример хорошей конфигурации

```yaml
# config.yaml - ПРАВИЛЬНО
hive_config:
  storage_overcommit: 1.1  # Умеренный коэффициент для пиковых нагрузок
  storage_safe_mode: true  # Включена защита от перегрузки
  storage_balance_strategy: "HIVE_STORAGE_BALANCE_STRATEGY_AUTO"
  min_period_between_reassign: 300
```

**Преимущества:**
- Умеренный коэффициент позволяет эффективно использовать ресурсы без риска перегрузки
- Включенный безопасный режим предотвращает критические ситуации
- Автоматический выбор стратегии балансировки адаптируется к текущей нагрузке
- Минимальный период между перераспределениями предотвращает излишнюю миграцию данных

### Пример мониторинга через CLI

```bash
# Проверка текущего состояния групп хранения
ydb admin bs config view --domain <domain-name>

# Мониторинг метрик Hive
ydb monitoring hive --format json

# Проверка балансировки каналов
ydb admin tablet stat --tablet <tablet-id>
```

### Пример диагностики проблем

```sql
-- Проверка распределения данных по группам хранения
SELECT
    storage_pool_id,
    COUNT(*) as tablet_count,
    SUM(data_size) as total_size
FROM system.tablets
GROUP BY storage_pool_id
ORDER BY total_size DESC;
```


## Ключевые выводы

1. **Начинайте с консервативных значений** - используйте значение по умолчанию 1.0 для новых систем
2. **Настраивайте постепенно** - увеличивайте коэффициент только при наличии обоснованной необходимости
3. **Мониторьте постоянно** - регулярно отслеживайте метрики использования ресурсов
4. **Тестируйте изменения** - проверяйте влияние изменений коэффициента на тестовых окружениях
5. **Используйте безопасный режим** - всегда оставляйте включенным `storage_safe_mode` для защиты от перегрузок

Правильная настройка коэффициента переподписки групп хранения - ключевой фактор стабильной работы {{ ydb-short-name }} кластера при росте данных и изменении рабочих нагрузок.
