# Оптимальная настройка run-interval для TTL

## Проблема

Настройка оптимального run-interval для TTL важна: увеличение без необходимости может не дать ожидаемого улучшения производительности, но задержит удаление данных

## Решение

При выборе значения `run-interval` для TTL нужно учитывать баланс между задержкой удаления просроченных данных и дополнительной нагрузкой на кластер. Слишком маленький интервал приводит к частому запуску фоновых задач удаления и может заметно увеличить нагрузку на CPU и I/O, особенно при большом объёме данных. Слишком большой интервал, наоборот, задерживает физическое удаление записей: логически «просроченные» строки будут ещё долго занимать место на диске и участвовать в сканированиях.

Практический подход:

- ориентируйтесь на допустимую «устарелость» данных в вашем сценарии (SLA по тому, через какое время после наступления TTL запись должна быть реально удалена);
- устанавливайте `run-interval` не меньше этого значения и, как правило, кратным ему (например, 5–15 минут для фоновой очистки журналов или метрик);
- начинайте с относительно консервативного интервала (например, 10–15 минут) и постепенно уменьшайте его, наблюдая за нагрузкой на кластер и временем жизни просроченных записей;
- для таблиц с редкими изменениями и небольшим объёмом данных можно использовать более редкий запуск, а для высоконагруженных таблиц с большим количеством истекающих записей — более частый, но всё равно не «ежесекундный».

Важно настраивать `run-interval` отдельно для разных классов таблиц (если это поддерживается конфигурацией) и периодически пересматривать значение по мере роста объёма данных и изменения профиля нагрузки. Это позволит избежать как лишней нагрузки от слишком частого удаления, так и накопления большого количества «мусорных» строк.
