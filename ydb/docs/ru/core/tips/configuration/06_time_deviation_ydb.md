# Расхождение времени на хостах {{ ydb-short-name }}

## Проблема

Расхождение системного времени между узлами {{ ydb-short-name }} является критической проблемой, которая может привести к серьезным последствиям для работы распределенной базы данных. Основные проблемы:

### Замедление выполнения распределенных транзакций
- Время выполнения транзакций увеличивается ровно на величину расхождения времени между узлами
- При расхождении в 1 секунду транзакции будут выполняться на 1 секунду дольше
- Это связано с механизмом координации транзакций, который требует согласованного времени на всех узлах

### Невозможность выполнения транзакций
- При расхождении времени **более 30 секунд** транзакции над данными перестают выполняться полностью
- Кластер становится недоступным для операций записи и чтения

### Проблемы с дедлайнами и таймаутами
- Неправильное срабатывание дедлайнов из-за рассинхронизации времени
- Ложные ошибки таймаутов транзакций
- Проблемы с аутентификацией (истечение токенов)

### Критические пороговые значения
- **5 секунд** - порог для предупреждений (yellow)
- **25 секунд** - порог для критических событий (orange)  
- **30 секунд** - полная недоступность транзакций

## Диагностика

### Мониторинг через встроенный UI {{ ydb-short-name }}
1. Перейдите на вкладку **Databases** во встроенном UI
2. Выберите нужную базу данных
3. Откройте вкладку **Diagnostics** → **Healthcheck**
4. Проверьте раздел **NODES_TIME_DIFFERENCE**

### Командная диагностика
```bash
# Проверка времени на всех узлах
pssh -h hosts.txt "date +%s%N"

# Проверка статуса синхронизации времени
timedatectl show-timesync --all
chronyc sources -v
```

### Метрики мониторинга
- **Interconnect ClockSkew** - разница времени между нодами
- **Transaction Latency** - время выполнения транзакций
- **GRPC Success Rate** - процент успешных запросов

## Решение

### Настройка синхронизации времени

#### Рекомендуемые NTP серверы
```bash
# Конфигурация для chrony (/etc/chrony/chrony.conf)
server ntp1.yandex.net iburst
server ntp2.yandex.net iburst  
server ntp3.yandex.net iburst
server ntp4.yandex.net iburst
maxslewrate 10000
local stratum 10
```

#### Конфигурация для systemd-timesyncd
```ini
# /etc/systemd/timesyncd.conf.d/yandex.conf
[Time]
NTP=ntp1.yandex.net ntp2.yandex.net ntp3.yandex.net ntp4.yandex.net
FallbackNTP=ntp1.yandex.net ntp2.yandex.net ntp3.yandex.net ntp4.yandex.net
```

### Мониторинг и алертинг

#### Критические алерты
- **ydb_ntp** - алерт на расхождение времени
- **Interconnect ClockSkew** превышение порогов
- **Transaction Latency** аномальный рост

#### Рекомендуемые пороги
- Предупреждение: > 5 секунд
- Критическое: > 25 секунд
- Аварийное: > 30 секунд

### Аварийное восстановление

#### При критическом расхождении
```bash
# Остановка синхронизации и принудительная синхронизация
sudo systemctl stop chrony
sudo chronyd -q 'server ntp1.yandex.net iburst'

# Или для systemd-timesyncd
sudo systemctl stop systemd-timesyncd
sudo systemd-timesyncd --force
```

#### План восстановления
1. Остановить все демоны синхронизации времени на кластере
2. Принудительно синхронизировать с надежным NTP сервером
3. Перезапустить демоны синхронизации
4. Проверить сходимость времени через мониторинг

## Профилактика

### Регулярные проверки
- Ежедневный мониторинг метрик ClockSkew
- Периодические учения по синхронизации времени
- Тестирование отказоустойчивости NTP инфраструктуры

### Рекомендации по архитектуре
- Использование минимум 4 независимых NTP источников
- Развертывание собственного слоя NTP серверов {{ ydb-short-name }}
- Регулярное обновление конфигурации синхронизации

### Важные замечания
- Все узлы кластера должны использовать одинаковые источники времени
- Рекомендуется использовать chrony вместо systemd-timesyncd для лучшей точности
- Регулярно проверяйте статус синхронизации на всех узлах
