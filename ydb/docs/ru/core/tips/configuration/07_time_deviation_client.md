# Расхождение времени между клиентскими хостами и кластером {{ ydb-short-name }}

## Проблема

Расхождение системного времени между клиентскими хостами и кластером {{ ydb-short-name }} приводит к серьезным проблемам с аутентификацией и работой приложений. Основные проблемы:

### Инвалидация токенов аутентификации
- **JWT токены** имеют временные ограничения (expiration time)
- При расхождении времени клиент может отправлять токены, которые кластер считает устаревшими
- Ошибки аутентификации: `Token expired`, `Invalid token`

### Проблемы с временными метками транзакций
- Клиентские приложения используют локальное время для временных меток
- При расхождении времени транзакции могут получать некорректные временные метки
- Возможны конфликты при обработке временных интервалов

### Ошибки дедлайнов и таймаутов
- Клиентские дедлайны рассчитываются на основе локального времени
- При расхождении времени дедлайны могут срабатывать преждевременно или с задержкой
- Ложные ошибки таймаутов и прерывание операций

### Критические пороговые значения
- **30 секунд** - стандартный порог для JWT токенов
- **5 минут** - максимальное рекомендуемое расхождение для стабильной работы
- **Более 10 минут** - критическое расхождение, приводящее к постоянным ошибкам

## Диагностика

### Проверка времени на клиентских хостах
```bash
# Проверка системного времени
date

# Проверка статуса синхронизации времени
timedatectl show-timesync --all
chronyc sources -v

# Сравнение времени с кластером YDB
curl -s https://ydb-cluster.example.com/ping | grep -i date
```

### Мониторинг ошибок аутентификации
- Логи приложений: поиск ошибок `Token expired`, `Authentication failed`
- Мониторинг метрик GRPC: `grpc_client_errors`, `auth_failures`
- Анализ HTTP статусов: 401 Unauthorized, 403 Forbidden

### Проверка расхождения времени
```bash
# Скрипт для проверки расхождения времени
#!/bin/bash
CLUSTER_TIME=$(curl -sI https://ydb-cluster.example.com/ping | grep '^Date:' | cut -d' ' -f3-6)
CLUSTER_TS=$(date -d "$CLUSTER_TIME" +%s)
LOCAL_TS=$(date +%s)
DIFF=$((LOCAL_TS - CLUSTER_TS))

echo "Расхождение времени: $DIFF секунд"
if [ $DIFF -gt 30 ]; then
    echo "ВНИМАНИЕ: Критическое расхождение времени!"
fi
```

## Решение

### Настройка синхронизации времени на клиентских хостах

#### Рекомендуемые NTP серверы для клиентов
```bash
# Конфигурация для chrony (/etc/chrony/chrony.conf)
server ntp1.yandex.net iburst
server ntp2.yandex.net iburst
server ntp3.yandex.net iburst
server ntp4.yandex.net iburst
maxslewrate 10000
local stratum 10
```

#### Конфигурация для systemd-timesyncd
```ini
# /etc/systemd/timesyncd.conf.d/yandex.conf
[Time]
NTP=ntp1.yandex.net ntp2.yandex.net ntp3.yandex.net ntp4.yandex.net
FallbackNTP=ntp1.yandex.net ntp2.yandex.net ntp3.yandex.net ntp4.yandex.net
```

### Настройка приложений для работы с расхождением времени

#### Увеличение времени жизни токенов
```yaml
# Конфигурация приложения
auth:
  token_lifetime: 3600  # 1 час вместо 30 минут
  clock_skew_tolerance: 300  # Допуск расхождения 5 минут
```

#### Использование времени кластера для критичных операций
```python
# Пример для Python приложения
import requests
from datetime import datetime

def get_cluster_time():
    response = requests.head('https://ydb-cluster.example.com/ping')
    cluster_time = response.headers.get('Date')
    return datetime.strptime(cluster_time, '%a, %d %b %Y %H:%M:%S GMT')

# Использование времени кластера для токенов
cluster_time = get_cluster_time()
token = create_jwt_token(expires_at=cluster_time + timedelta(hours=1))
```

### Мониторинг и алертинг

#### Критические алерты для клиентских хостов
- **client_clock_skew** - расхождение времени с кластером
- **auth_token_expired** - количество ошибок истечения токенов
- **grpc_deadline_exceeded** - ошибки дедлайнов

#### Рекомендуемые пороги
- Предупреждение: расхождение > 30 секунд
- Критическое: расхождение > 5 минут
- Аварийное: расхождение > 10 минут

### Аварийное восстановление

#### При критическом расхождении времени
```bash
# Принудительная синхронизация времени
sudo systemctl stop chrony
sudo chronyd -q 'server ntp1.yandex.net iburst'
sudo systemctl start chrony

# Проверка синхронизации
chronyc tracking
```

#### Временное решение для приложений
- Увеличить время жизни токенов
- Использовать аутентификацию по ключам вместо токенов
- Реализовать механизм автоматического обновления токенов

## Профилактика

### Регулярные проверки
- Ежедневный мониторинг расхождения времени на всех клиентских хостах
- Автоматические проверки синхронизации времени при развертывании
- Периодические тесты аутентификации с различными расхождениями времени

### Рекомендации по архитектуре
- Все клиентские хосты должны использовать одинаковые источники времени
- Реализовать механизм автоматической коррекции времени в приложениях
- Использовать NTP серверы с высокой доступностью и точностью

### Важные замечания
- Время на клиентских хостах должно быть синхронизировано с кластером {{ ydb-short-name }}
- Рекомендуется использовать chrony для лучшей точности синхронизации
- Регулярно проверяйте статус синхронизации на всех клиентских хостах
- Имейте план действий на случай критического расхождения времени
