# Память и CPU: узлы хранения и узлы БД на одном хосте

## Проблема

На одной виртуальной машине или физическом сервере часто запускают несколько процессов {{ ydb-short-name }} (`ydbd`) с разными ролями — например, [узел хранения](../../concepts/glossary.md#storage-node) ([статический узел](../../concepts/glossary.md#static-node)) и [динамический узел](../../concepts/glossary.md#dynamic) ([узел базы данных](../../concepts/glossary.md#database-node)). Без явных лимитов памяти процессы конкурируют за оперативную память всего хоста: при нехватке памяти ОС может завершить процесс ([OOM killer](https://en.wikipedia.org/wiki/Out_of_memory#Linux)), чаще всего страдает наиболее прожорливый узел — как правило, вычислительный. Это выглядит «шумно» при отладке клиентов и SDK: падения непредсказуемы и слабо связаны с логикой приложения.

В [справочнике по конфигурации](../../reference/configuration/memory_controller_config.md) описано, как ограничить память через [cgroups](https://en.wikipedia.org/wiki/Cgroups) и секцию `memory_controller_config`. Курсы и руководства по развёртыванию обычно не дают готовой «пропорции на 8 ГБ RAM между storage и dynamic» — из‑за этого неочевидно, как планировать маленькие хосты и сколько узлов какого типа на них уместить без постоянного риска OOM.

## Решение

### Ориентиры по CPU и RAM (рабочие нагрузки)

Ниже — не жёсткие лимиты продукта, а практические ориентиры для планирования ресурсов.

**Узлы базы данных** (включая динамические узлы, обрабатывающие запросы):

* ориентир **около 4 ГиБ оперативной памяти на один vCPU**;
* минимальная **разумная** рабочая конфигурация на один такой узел — **4 vCPU и 16 ГиБ RAM**;
* вариант **2 vCPU и 8 ГиБ** технически возможен, но **для продакшена пока не рекомендуется** — при росте нагрузки выше базовой выше риск нехватки памяти.

**Узлы хранения** в типичных сценариях **менее требовательны к памяти**, чем узлы БД:

* ориентир по соотношению ресурсов — порядка **1 vCPU на 2 ГиБ RAM** (в пересчёте на узел — меньше «ГиБ на vCPU», чем у узла БД);
* разумный минимум — **не менее 2 vCPU** на узел хранения.

**Очень маленькие конфигурации** (в том числе совмещение ролей на одном хосте с малым объёмом RAM) остаются нормой для **тестов и отладки**. Их нужно воспринимать как сознательный компромисс: при превышении базовой нагрузки **OOM возможен и ожидаем**, даже если конфигурация формально запускается.

### Совмещение ролей на одном хосте: обязательные лимиты памяти

Если на одной ВМ несколько процессов {{ ydb-short-name }}, **критически важно** задать **лимиты памяти на каждый процесс**, чтобы суммарное потребление не съедало всю память хоста и страничный кеш ОС.

* **Проще всего в продакшене** выставить лимиты через **cgroups** (в том числе через unit-файлы **systemd**, например `MemoryMax=` / `MemoryHigh=` для сервиса `ydbd`). Тогда [жёсткий лимит памяти](../../reference/configuration/memory_controller_config.md#hard-memory-limit) внутри процесса по умолчанию согласуется с cgroup, как описано в справочнике.
* **Дополнительно или вместо этого** можно задать лимиты в секции **`memory_controller_config`** (например, `hard_limit_bytes`). Имейте в виду: процесс всё ещё может в отдельных случаях оказаться выше заданного лимита, поэтому для **строгого** контроля **cgroups на уровне ОС остаются предпочтительными** в боевых средах — это также отражено в документации по `memory_controller_config`.

Лимиты нужно **согласовать с ролью узла**: под узел БД закладывать бюджет по ориентиру «~4 ГиБ на vCPU», под узел хранения — по более скромному соотношению выше. **Универсальной таблицы «возьми 60% storage / 40% dynamic» нет**: сначала считается потребность каждого процесса по роли и числу vCPU, затем проверяется, что **сумма cgroup-лимитов плюс запас на ОС** (порядка **1–2 ГиБ и выше**, плюс место под системные сервисы) **не превышает** физическую RAM хоста.

**Несколько узлов БД на одном хосте** — это суммирование бюджета: два динамических узла по 4 vCPU в ориентире потребуют порядка **двух** таких «16 ГиБ» профилей по памяти (если не снижать класс конфигурации осознанно), то есть отдельный хост с существенно большей RAM или разнесение по машинам.

**Пример на 8 ГиБ RAM одной ВМ:** по ориентиру выше **одного** полноценного узла БД (4 vCPU / 16 ГиБ) на такой объём уже **не хватает**; совместить на 8 ГиБ узел хранения и узел БД «по рекомендациям для прода» **нельзя без урезания класса и принятия риска OOM**. Такой хост остаётся сценарием **эксперимента или учебного стенда**.

### Разнесение узлов по отдельным ВМ

Практичный вариант для стабильности и предсказуемого использования ресурсов:

* выделить **отдельные виртуальные машины под узлы хранения** — их может быть много, на каждую закладывать **умеренный** объём RAM по ориентиру для storage (например, **4 vCPU / 8 ГиБ** как класс соотношения «1 к 2»);
* выделить **отдельные ВМ под динамические узлы (узлы БД)** — например, **4 vCPU / 16 ГиБ** на узел в каждом дата-центре (или вашей зоне отказа);
* **масштабировать вычисления** добавлением ВМ с новыми узлами БД при **общем** слое хранения.

Так вы уменьшаете взаимное влияние процессов, упрощаете лимиты (**один процесс — одна cgroup** на ВМ) и автоматизацию: те же правила можно **единообразно зашить в Ansible** или другой конфигурационный пайплайн (systemd + лимиты памяти по роли узла).

### Как грубо оценить «влезет ли» на хост (словесный калькулятор)

1. Для **каждого узла БД** на хосте: `RAM_узла_БД ≈ 4 × (число vCPU)` ГиБ; для прода планируйте **не меньше 4 vCPU** на узел, если нет явной причины обратного.
2. Для **каждого узла хранения**: `RAM_узла_хранения ≈ 2 × (число vCPU)` ГиБ (в том же смысле ориентира «1 vCPU : 2 ГиБ»); **не меньше 2 vCPU** на узел.
3. `RAM_хоста ≥ Σ RAM_лимитов_по_cgroup + RAM_ОС_запас` (на практике часто **от 2 ГиБ** и выше, зависит от мониторинга, агентов и т.д.).
4. Если сумма не сходится — **уменьшите число процессов на хосте**, **увеличьте RAM** или **разнесите роли по разным ВМ**.

**Числовой пример:** один узел хранения **4 vCPU / ~8 ГиБ** и один узел БД **4 vCPU / ~16 ГиБ** в ориентире безопаснее не складывать в одну ВМ на **24 ГиБ** «впритык»: нужен **запас на ОС** и пики, либо **две ВМ** (8 ГиБ и 16 ГиБ под типовые профили) плюс планирование следующего хоста под второй узел БД.

## Ключевые выводы

1. **Без лимитов памяти** (cgroups / systemd + при необходимости `memory_controller_config`) совмещение узлов на одном хосте **часто заканчивается OOM** при росте нагрузки.
2. **Узлы БД** планируйте **плотнее по RAM к CPU** (~**4 ГиБ на vCPU**), **узлы хранения** — **реже** (~**2 ГиБ на vCPU**), минимумы **4 vCPU / 16 ГиБ** (БД) и **2 vCPU** (хранение) для «разумного» прода.
3. **Маленькие и совмещённые** конфигурации — для **тестов**; для прода выгоднее **разнос по ВМ** и явные лимиты **в инфраструктуре** (в т.ч. в Ansible).

Подробности по внутренним лимитам и поведению при нехватке памяти см. в [{#T}](../../reference/configuration/memory_controller_config.md). Общие вопросы развёртывания — в разделе [{#T}](../../devops/index.md).
