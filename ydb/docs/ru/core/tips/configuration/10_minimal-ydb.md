# Минимальный {{ ydb-short-name }}: выбор конфигурации под задачу

## Проблема

Один из самых частых вопросов в сообществе {{ ydb-short-name }} звучит примерно так:

> «Хочу попробовать {{ ydb-short-name }} / поставить для разработки / развернуть в CI. Какое минимальное железо нужно? Можно ли уместить всё в одну виртуалку?»

Ответ зависит от того, **что именно вы хотите получить**. Ниже — три сценария с чёткими рекомендациями.

---

## Сценарий 1: Тестирование, CI, локальная разработка

**Используйте Docker-образ `local-ydb`.**

```bash
docker run -d \
  --name ydb-local \
  -p 2135:2135 -p 8765:8765 \
  -e GRPC_TLS_PORT=2135 \
  -e GRPC_PORT=2136 \
  -e MON_PORT=8765 \
  ydbplatform/local-ydb:latest
```

Образ `ydbplatform/local-ydb:latest` специально создан для сценариев «happy path»:

- Запускается на одном контейнере без внешних зависимостей
- Не требует настройки кластера, сертификатов, дисков
- Подходит для unit/integration-тестов и CI-пайплайнов
- Работает на любой машине с поддержкой Docker (включая ноутбуки)

{% note warning %}

Процессор должен поддерживать инструкции **AVX2** — это обязательное требование для запуска `ydbd`. Большинство современных x86-64 процессоров (Intel Haswell 2013+, AMD Ryzen) поддерживают AVX2.

{% endnote %}

### Ожидаемые ограничения

`local-ydb` — это намеренно упрощённая конфигурация. При работе с ней нормально видеть:

- **`BadSession`** — ожидаемая ошибка при ограниченных ресурсах контейнера; SDK умеет её обрабатывать автоматически
- Медленные запросы при нехватке памяти/CPU
- Потерю данных при перезапуске контейнера (если не примонтирован том)

Для сохранения данных между перезапусками:

```bash
docker run -d \
  --name ydb-local \
  -p 2135:2135 -p 8765:8765 \
  -v $(pwd)/ydb_data:/ydb_data \
  -e GRPC_TLS_PORT=2135 \
  -e MON_PORT=8765 \
  ydbplatform/local-ydb:latest
```

## Сценарий 2: «Минимальный» {{ ydb-short-name }} на своём железе — почему одноузловой не подходит для прода

Соблазн велик: поднять один узел {{ ydb-short-name }} на одной виртуалке и считать это «production-ready» инсталляцией. **Это не работает.**

### Почему одноузловой {{ ydb-short-name }} не обеспечивает отказоустойчивости

{{ ydb-short-name }} — распределённая система, спроектированная под модель отказов, при которой **отдельные узлы падают, а система продолжает работать**. Одноузловая конфигурация нарушает эту модель фундаментально:

1. **Нет репликации данных** — при `erasure=none` (единственный вариант для одного узла) данные хранятся без избыточности. Сбой диска = потеря данных.

2. **Проблема перезапуска storage** — при рестарте [узла хранения](../../concepts/glossary.md#storage-node) [динамические узлы](../../concepts/glossary.md#dynamic) не восстанавливают соединение автоматически. Требуется ручной рестарт всех динамических нод. В кластере эта проблема не возникает, потому что storage всегда доступен через другие узлы.

3. **Нет смысла в транзакционных гарантиях** — {{ ydb-short-name }} гарантирует ACID при условии, что кворум узлов доступен. На одном узле кворум = 1/1, то есть любой сбой = полная недоступность.

### Минимальная конфигурация для реальной отказоустойчивости

| Конфигурация | Узлов | Erasure | Выдерживает |
|---|---|---|---|
| `mirror-3-dc` | 3+ | mirror-3-dc | 1 узел из 3 |
| `block-4-2` | 8+ | block-4-2 | 2 узла из 8 |

Трёхузловая конфигурация с `mirror-3-dc` технически возможна, но считается «шаткой» — запас прочности минимален. Рекомендуемый минимум для production — **8 узлов** с `block-4-2`.

### Минимальные требования к ресурсам на узел

Подробнее — в статье [{#T}](09_memory-cpu-mixed-hosts.md). Краткие ориентиры:

| Роль | CPU | RAM | Диск |
|---|---|---|---|
| [Узел хранения](../../concepts/glossary.md#storage-node) | 2+ vCPU | 4+ ГиБ | SSD, желательно выделенный |
| [Узел базы данных](../../concepts/glossary.md#database-node) | 4+ vCPU | 16+ ГиБ | — |
| Совмещённый узел | 4+ vCPU | 16+ ГиБ | SSD, выделенный |

{% note info %}

{{ ydb-short-name }} изначально проектировался под выделенное железо с большими дисками. На виртуалках с маленькими дисками (особенно сетевыми) производительность будет существенно ниже ожидаемой.

{% endnote %}

## Сценарий 3: Нужна отказоустойчивость, но ресурсов мало

Если вам нужен **настоящий отказоустойчивый {{ ydb-short-name }}**, но нет возможности поднять кластер из 8+ узлов — используйте **Serverless {{ ydb-short-name }} в облаке**.

### Почему Serverless {{ ydb-short-name }} — правильный выбор при ограниченных ресурсах

- **Free tier** — небольшой объём запросов бесплатно
- **Микробиллинг** — оплата по числу «потроганных» строк (Request Units, RU), а не за зарезервированные ресурсы
- **Полная отказоустойчивость** — управляется облаком, данные реплицированы в нескольких ДЦ
- **Нет операционной нагрузки** — не нужно следить за узлами, дисками, обновлениями

### Как начать

1. Зарегистрируйтесь в [Yandex Cloud](https://cloud.yandex.ru/)
2. Создайте Serverless базу данных {{ ydb-short-name }} в консоли
3. Подключайтесь через стандартный {{ ydb-short-name }} SDK — API идентичен self-hosted {{ ydb-short-name }}

```bash
# Пример подключения через {{ ydb-short-name }} CLI
ydb \
  -e grpcs://ydb.serverless.yandexcloud.net:2135 \
  -d /ru-central1/b1g.../etn... \
  --iam-token-file ~/.ydb/token \
  scheme ls
```

## Итоговая схема выбора

```text
Что вам нужно?
│
├─► Локальная разработка / тесты / CI
│   └─► ydbplatform/local-ydb:latest (Docker)
│
├─► Production с отказоустойчивостью
│   │
│   ├─► Есть ресурсы на кластер (8+ узлов)?
│   │   └─► Self-hosted {{ ydb-short-name }}, erasure=block-4-2
│   │
│   └─► Нет ресурсов на кластер?
│       └─► Serverless {{ ydb-short-name }} (Yandex Cloud)
│
└─► «Просто один узел для прода»
    └─► ❌ Не делайте так. Одноузловой {{ ydb-short-name }} не обеспечивает
        отказоустойчивости и не соответствует ожидаемой
        модели поведения {{ ydb-short-name }} при сбоях.
```
