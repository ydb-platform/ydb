# Запросы YQL от ORM-фреймворков

## Проблема

ORM-фреймворки часто генерируют очень большие YQL-запросы, особенно при выполнении массовых операций (bulk insert, batch update). Это может привести к следующим проблемам:

- **Превышение лимита размера gRPC-сообщения** — {{ ydb-short-name }} имеет ограничение на размер gRPC-сообщения (обычно 64 МБ). При превышении этого лимита запрос завершается ошибкой ResourceExhausted.
- **Превышение лимита длины запроса** — сервер {{ ydb-short-name }} имеет ограничения на длину YQL-запроса 16 МБ.
- **Неэффективное использование ресурсов** — большие запросы требуют много времени на компиляцию.
- **Сложность исполнения** — очень длинные запросы сложнее оптимизировать и строить для них план выполнения.

Типичные сценарии, когда ORM генерирует большие запросы:

- Массовая вставка данных (INSERT с множеством VALUES).
- Массовое обновление данных через UPDATE .. SET .. WHERE col IN (?, ... ?) или сложные наборы условий.
- Массовое удаление данных через DELETE FROM .. WHERE col IN (?, ... ?) или сложные наборы условий.
- Сложные JOIN-запросы с множеством таблиц без указания вторичных индексов (в т.ч. когда инструмент не может выбрать оптимальный путь доступа к данным).

## Решение

Для оптимизации больших запросов от ORM-фреймворков используйте следующие подходы:

**1. Полагайтесь на нативную пакетную передачу параметров (List/Struct): инструменты преобразуют запрос под оптимальный формат {{ ydb-short-name }}**

Во многих интеграциях с {{ ydb-short-name }} (например, через **ADO.NET**, **Entity Framework**, **JDBC**) массовые операции выполняются не через генерацию тысяч плейсхолдеров (`?, ?, ?, ...`), а через передачу **одного параметра-пакета** — например `List<Struct<...>>`. При этом инструмент **преобразует исходную “ORM-логику” запроса** в компактный и более эффективный для {{ ydb-short-name }} вид: данные передаются параметром-списком, а сам YQL строится вокруг конструкции `AS_TABLE($list)`.

Пример:

```sql
DECLARE $rows AS List<Struct<
  id: Uint64,
  status: Utf8,
  updated_at: Timestamp
>>;

INSERT INTO documents
SELECT id, status, updated_at
FROM AS_TABLE($rows);
```

Этот подход делает запросы более оптимальными для выполнения в {{ ydb-short-name }}:

- текст YQL остаётся небольшим и стабильным;
- компиляция проще и быстрее;
- снижается вероятность упереться в лимит длины запроса (16 МБ).

Однако важно учитывать, что данные из списка всё равно отправляются одним gRPC-сообщением — поэтому можно упереться в лимит размера gRPC message (например, 64 МБ) уже не из‑за текста запроса, а из‑за размера самого списка/пакета данных.

Рекомендация: делите массовые операции на чанки (batch) по числу строк или по суммарному размеру payload и отправляйте их несколькими запросами.

**2. Если инструмент не смог “сгруппировать” запрос — используйте нативный YQL**

Не все запросы удаётся автоматически привести к компактному виду с AS_TABLE($list). В сложных сценариях (нетривиальные UPDATE/DELETE, сложные условия, join’ы, подзапросы, динамически собранные фильтры) ORM/библиотека может сгенерировать:

- слишком длинный текст запроса;
- большое число условий/выражений;
- структуру запроса, которая тяжело компилируется и/или приводит к неоптимальному плану.

В таких случаях рекомендуется перейти на нативный YQL, чтобы:

- явно передавать данные параметром-списком (List<Struct<...>>) и обрабатывать их через AS_TABLE;
- держать текст запроса небольшим и стабильным;
- лучше контролировать логику запроса и план выполнения (включая доступ по индексам);
- избежать превышения лимитов длины запроса и непредсказуемого времени компиляции.

**3. Используйте вторичные индексы и (где доступно) хинты/подсказки к ним**

Если массовые операции выполняются по полям, отличным от первичного ключа, важно, чтобы запрос использовал вторичные индексы. Некоторые инструменты и фреймворки (например, Hibernate, Spring Data JDBC в рамках соответствующих интеграций/диалектов) предоставляют механизмы/хинты для выбора индекса.

Это позволяет:

- избежать полного сканирования таблиц;
- получить более стабильную производительность;
- упростить оптимизацию плана выполнения.

**4. Для максимальной пропускной способности массовых вставок/обновлений используйте BulkUpsert**

Если сценарий сводится к массовой вставке/обновлению по ключу без сложной серверной логики, рассмотрите нативный RPC-вызов BulkUpsert:

- данные передаются эффективно (структурированно, без раздувания текста запроса);
- уменьшается нагрузка на компилятор YQL;
- обычно достигается лучшая пропускная способность на больших объёмах данных.
