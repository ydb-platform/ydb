# 1000 и одна строка {{ ydb-short-name }}

## Проблема

В {{ ydb-short-name }} у запросов через table-сервис ([`ExecuteDataQuery`](https://github.com/ydb-platform/ydb-api-protos/blob/master/ydb_table_v1.proto#L58)) есть [ограничение в 1000 строк](https://ydb.tech/docs/ru/concepts/limits-ydb#query) в ответе. Оно связано с OLTP-профилем этого API и использованием gRPC unary-вызовов (лимит ответа ~50 МБ). Для точечных запросов это обычно не проблема, но при выборке больших объёмов данных приходится либо реализовывать [пагинацию](https://ydb.tech/docs/ru/dev/paging) и собирать страницы на клиенте, либо переходить на [StreamExecuteScanQuery](https://ydb.tech/docs/ru/concepts/scan_query), где нет интерактивных транзакций и используется снимок на момент старта запроса. Оба варианта усложняют код и не всегда подходят под задачу.

## Решение

Используйте [query-сервис](https://github.com/ydb-platform/ydb-api-protos/blob/master/ydb_query_v1.proto): он выполняет [`ExecuteQuery`](https://github.com/ydb-platform/ydb-api-protos/blob/master/ydb_query_v1.proto#L38) в потоковом режиме и **снимает лимит в 1000 строк**, сохраняя поддержку интерактивных транзакций и работу с колоночными таблицами. Для production-сценариев query-сервис рекомендуется начиная с версии 24.3.

**Рекомендация:** переключите выполнение запросов на query-сервис через opt-in в вашем SDK, не переписывая приложение:

- **Go (table client):** `ydb.WithExecuteDataQueryOverQueryClient(true)` — все `ExecuteDataQuery` идут через `ExecuteQuery`.
- **Go (database/sql):** `ydb.WithQueryService(true)` в строке подключения или при создании драйвера (доступно с [v3.95.0](https://github.com/ydb-platform/ydb-go-sdk/releases/tag/v3.95.0)).
- **JDBC / ADO.NET:** query-сервис уже используется по умолчанию.

**Важно:** данные материализуются в памяти; при очень больших результатах возможен OOM — проектируйте запросы и лимиты с учётом этого. Ретраи при временных сбоях нужно настраивать и на вызов запроса, и на чтение стрима ответа.
