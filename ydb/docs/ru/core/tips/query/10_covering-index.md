# Ускорение запросов с помощью covering индексов

## Проблема

При использовании обычного вторичного индекса для выполнения запроса {{ ydb-short-name }} выполняет два шага:
1. **Поиск в индексе** — находит соответствующие ключи по условию WHERE
2. **Lookup в основную таблицу** — для каждого найденного ключа выполняет поиск в основной таблице, чтобы получить остальные поля запроса

Этот двухшаговый процесс (index scan + table lookup) может быть медленным, особенно при большом количестве найденных записей, так как:
- Требуется дополнительный доступ к основной таблице для каждого результата
- Увеличивается количество операций ввода-вывода
- Растет latency запроса
- Увеличивается нагрузка на систему

## Решение

Covering индекс (покрывающий индекс) — это индекс, который содержит все необходимые данные для запроса, включая не только ключевые поля, но и дополнительные колонки, используемые в SELECT. Это позволяет выполнить запрос, используя только индекс, без обращения к основной таблице.

**Преимущества covering индексов:**
- **Избежание lookup в основную таблицу** — все данные берутся из индекса
- **Снижение latency** — один шаг вместо двух
- **Уменьшение нагрузки на систему** — меньше операций ввода-вывода
- **Повышение производительности** — особенно заметно при большом количестве результатов

**Когда использовать:**
- Запросы, которые читают только индексированные поля и несколько дополнительных колонок
- Часто выполняемые запросы с фиксированным набором полей
- Запросы, где покрывающий индекс может полностью удовлетворить потребности SELECT

**Ограничения:**
- Covering индекс занимает больше места, так как хранит дополнительные данные
- Замедляет операции записи, так как нужно обновлять больше данных в индексе
- Не подходит для запросов, которые читают много разных полей


{% list tabs %}

- YQL

  {% cut "Плохой пример" %}
  ```yql
-- Плохой пример: обычный индекс требует lookup в основную таблицу
CREATE TABLE users (
    id Uint64,
    email Utf8,
    name Utf8,
    status Utf8,
    created_at Timestamp,
    PRIMARY KEY (id),
    INDEX idx_email GLOBAL ON (email)
);

-- Запрос использует индекс, но требует lookup для получения name и status
SELECT name, status FROM users VIEW idx_email WHERE email = 'user@example.com';
-- Выполняется: поиск в индексе + lookup в таблицу для каждого результата
  ```
  {% endcut %}

  {% cut "Хороший пример" %}
  ```yql
-- Хороший пример: covering индекс содержит все необходимые данные
CREATE TABLE users (
    id Uint64,
    email Utf8,
    name Utf8,
    status Utf8,
    created_at Timestamp,
    PRIMARY KEY (id),
    INDEX idx_email_covering GLOBAL ON (email) COVER (name, status)
);

-- Запрос выполняется только с использованием индекса
SELECT name, status FROM users VIEW idx_email_covering WHERE email = 'user@example.com';
-- Выполняется: только поиск в индексе, lookup не требуется!
  ```
  {% endcut %}

- Go

  {% cut "Плохой пример" %}
  ```go
// Плохой пример: запрос с обычным индексом
// Требует lookup в основную таблицу
result, err := session.Execute(ctx, `
    SELECT name, status FROM users WHERE email = $email
`, table.NewQueryParameters(
    table.ValueParam("$email", types.UTF8Value("user@example.com")),
))
// Выполняется: индекс scan + table lookup
  ```
  {% endcut %}

  {% cut "Хороший пример" %}

  ```go
// Хороший пример: запрос с covering индексом
// Все данные берутся из индекса
result, err := session.Execute(ctx, `
    SELECT name, status FROM users VIEW idx_email_covering
    WHERE email = $email
`, table.NewQueryParameters(
    table.ValueParam("$email", types.UTF8Value("user@example.com")),
))
// Выполняется: только индекс scan, без lookup
  ```
