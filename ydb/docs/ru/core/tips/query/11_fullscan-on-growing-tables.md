# Избежание FullScan на растущих таблицах с помощью индексов

## Проблема

При работе с растущими таблицами в {{ ydb-short-name }} запросы, которые не используют индексы, выполняют полное сканирование таблицы (`TableFullScan`). Это приводит к следующим проблемам:

- **Линейная деградация производительности**: время выполнения запроса растет пропорционально размеру таблицы
- **Высокая нагрузка на систему**: `TableFullScan` потребляет значительные ресурсы CPU и I/O
- **Непредсказуемое время отклика**: производительность запросов становится зависимой от объема данных
- **Ограничение масштабируемости**: при больших объемах данных `TableFullScan` становится невыполнимым

Типичные сценарии, приводящие к полному сканированию:
- Фильтрация по неключевым колонкам
- JOIN операции по неключевым колонкам
- Использование функций над ключевыми колонками

## Решение

Для обеспечения стабильной производительности на растущих таблицах необходимо использовать вторичные индексы, которые ускоряют чтение при использовании в запросах неключевых колонок таблицы

### 1. Создание вторичных индексов

Вторичные индексы позволяют эффективно выполнять запросы по неключевым полям. Рекомендуется создавать индексы для часто используемых в предикатов полей таблицы.

```sql
ALTER TABLE episodes
ADD INDEX title_index GLOBAL SYNC ON (title)
```

### 2. Явное указание индексов в запросах

Оптимизатор обычно выбирает оптимальный индекс для эффективного чтения данных. Но если по каким-то причинам он не может этого сделать, то необходимо явно указывать нужный индекс через конструкцию `VIEW`.

**Пример запроса с явным выбором индекса:**

```sql
DECLARE $customer_id AS Uint64;
SELECT *
FROM orders VIEW idx_customer AS o
WHERE o.id_customer = $customer_id

SELECT season_id, episode_id
FROM episodes VIEW title_index
WHERE title = 'The Work Outing'
```

**Пример явного выбора индекса для оператора JOIN:**

```sql
ALTER TABLE nation
ADD INDEX r_index GLOBAL SYNC ON (n_regionkey);

SELECT *
FROM region r
JOIN nation VIEW r_index n
ON n.n_regionkey = r.r_regionkey;
```

### 3. Оптимизация структуры индексов

По умолчанию индекс работает следующим образом
- Выбирается множество записей (значения первичного ключа), удовлетворяющих данном условию
- Для этих записей происходит запрос необходимых пользователю полей из основной таблицы

Можно оптимизировать этот процесс ещё больше, если включить дополнительные (неиндексируемые) поля непосредственно в индекс. Такой индекс называется **покрывающим**, он позволяет избежать второго чтения из основной таблицы. Список полей, которые необходимо включить в индекс, указываются через запятую в дополнительном выражении `COVER`:

```sql
ALTER TABLE orders
ADD INDEX idx_customer_covering GLOBAL SYNC ON (id_customer)
COVER (order_date, amount, status)
```

Покрывающий индекс позволяет эффективно получать значения следующих полей:
- ключевых полей индекса, указанных в выражении `ON`
- первичного ключа основной таблицы (эти поля включаются всегда, их не надо указывать явно)
- дополнительных полей, указанных в выражении `COVER`

### 4. Порядок полей в индексе

Порядок полей во вторичном индексе имеет то же самое значение, что и в первичном ключе таблицы. Для оптимизации поиска могут быть использован только префикс индекса. Предположим, что у нас индекс содержит три поля `ON (a, b, c)`, тогда:

- ✅ `WHERE a = $v1 AND b = $v2 AND c = $v3` - использует индекс полностью
- ✅ `WHERE a = $v1 AND b = $v2` - использует префикс индекса из полей `a` и `b`
- ✅ `WHERE a = $v1 AND c = $v2` - использует префикс индекса только из поля `a` (поле `c` уже не входит в префикс и не используется для оптимизации)
- ❌ `WHERE b = $v2` - не использует индекс

### 5. Мониторинг и анализ планов запросов

Для анализа структуры запроса используйте {{ ydb-short-name }} CLI - команды `ydb sql --explain` и `ydb sql --explain-analyze` (или аналогичные вызовы в {{ ydb-short-name }} UI)

- ✅ операторы `TableRangeScan`, `TableLookup`, а также `LookupJoin` используют эффективный поиск по индексу
- ❌ оператор `TableFullScan` и другие виды `Join` не используют индексы

### 6. Практические рекомендации

**Используйте индексы для:**
- ✅ частых запросов с фильтрацией по неключевым полям
- ✅ объединения Join операций с большими таблицами
- ✅ запросов с ограничение по диапазону (BETWEEN, >, <)
- ✅ сортировки результатов по неключевым полям
- ✅ не более 5 вторичных индексов на таблицу

**Избегайте избыточных индексов:**
- ❌ на редко используемые поля
- ❌ на поля с низкой селективностью (например, булевы флаги)
- ❌ более 5 индексов на одну таблицу

### 7. Типичные проблемы и их решение

**Ошибка:** Запрос с фильтрацией по индексированному полю все равно оставляет в плане оператор `TableFullScan`
**Решение:** Явно укажите индекс через `VIEW index_name`

**Ошибка:** JOIN операции не используют индексы
**Решение:** Убедитесь, что условия JOIN соответствуют структуре индекса

**Ошибка:** Функции над ключевыми колонками приводят к `TableFullScan`
**Решение:** Избегайте применения функций к индексированным полям

### 8. Мониторинг эффективности

Регулярно проверяйте планы запросов с помощью:
- {{ ydb-short-name }} CLI: `ydb sql --explain` и `ydb sql --explain-analyze`
- {{ ydb-short-name }} UI
- плана запроса доступного в {{ ydb-short-name }} SDK

Ищите узлы `TableFullScan` и заменяйте их на `TableRangeScan` и `TableLookup`, добавляя необходимые индексы

## 9. Заключение

Использование вторичных индексов - ключевой механизм для обеспечения стабильной производительности на растущих таблицах {{ ydb-short-name }}. Правильное проектирование индексов, их явное указание в запросах и регулярный мониторинг позволяют уменьшить нагрузку на сервер {{ ydb-short-name }} и избежать деградации производительности при увеличении объема данных.
