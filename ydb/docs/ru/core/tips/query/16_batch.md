# Оптимальный размер batch для эффективных запросов

## Проблема

При использовании оператора `IN` в YQL-запросах важно контролировать размер передаваемого списка значений. {{ ydb-short-name }} автоматически выбирает стратегию выполнения запроса в зависимости от размера списка:

- **Lookup/MultiLookup** - для небольших списков (обычно до 1000-10000 элементов)
- **FullScan** - для больших списков, когда оптимизатор считает полное сканирование более эффективным

Пример запроса: `SELECT * FROM t WHERE id IN $ids`

**Проблемы при превышении оптимального размера:**

1. **Непредсказуемый переход к FullScan** - при большом количестве элементов {{ ydb-short-name }} может выбрать полное сканирование таблицы вместо точечных чтений
2. **Увеличение времени выполнения** - FullScan требует чтения всех записей таблицы, что значительно медленнее Lookup
3. **Потребление ресурсов** - полное сканирование создает нагрузку на все шарды таблицы
4. **Риск превышения лимитов** - возможны ошибки "Datashard: Reply size limit exceeded" (ограничение 50 МБ)

## Решение

### 1. Определение оптимального размера batch

Рекомендуемые практики:

- **Для запросов с IN**: ограничивать размер списка 1000-5000 элементов
- **Для пакетных операций**: использовать параметры `$ydb_batch_size_rows` (по умолчанию 10000) и `$ydb_batch_size_bytes`
- **Анализ плана запроса**: всегда проверять план выполнения через `EXPLAIN`

### 2. Стратегии разбиения на пакеты

**Адаптивное пакетирование в CLI:**
```bash
ydb table query execute \
  --query "SELECT * FROM t WHERE id IN $ids" \
  --batch adaptive \
  --batch-limit 1000 \
  --batch-max-delay 1s
```

**Программное разбиение:**
```python
def batch_select(ids, batch_size=1000):
    results = []
    for i in range(0, len(ids), batch_size):
        batch = ids[i:i + batch_size]
        # Выполнить запрос для batch
        results.extend(execute_query(batch))
    return results
```

### 3. Использование вторичных индексов

Для часто используемых фильтров создавайте вторичные индексы:
```sql
ALTER TABLE t ADD INDEX idx_id GLOBAL ON (id);
```

И используйте их в запросах:
```sql
SELECT * FROM t VIEW idx_id WHERE id IN $ids;
```

### 4. Мониторинг и оптимизация

- **Анализ планов запросов**: используйте `EXPLAIN` для проверки стратегии выполнения
- **Мониторинг производительности**: отслеживайте метрики QueryResultBytes, YdbResponses
- **Настройка автосплита**: для больших таблиц настройте автоматическое разделение партиций

## Рекомендации

1. **Тестируйте на реальных данных** - оптимальный размер batch зависит от структуры данных и нагрузки
2. **Используйте параметризованные запросы** - для избежания SQL-инъекций и лучшего кэширования планов
3. **Проверяйте наличие индексов** - убедитесь, что колонки в условии IN индексированы
4. **Избегайте FullScan в OLTP-сценариях** - для транзакционных запросов предпочтительны Lookup операции

Помните: оптимальный размер batch - это баланс между количеством round-trip'ов и эффективностью отдельных запросов.
