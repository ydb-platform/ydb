# `INSERT` vs `UPSERT` - выбор оптимальной операции вставки

## Основные различия

**INSERT** - операция вставки с проверкой уникальности по PRIMARY KEY. Если запись с таким ключом уже существует, операция завершится ошибкой. Требует обязательного чтения данных для проверки существования записи.

**UPSERT** - операция "вставки или обновления" (upsert). Если запись с таким ключом существует - обновляет её, если не существует - вставляет новую. При отсутствии вторичных индексов на таблице UPSERT работает как "слепая" операция без чтения данных.

## Проблема выбора

Выбор между `INSERT` и `UPSERT` критически влияет на производительность приложения, особенно при высокой нагрузке. Неправильный выбор может привести к:
- Избыточным операциям чтения
- Снижению пропускной способности

## Рекомендации по выбору

### Когда использовать UPSERT:

1. **Высоконагруженные сценарии без строгих требований к уникальности**
2. **Пакетная обработка данных**, где дубликаты допустимы
3. **Таблицы без синхронных вторичных индексов**
4. **Сценарии "последняя запись побеждает"**

### Когда использовать INSERT:

1. **Критически важные данные**, где дубликаты недопустимы
2. **Таблицы с синхронными вторичными индексами**, особенно если вторичных индексов несколько
3. **Сценарии с строгими требованиями к целостности данных**
4. **Когда приложение может корректно обработать ошибку дубликата**

## Влияние вторичных индексов

**С синхронными индексами**: `INSERT` обычно быстрее, так как UPSERT требует дополнительных операций для обновления индексов при замене существующих записей.

**Без индексов**: `UPSERT` значительно быстрее благодаря "слепой" записи без предварительного чтения.

## Примеры кода


### ❌ Плохой пример - неоправданное использование UPSERT с индексами

```sql
-- Медленно на таблице с синхронными индексами
UPSERT INTO users (id, name, email) VALUES (1, 'Иван', 'ivan@example.com');
```

### ✅ Хороший пример - правильное использование INSERT с обработкой ошибок

```sql
-- Быстро на таблице с индексами, с обработкой дубликатов
DECLARE $id AS Uint64;
DECLARE $name AS Utf8;
DECLARE $email AS Utf8;

INSERT INTO users (id, name, email) VALUES ($id, $name, $email);
-- В приложении обрабатываем ошибку "уже существует"
```

### ✅ Хороший пример - UPSERT для таблиц без индексов

```sql
-- Оптимально для таблиц без вторичных индексов
UPSERT INTO analytics_events (event_id, timestamp, data)
VALUES ($event_id, $timestamp, $data);
```

### ✅ Хороший пример - пакетный UPSERT для массовых операций

```sql
-- Эффективно для пакетной вставки/обновления
DECLARE $data AS List<Struct<product_id:Uint64, price:Decimal(22,9)>>;
UPSERT INTO product_prices (product_id, price, updated_at)
SELECT product_id, price, CurrentUtcTimestamp()
FROM AS_TABLE($data);
```


## Производительность

- **UPSERT без индексов**: до 2-3x быстрее INSERT
- **INSERT с индексами**: до 30-50% быстрее UPSERT
- **Пакетные операции**: всегда предпочтительнее одиночных

## Заключение

Выбор между INSERT и UPSERT зависит от конкретного сценария использования, наличия индексов и требований к целостности данных. Для максимальной производительности:
- Используйте UPSERT для таблиц без индексов и массовых операций
- Предпочитайте INSERT для таблиц с синхронными индексами
- Всегда используйте пакетные операции при работе с большими объемами данных
