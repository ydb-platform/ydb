# Prepared statements vs параметризованные запросы

## Проблема

В {{ ydb-short-name }} для кэширования скомпилированных запросов исторически использовали **Prepared statements** (подготовленные запросы): `Prepare` компилирует запрос и возвращает `query_id`, затем `Execute` по этому `query_id` выполняется без повторной компиляции. В распределённой СУБД это создаёт ряд проблем:

- **Привязка к сессии** — подготовленный запрос живёт только пока жива сессия. При серверной балансировке или обслуживании сессия может быть закрыта, и тогда запрос с неизвестным `query_id` вернёт ошибку (например, `NotFound`). Клиенту нужно по кодам ошибок понимать, когда инвалидировать свой statement и переподготавливать запрос на другой сессии.
- **Отказ от клиентской балансировки** — все вызовы `Execute` по одному statement идут на один и тот же узел {{ ydb-short-name }}. Нагрузка перестаёт равномерно распределяться по кластеру.
- **Риск при ретраях** — если statement хранится вне ретраера и после одной из попыток сессия была инвалидирована, последующие попытки продолжают использовать уже невалидный statement. Либо вызывать `Prepare` внутри ретраера — тогда при каждой попытке уходит лишний запрос на сервер.

В итоге требуется аккуратный микро-менеджмент жизненного цикла подготовленных запросов, что усложняет код и увеличивает риск ошибок.

## Решение

Вместо **Prepare + Execute** используйте **параметризованный запрос и один вызов Execute с флагом KeepInCache**. Запрос по-прежнему компилируется один раз, но результат компиляции кэшируется **на узле {{ ydb-short-name }}** (в LRU-кэше узла, по умолчанию до 1000 записей), а не привязывается к сессии. Первый `Execute` с этим текстом запроса на узле приводит к компиляции, последующие — к повторному использованию кэша.

**Плюсы подхода:**

- Сохраняется клиентская балансировка: запросы распределяются по узлам, на каждом узле кэш заполняется при первом выполнении запроса с этим текстом.
- Не нужно отслеживать инвалидацию statement при закрытии сессии — жизненный цикл кэша на стороне {{ ydb-short-name }}.
- Один вызов вместо двух (не нужен отдельный `Prepare`).

Во многих SDK флаг **KeepInCache** для параметризованных запросов включён по умолчанию (например, в Go — при передаче хотя бы одного параметра). При необходимости кэширование можно отключить опцией (например, `WithKeepInCache(false)` в Go, `KeepInQueryCache = false` в C#).

**Важно:** при большом числе уникальных текстов запросов (например, при интерполяции параметров в строку запроса) кэш на узле может «вымываться». Используйте именно параметризованные запросы (параметры передаются отдельно), а не подстановку значений в текст.
