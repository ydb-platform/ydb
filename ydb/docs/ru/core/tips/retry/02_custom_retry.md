# Готовые ретраеры из официальных SDK

{{ ydb-short-name }} может возвращать на запросы разнообразные ошибки.

При работе с {{ ydb-short-name }} важно правильно обрабатывать ошибки и выбирать стратегию повторов. Ошибки можно разделить на три категории:

- **Временные сбои (retryable)** — можно ретраить безопасно с соответствующей стратегией задержек
- **Условно повторяемые ошибки (conditionally retryable)** — можно ретраить, если запрос идемпотентный
- **Неповторяемые ошибки (non-retryable)** — не имеет смысла ретраить вообще (от повтора запроса успех не случится)

Для разных типов ошибок требуются разные стратегии:
- Одни ошибки нужно ретраить с короткой экспоненциальной задержкой (несколько миллисекунд)
- Другие ошибки нужно ретраить с длинной экспоненциальной задержкой (несколько секунд)
- На некоторые ошибки следует удалить сессию и создать новую

Официальная классификация статусов: [{#T}](../../reference/ydb-sdk/ydb-status-codes.md), транспортный слой: [{#T}](../../reference/ydb-sdk/grpc-status-codes.md).

## Проблема

Создание собственных (самодельных) ретраеров — это сложная задача, требующая глубокого понимания:
- Типов ошибок {{ ydb-short-name }} и их статусов завершения (коды в диапазонах 400000–400999 для серверных ошибок, 401000–401999 для SDK-специфичных ошибок, а также gRPC статусы; см. ссылки выше)
- Правильного выбора стратегии задержек для каждого типа ошибки
- Обработки идемпотентных и неидемпотентных операций
- Предотвращения "retry storm" — ситуации, когда повторные запросы перегружают отказавший сервис
- Корректной обработки сессий и их пересоздания при необходимости

Самодельные ретраеры часто:
- Не учитывают все нюансы обработки ошибок
- Создают busy loop (бесконечные циклы повторов)
- Неправильно выбирают стратегию задержек
- Не обрабатывают граничные случаи
- Не учитывают негативный опыт эксплуатации, накопленный командой {{ ydb-short-name }}

## Решение

Используйте ретраеры из состава официальных SDK {{ ydb-short-name }}. Официальные SDK предоставляют проверенные ретраеры с гарантией:
- Корректной обработки всех типов ошибок
- Отсутствия busy loop
- Правильного выбора стратегии задержек
- Оптимизации числа повторных попыток и интервала между ними
- Учета всех граничных случаев и особенностей {{ ydb-short-name }}

Официальные ретраеры используют рекомендованную политику повторов, которую можно настроить в соответствии с требованиями вашего приложения, но при этом они гарантируют корректное поведение в стандартных ситуациях.

// Плохой пример: самодельный ретраер без учета типов ошибок

        err := fn()
        if err == nil {
            return nil
        }
        time.Sleep(100 * time.Millisecond) // Фиксированная задержка

    return errors.New("max retries exceeded")
}
// Проблемы:
