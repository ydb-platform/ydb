# Временные объекты в лямбдах ретраера

Контекст: повторные попытки и ретраеры — [{#T}](../../recipes/ydb-sdk/retry.md), коды ошибок — [{#T}](../../reference/ydb-sdk/ydb-status-codes.md).

## Проблема

Безопасное использование временных объектов в лямбдах ретраера требует понимания их времени жизни: временным объектам не гарантировано время жизни за пределами лямбды. Поэтому пытаться вынести через замыкания временные объекты из лямбды ретраера может привести к неожиданным и необработанным ошибкам.

Ретраеры {{ ydb-short-name }} SDK могут выполнять лямбда-функцию несколько раз при возникновении ретраибельных ошибок. Если временный объект (например, результат запроса) выносится через замыкание за пределы лямбды, при повторном выполнении лямбды этот объект может оказаться невалидным или содержать данные из предыдущего запуска.

## Пример плохого кода

### Go (плохой пример)
```go
var res result.Result
err := db.Table().Do(ctx, func(ctx, s) (err error) {
    res, err = s.Execute(ctx, `SELECT * FROM bigTable;`)
    return err
})
for res.NextResultSet(ctx) {
    for res.Next() {
        if err = res.Scan(&col1, &col2, &col3); err != nil {
            // process error
        }
    }
}
```

**Проблема:** Результат запроса `res` выносится через замыкание за пределы лямбды. При повторном выполнении лямбды ретраером, переменная `res` может содержать невалидные данные или данные из предыдущего неудачного выполнения.

### Go (еще один плохой пример)
```go
var myObjs []MyObject
err := db.Table().Do(ctx, func(ctx, s) (err error) {
    res, err = s.Execute(ctx, `SELECT * FROM bigTable;`)
    for res.NextResultSet(ctx) {
        for res.Next() {
            var myObj MyObject
            if err = res.Scan(&myObj.col1, &myObj.col2, &myObj.col3); err != nil {
                return err
            }
            myObjs = append(myObjs, myObj)
        }
    }
    return err
})
```

**Проблема:** Срез `myObjs` накапливается через замыкание. При повторном выполнении лямбды данные будут добавляться повторно, что приведет к дублированию.

## Решение

Все операции с временными объектами должны выполняться внутри лямбды ретраера. Не следует выносить временные объекты через замыкание за пределы лямбды.

### Go (хороший пример)
```go
var myObjs []MyObject
err := db.Table().Do(ctx, func(ctx, s) (err error) {
    // Очищаем срез при каждом выполнении лямбды
    myObjs = nil

    res, err := s.Execute(ctx, `SELECT * FROM bigTable;`)
    if err != nil {
        return err
    }

    var tempObjs []MyObject
    for res.NextResultSet(ctx) {
        for res.Next() {
            var myObj MyObject
            if err = res.Scan(&myObj.col1, &myObj.col2, &myObj.col3); err != nil {
                return err
            }
            tempObjs = append(tempObjs, myObj)
        }
    }

    // Присваиваем результат только после успешного выполнения
    myObjs = tempObjs
    return nil
})
```

## Рекомендации

1. **Очищайте временные объекты** в начале лямбды, чтобы избежать накопления данных при повторных выполнениях
2. **Не выносите временные объекты** через замыкание - работайте с ними только внутри лямбды
3. **Используйте локальные переменные** для промежуточных результатов внутри лямбды
4. **Присваивайте финальный результат** только после успешного выполнения всех операций внутри лямбды

## Почему это важно

- **Корректность данных**: предотвращает дублирование или потерю данных при повторных выполнениях
- **Стабильность приложения**: избегает использования невалидных объектов после повторных попыток
- **Производительность**: предотвращает утечки памяти и неэффективное использование ресурсов
