# Ретраеры

Советы по правильной обработке ошибок и повторных попыток при работе с {{ ydb-short-name }}

Связанные материалы: [{#T}](../../recipes/ydb-sdk/retry.md), [{#T}](../../reference/ydb-sdk/ydb-status-codes.md), [{#T}](../../reference/ydb-sdk/grpc-status-codes.md).

## Оптимистические блокировки в {{ ydb-short-name }}

{{ ydb-short-name }} использует оптимистичные блокировки: конфликты проверяются при коммите, а не во время работы с данными. Поэтому **любой код работы с {{ ydb-short-name }} должен быть в ретраерах** — это не опция, а требование архитектуры.

## Классификация ошибок

{{ ydb-short-name }} может возвращать на запросы разнообразные ошибки. 

Любая ошибка помимо семантической составляющей позволяет подобрать оптимальную реакцию на нее:
- некоторые ошибки можно ретраить безопасно,
- часть ошибок можно ретраить, если запрос идемпотентный,
- отдельные ошибки не имеет смысла ретраить вообще (от повтора запроса успех не случится),
- одни ошибки нужно ретраить с короткой экспоненциальной задержкой,
- другие ошибки нужно ретраить с длинной экспоненциальной задержкой,
- на некоторые ошибки следует удалить сессию.

Подробнее о классификации статусов — в [{#T}](../../reference/ydb-sdk/ydb-status-codes.md).
