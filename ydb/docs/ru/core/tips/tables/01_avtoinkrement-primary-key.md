# Монотонно возрастающий `PRIMARY KEY`

## Проблема

В {{ ydb-short-name }} строковые таблицы всегда партиционированы по диапазонам значений первичного ключа. Каждая партиция (шард) таблицы отвечает за свой диапазон первичных ключей. При росте объёма данных или нагрузки {{ ydb-short-name }} автоматически разделяет партиции.

Если `PRIMARY KEY` основан на монотонно возрастающем счётчике (например, автоинкрементном `ID` или последовательно генерируемых идентификаторах), возникает проблема «горячих точек»:

- Новые записи последовательно попадают в одну и ту же (последнюю) партицию, так как их ключи монотонно возрастают
- Это приводит к неравномерному распределению данных и перегрузке отдельных партиций при высокой нагрузке
- Одна партиция обрабатывает большую часть операций записи, в то время как другие партиции простаивают
- В результате снижается общая производительность и масштабируемость системы
- При достижении лимита размера партиции (по умолчанию 2000 MB) происходит разделение, но новые записи продолжают попадать в последнюю партицию

При этом важно учесть что в случае составного `PRIMARY KEY`, использование монотонно возрастающего значения в качестве его первой компоненты приведет к схожей проблеме, т.к. сортировка выполняется по всему `PRIMARY KEY`, и его первый компонент имеет в этом случае определяющее значение.

**Важно:** {{ ydb-short-name }} поддерживает автоинкремент через серийные типы данных (`SERIAL`), но при использовании их в качестве единственного или первого компонента `PRIMARY KEY` возникает описанная проблема.

## Решение
Рекомендуется воздержаться от использования монотонно возрастающих значений в качестве единственной или первой компоненты `PRIMARY KEY`, вместо этого:
- Используйте UUID или другие случайные идентификаторы в составе первичного ключа
- Вместо искусственных идентификаторов используйте составной ключ согласно вашей бизнес-логике, если он не попадает под условие монотонности

Если вам все же необходимо использовать монотонно возрастающие значения в качестве идентификатора, то чтобы равномерно распределить нагрузку по партициям, используйте составной `PRIMARY KEY`, где:
- **Первая компонента** — хеш от значения счетчика (обеспечивает равномерное распределение записей по партициям)
- **Вторая компонента** — монотонно возрастающий счётчик (гарантирует уникальность записей)

Такой подход позволяет:
- Сбалансировать нагрузку между партициями за счет равномерного распределения хеша
- Эффективно масштабировать таблицу при росте данных
- Избежать «горячих точек» и перегрузки отдельных партиций


{% cut "Плохой пример" %}
```yql
CREATE TABLE orders (
    order_id Uint64,
    customer_id Uint64,
    amount Double,
    PRIMARY KEY (order_id)
);

INSERT INTO orders (order_id, customer_id, amount) VALUES 
    (1001, 10, 999.99),
    (1002, 10, 999.99),
    (1003, 10, 999.99);

SELECT * FROM orders
WHERE order_id = 1001;
```
{% endcut %}

{% cut "Хороший пример" %}
```yql
CREATE TABLE orders (
    hash Uint64,
    order_id Uint64,
    customer_id Uint64,
    amount Double,
    PRIMARY KEY (hash, order_id)
);

INSERT INTO orders (hash, order_id, customer_id, amount) VALUES 
    (Digest::NumericHash(1001), 1001, 10, 999.99),
    (Digest::NumericHash(1002), 1002, 10, 999.99),
    (Digest::NumericHash(1003), 1003, 10, 999.99);

SELECT * FROM orders
WHERE hash = Digest::NumericHash(1001) AND order_id = 1001;
```
{% endcut %}
