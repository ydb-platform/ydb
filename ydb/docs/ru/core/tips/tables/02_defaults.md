# Настройки партицирования таблиц по умолчанию

## Проблема

По умолчанию в {{ ydb-short-name }} включено автоматическое партицирование таблиц со следующими настройками (указанные настройки могут быть изменены в будущем):
- включено партицирование по размеру с размер партиции по умолчанию `2ГБ`,
- минимальное количество партиций на таблицу - `1`,
- максимальное количество партиций на таблицу - `50`,
- партицирование по нагрузке выключено.

Если дополнительно не настраивать партицирование таблиц, то могут возникнуть следующие проблемы:
1. **Перегрузка шардов**. Если размер данных в шарде превышает допустимые значения, производительность шарда может упасть, что приведет к увеличению latency и ухудшению общей картины с нагрузкой на базу данных;
2. **Ухудшение производительности**. Один перегруженный шард будет обрабатывать больше запросов, чем другие, что может привести к снижению общей пропускной способности системы. Неэффективное распределение нагрузки между шардами может вызвать деградацию производительности.
3. **Несбалансированное распределение данных**. Отсутствие автоматического шардирования может привести к тому, что данные будут сосредоточены в небольшом числе шардов, а другие останутся незагруженными. Это приведет к тому, что запросы к перегруженным шардам будут обрабатываться дольше, а к менее загруженным — быстрее 
4. **Проблемы с горячими ключами**. Если первичный ключ таблицы является монотонно возрастающим, данные будут скапливаться в одном шарде, что приведет к его перегрузке. Это особенно критично, когда таблица активно используется. Например, если ключ содержит постоянно меняющийся идентификатор, и его разбиение невозможно задать изначально, отсутствие автосплита по нагрузке может вызвать проблемы 
5. **Невозможность масштабирования**. Без автосплита таблица будет иметь фиксированное количество партиций, и при увеличении объема данных не будет автоматически увеличиваться количество партиций. Это ограничит масштабируемость базы данных.
6. **Риск образования "гигантов"**. Если не настроить минимальное и максимальное количество партиций, может возникнуть ситуация, когда шард станет слишком большим («гигант»), что затруднит его обслуживание и может вызвать проблемы с производительностью.
8. **Частные операции split/merge партиций таблицы**, приводящие к фону TLI и необходимости повторов операций с таблицей.

## Рекомендации

1. Установить минимальное количество партиций таблицы - для исключения объединения партиций


{% cut "Плохой пример" %}
```yql
-- Плохой пример: создание таблицы с настройками по умолчанию
-- Может привести к проблемам при высокой нагрузке
CREATE TABLE orders (
    order_id Uint64,
    customer_id Uint64,
    amount Double,
    created_at Timestamp,
    PRIMARY KEY (order_id)
);
-- Проблемы:
-- - Партицирование по нагрузке выключено
-- - Минимальное количество партиций = 1 (может объединиться)
-- - Максимальное количество партиций = 50 (может ограничить масштабирование)
-- - При монотонно возрастающем order_id все данные попадут в одну партицию
```
{% endcut %}

{% cut "Хороший пример" %}
```yql
-- Альтернативный хороший пример: настройка для таблицы с умеренной нагрузкой
ALTER TABLE orders SET (
    AUTO_PARTITIONING_BY_LOAD = ENABLED,
    AUTO_PARTITIONING_MIN_PARTITIONS_COUNT = 2,
    AUTO_PARTITIONING_MAX_PARTITIONS_COUNT = 20
);
-- Для таблиц с умеренной нагрузкой можно использовать меньшие значения
```
{% endcut %}
