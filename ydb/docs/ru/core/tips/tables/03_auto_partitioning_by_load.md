# Автоматическое партицирование таблиц по нагрузке

## Проблема

В {{ ydb-short-name }} каждая партиция таблицы может находиться только на одном сервере и использовать не более 1 ядра CPU для операций изменения данных. Когда нагрузка на партицию превышает возможности одного ядра (CPU > 50-80%), добавление дополнительных CPU не поможет - производительность упирается в однопоточность шарда.

**Типичные симптомы перегруженной партиции:**
- Высокий CPU на отдельных партициях при низкой общей загрузке кластера
- Задержки выполнения запросов к "горячим" данным
- Ошибки `STATUS_OVERLOADED` при записи в перегруженные партиции

## Решение

Автоматическое партицирование по нагрузке (`AUTO_PARTITIONING_BY_LOAD`) позволяет {{ ydb-short-name }} динамически разделять перегруженные партиции и объединять малонагруженные.

### Принцип работы

**Разделение партиции (split):**
- Если партиция потребляет >50% CPU в течение нескольких десятков секунд
- {{ ydb-short-name }} собирает сэмпл запрашиваемых ключей и выбирает точку разделения для равномерного распределения нагрузки
- Новые партиции могут иметь существенно отличающийся размер

**Объединение партиций (merge):**
- Если суммарная нагрузка на 2+ соседних партиций <35% одного ядра CPU в течение часа
- {{ ydb-short-name }} объединяет малонагруженные партиции для оптимизации ресурсов

### Рекомендации по настройке

1. **Минимальное количество партиций:** Устанавливайте `AUTO_PARTITIONING_MIN_PARTITIONS_COUNT` отличным от 1, чтобы избежать частых разделений при плавающей нагрузке
2. **Ориентир на ресурсы:** Минимальное количество партиций должно быть не менее количества узлов, а лучше - порядка количества ядер CPU
3. **Время операции:** Разделение/объединение занимает ~500 мс, в течение которых данные временно недоступны

## Примеры кода

### Создание таблицы с автопартицированием по нагрузке

```sql
-- Хороший пример: таблица с автопартицированием и минимальным количеством партиций
CREATE TABLE events (
    user_id Uint64,
    event_time Timestamp,
    event_type String,
    payload Json,
    PRIMARY KEY (user_id, event_time)
)
WITH (
    AUTO_PARTITIONING_BY_LOAD = ENABLED,
    AUTO_PARTITIONING_MIN_PARTITIONS_COUNT = 4,
    AUTO_PARTITIONING_MAX_PARTITIONS_COUNT = 100
);
```

```sql
-- Плохой пример: таблица без автопартицирования для высоконагруженных данных
CREATE TABLE hot_data (
    id Uint64,
    data String,
    PRIMARY KEY (id)
)
WITH (
    AUTO_PARTITIONING_BY_LOAD = DISABLED  -- Ошибка! Для горячих данных нужно включить
);
```

### Изменение настроек существующей таблицы

```sql
-- Включение автопартицирования для существующей таблицы
ALTER TABLE events SET (
    AUTO_PARTITIONING_BY_LOAD = ENABLED,
    AUTO_PARTITIONING_MIN_PARTITIONS_COUNT = 8
);
```

### Настройка автопартицирования для индексов

```sql
-- Автопартицирование для вторичного индекса
ALTER TABLE events ALTER INDEX event_type_index SET (
    AUTO_PARTITIONING_BY_LOAD = ENABLED,
    AUTO_PARTITIONING_MIN_PARTITIONS_COUNT = 4
);
```

### Terraform конфигурация

```hcl
resource "yandex_ydb_table" "events" {
  # ... другие параметры
  
  partitioning_settings = {
    auto_partitioning_by_load            = true
    auto_partitioning_min_partitions_count = 4
    auto_partitioning_max_partitions_count = 50
    auto_partitioning_partition_size_mb  = 2048
  }
}
```

## Практические рекомендации

1. **Для таблиц с высокой нагрузкой** всегда включайте `AUTO_PARTITIONING_BY_LOAD`
2. **Устанавливайте разумные лимиты** на минимальное и максимальное количество партиций
3. **Мониторьте метрики** CPU по партициям через {{ ydb-short-name }} Monitoring
4. **Избегайте "горячих ключей"** - равномерное распределение данных улучшает эффективность партицирования
5. **Для индексов** также настраивайте автопартицирование, если они подвержены высокой нагрузке

## Когда не использовать

- Таблицы с очень маленьким объемом данных (< 1GB)
- Таблицы с равномерно низкой нагрузкой
- Временные таблицы с коротким временем жизни


```sql
-- Пример плохого дизайна: все данные попадают в одну партицию
CREATE TABLE bad_design (
    status String,  -- Всегда "active"
    id Uint64,
    PRIMARY KEY (status, id)  -- status имеет низкую кардинальность
)
WITH (
    AUTO_PARTITIONING_BY_LOAD = ENABLED
);

-- Хороший дизайн: равномерное распределение по ключу
CREATE TABLE good_design (
    user_id Uint64,
    event_time Timestamp,
    status String,
    PRIMARY KEY (user_id, event_time)  -- Высокая кардинальность
)
WITH (
    AUTO_PARTITIONING_BY_LOAD = ENABLED,
    AUTO_PARTITIONING_MIN_PARTITIONS_COUNT = 4
);
```
