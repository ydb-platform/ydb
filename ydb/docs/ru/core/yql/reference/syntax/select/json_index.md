# Чтение по JSON-индексу

[JSON-индекс](../../../../dev/json-indexes.md) ускоряет запросы `SELECT`, которые ищут строки по содержимому JSON-документов с помощью функций [JSON_EXISTS](../../builtins/json.md#json_exists) и [JSON_VALUE](../../builtins/json.md#json_value). JSON-индекс можно использовать **двумя способами**: положиться на автоматический выбор индекса оптимизатором или указать индекс явно через `VIEW`.

## Автоматический выбор {#auto}

В типовом случае достаточно написать обычное условие `WHERE` с `JSON_EXISTS` / `JSON_VALUE` по индексированной JSON-колонке — [оптимизатор](../../../../concepts/glossary.md#optimizer) сам распознает такой предикат и заменит его на чтение по индексу. Новые функции и секция `VIEW` не требуются:

```yql
-- Существование пути
SELECT * FROM t WHERE JSON_EXISTS(v, '$.abc.def');

-- Значение в пути
SELECT * FROM t WHERE JSON_VALUE(v, '$.abc' RETURNING Int32) == 10;
```

### Правила автоматического выбора {#auto-rules}

* JSON-индекс рассматривается оптимизатором с **наименьшим приоритетом**: это запасной вариант, который выбирается только после того, как остальные способы доступа исключены.
* JSON-индекс **не** выбирается, если запрос уже может быть обслужен [первичным ключом](../../../../concepts/glossary.md#primary-index) или другим, более специфичным [вторичным индексом](../../../../concepts/glossary.md#secondary-index).
* Явное указание `VIEW` **переопределяет** решение оптимизатора: имя индекса в `VIEW` принудительно включает его использование (см. раздел [Явное указание индекса](#view)). Это соответствует тому, как `VIEW` имеет приоритет над оптимизатором для [обычных вторичных индексов](../../../../dev/secondary-indexes.md#use).
* Все индексируемые подвыражения в одном запросе должны ссылаться на **одну и ту же** индексированную JSON-колонку. Если соединить через `AND` / `OR` предикаты по разным индексированным колонкам, автоматический выбор не применится.

{% note info %}

Автоматический выбор JSON-индекса является правило-ориентированным (rule-based) и использует только текст запроса. Логика выбора может измениться в будущих версиях {{ ydb-short-name }}.

{% endnote %}

### Откат при автоматическом выборе {#auto-fallback}

Если оптимизатор не смог обработать предикат как индексный, запрос просто выполняется обычным способом — без ускорения индексом. Результат при этом остаётся **корректным**: запрос отрабатывает как полное сканирование (или по другому доступному плану), без ошибки.

## Явное указание индекса (VIEW) {#view}

JSON-индекс, как и любой другой вторичный индекс, можно задействовать принудительно, указав его имя в секции `VIEW`:

```yql
SELECT *
FROM t VIEW json_idx
WHERE JSON_EXISTS(v, '$.abc.def');
```

**Ключевое отличие от автоматического выбора:** если предикат под `VIEW` не получается обработать через индекс, запрос завершается **ошибкой** (пользователь явно потребовал индекс, но индекс не может обслужить такой предикат).

## Поддерживаемые предикаты {#supported}

Ниже перечислены формы предикатов, которые JSON-индекс умеет ускорять.

### Существование пути {#supported-exists}

`JSON_EXISTS` проверяет, что путь присутствует в документе. Массивы при этом прозрачны: индекс одинаково обрабатывает обращение к массиву и к его элементам.

```yql
-- JSON-значение не равно NULL
SELECT * FROM t WHERE JSON_EXISTS(v, '$');

-- Существование пути
SELECT * FROM t WHERE JSON_EXISTS(v, '$.abc.def');
SELECT * FROM t WHERE JSON_EXISTS(v, '$[*].abc');
SELECT * FROM t WHERE JSON_EXISTS(v, '$.abc[0 to last]');
```

### Значение в пути {#supported-value}

Проверку равенства можно записать тремя равнозначными способами — через `JSON_VALUE` с `RETURNING`, через фильтр-предикат `? (...)` внутри `JSON_EXISTS` или через `JSON_VALUE` с предикатом, возвращающим `Bool`:

```yql
-- Численный литерал
SELECT * FROM t WHERE JSON_VALUE(v, '$.abc' RETURNING Int32) == 10;
SELECT * FROM t WHERE JSON_EXISTS(v, '$.abc ? (@ == 10)');
SELECT * FROM t WHERE JSON_VALUE(v, '$.abc == 10' RETURNING Bool);

-- Строчный литерал
SELECT * FROM t WHERE JSON_VALUE(v, '$.abc' RETURNING Utf8) == "abc"u;
SELECT * FROM t WHERE JSON_EXISTS(v, '$.abc ? (@ == "abc")');

-- Булевый литерал (true) и нулевой литерал (null)
SELECT * FROM t WHERE JSON_EXISTS(v, '$.abc ? (@ == true)');
SELECT * FROM t WHERE JSON_EXISTS(v, '$.abc ? (@ == null)');
```

Для сравнения по равенству поддерживаются литералы типов: строка, число, логическое значение и `null` (с оговоркой про `false` и `IS NULL` — см. [Неподдерживаемые предикаты](#unsupported)).

### Списки, диапазоны и параметры {#supported-in}

```yql
-- IN со списком литералов (объединяется как OR)
SELECT * FROM t WHERE JSON_VALUE(v, '$.abc' RETURNING Int32) IN (1, 2, 3);

-- BETWEEN раскрывается в >= AND <= (каждое сравнение даёт проверку существования пути)
SELECT * FROM t WHERE JSON_VALUE(v, '$.abc' RETURNING Int32) BETWEEN 10 AND 20;

-- Параметры
SELECT * FROM t WHERE JSON_VALUE(v, '$.abc' RETURNING Int32) == $paramValue;
SELECT * FROM t WHERE JSON_VALUE(v, '$.abc' RETURNING Int32) IN $paramList;

-- Переменные JsonPath через PASSING
SELECT * FROM t WHERE JSON_EXISTS(v, '$.abc ? (@ == $var)' PASSING 10 AS var);
SELECT * FROM t WHERE JSON_EXISTS(v, '$.abc ? (@ == $var)' PASSING $paramValue AS var);
```

В `IN $paramList` параметром может быть коллекция скалярных значений: `List<T>`, `Tuple<T, ...>`, `Dict<K, V>` или `Set<T>`.

### Объединение предикатов через AND / OR {#supported-and-or}

Несколько проверок можно объединять как внутри JsonPath, так и на уровне YQL:

```yql
-- Внутри JsonPath
SELECT * FROM t WHERE JSON_EXISTS(v, '$.abc ? (@ == 10 || @ == 12)');
SELECT * FROM t WHERE JSON_EXISTS(v, '$.abc ? (@ >= 10 && @ <= 12)');

-- На уровне YQL
SELECT * FROM t WHERE JSON_EXISTS(v, '$.abc ? (@ == 10)') OR JSON_EXISTS(v, '$.abc ? (@ == 12)');
SELECT * FROM t WHERE JSON_EXISTS(v, '$.abc') AND JSON_VALUE(v, '$.abc' RETURNING Int32) == 10;
```

Когда в одном запросе смешиваются `AND` и `OR`, индекс ведёт себя **консервативно**: при необходимости он использует более широкую выборку, чтобы не потерять ни одной подходящей строки, а лишние строки отсеиваются после чтения.

## Неподдерживаемые предикаты {#unsupported}

Перечисленные ниже формы индексом **не ускоряются**. При автоматическом выборе они либо молча отбрасываются (см. [Распознавание подвыражений](#recognition)), либо запрос выполняется без индекса; при явном `VIEW` неприменимый предикат приводит к ошибке.

### Отрицание {#unsupported-negation}

Отрицание нельзя использовать как индексный предикат ни в каком виде:

```yql
SELECT * FROM t WHERE NOT JSON_EXISTS(v, '$.abc');
SELECT * FROM t WHERE JSON_VALUE(v, '$.abc' RETURNING Bool) != true;
SELECT * FROM t WHERE JSON_EXISTS(v, '$.abc ? (!(@ == 10))'); -- ошибка JsonPath
```

В JsonPath конструкция `!pred` всегда является ошибкой. На уровне YQL `NOT`, `!= true` и `== false` не распознаются как индексный предикат: в `AND`-контексте такое подвыражение молча игнорируется (остальной предикат всё ещё использует индекс), а в `OR`-контексте вся `OR`-ветка перестаёт быть индексируемой.

### Сравнение по диапазону {#unsupported-range}

Операторы `<`, `<=`, `>`, `>=` и `!=` не дают сравнения по значению — они сводятся к проверке **существования пути**. Это означает, что индекс отберёт все строки, где путь присутствует, а само неравенство будет проверено уже после чтения:

```yql
-- Индекс отберёт строки с существующим путём $.abc
SELECT * FROM t WHERE JSON_VALUE(v, '$.abc' RETURNING Int32) >= 10;
```

### `false` и `IS NULL` {#unsupported-false-null}

Предикаты `JSON_VALUE(...) == false` и `JSON_VALUE(...) IS NULL` запрещены, поскольку они могут «отрицать» существование пути. Например, если путь `$.k` отсутствует, `JSON_VALUE` вернёт `NULL`, и условие `... IS NULL` окажется истинным — то есть запрос попросит все документы, где такого пути **нет**. Вместо этого используйте явную проверку значения внутри JsonPath:

```yql
SELECT * FROM t WHERE JSON_EXISTS(v, '$.k ? (@ == false)');
SELECT * FROM t WHERE JSON_EXISTS(v, '$.k ? (@ == null)');
SELECT * FROM t WHERE JSON_VALUE(v, '$.k == false' RETURNING Bool);
```

### Предикаты на верхнем уровне `JSON_EXISTS` {#unsupported-je-toplevel}

`JSON_EXISTS` принимает только **пути**. Проверка значения, метод или предикат на верхнем уровне `JSON_EXISTS` — это ошибка, потому что `JSON_EXISTS` возвращает `true` для любого непустого результата JsonPath, и отрицательный результат предиката был бы воспринят как «путь существует». Значение нужно проверять внутри фильтра `? (...)` или через `JSON_VALUE`:

```yql
-- Ошибка: предикат на верхнем уровне
SELECT * FROM t WHERE JSON_EXISTS(v, '$.key == 10');
SELECT * FROM t WHERE JSON_EXISTS(v, 'exists($.key)');
SELECT * FROM t WHERE JSON_EXISTS(v, '$.key starts with "a"');

-- Корректно
SELECT * FROM t WHERE JSON_EXISTS(v, '$.key ? (@ == 10)');
SELECT * FROM t WHERE JSON_VALUE(v, '$.key' RETURNING Int32) == 10;
```

## Обязательный `RETURNING` {#returning}

Для сравнения значения через `JSON_VALUE` всегда нужно указывать `RETURNING` с нужным типом. По умолчанию `JSON_VALUE` возвращает тип `Utf8`, что во время выполнения запроса приводит к некорректному сравнению — значение разных типов сравнивается как строка:

```yql
$tmp = Json('["1", 1]');
SELECT JSON_VALUE($tmp, '$[0]') == "1"; -- true: корректно, строка и строка
SELECT JSON_VALUE($tmp, '$[1]') == "1"; -- true: некорректно, число сравнивается со строкой
```

Поэтому для сравнения по значению указывайте подходящий тип, например `JSON_VALUE(v, '$.abc' RETURNING Int32) == 10`.

## Распознавание подвыражений {#recognition}

Оптимизатор знает, какие JSON-колонки индексированы, и для каждого вызова `JSON_EXISTS` / `JSON_VALUE` проверяет, есть ли по указанной колонке JSON-индекс. Если подвыражение не распознано как индексное, оно обрабатывается по правилам объединения:

* в `AND` нераспознанное подвыражение **молча отбрасывается**, а остальная часть предиката всё ещё использует индекс;
* в `OR` нераспознанное подвыражение делает **всю `OR`-ветку** неиндексируемой.

Все индексируемые подвыражения должны ссылаться на одну и ту же индексированную колонку (см. [Правила автоматического выбора](#auto-rules)).
