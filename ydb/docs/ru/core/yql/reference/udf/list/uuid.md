# Uuid

{% note warning "Экспериментальная возможность" %}

Модуль `Uuid` является экспериментальной возможностью. Его API и поведение могут измениться в будущих версиях.

{% endnote %}

Модуль `Uuid` предоставляет генераторы первичных ключей для строковых таблиц. В отличие от [`RandomUuid()`](../../builtins/basic.md#random), которая возвращает равномерно случайный [UUID версии 4](https://datatracker.ietf.org/doc/html/rfc4122#section-4.4), эти функции формируют 128-битные значения с осмысленной раскладкой битов так, чтобы порядок ключей и распределение по партициям соответствовали диапазонному партиционированию {{ ydb-short-name }}.

Все функции возвращают значение типа `Uuid` во внутреннем 16-байтовом представлении {{ ydb-short-name }} (формат Microsoft GUID / mixed-endian). Именно в таком порядке байтов сравниваются первичные ключи. Генераторы, ориентированные на ключи, записывают байты сразу в этом формате, без преобразования через RFC network-byte-order.

При приведении сгенерированного `Uuid` к типу `String` получается каноническое текстовое представление GUID. Для ключевых генераторов (`newChrono`, `newSharded` и их вариантов `Prefix`) видимая строка не отражает, как временная метка и префикс размещены в хранимых байтах. Для генерации и извлечения данных используйте функции ниже, а не разбор строки.

Общие рекомендации по выбору `Uuid` в качестве первичного ключа см. в разделе [UUID в качестве первичного ключа](../../../dev/primary-key/row-oriented.md#uuid-primary-key).

## Ключевые генераторы {#key-friendly}

Эти функции формируют UUID версии 8 (implementation-specific по [RFC 9562](https://datatracker.ietf.org/doc/html/rfc9562)), оптимизированные для сортировки ключей в {{ ydb-short-name }}. Порядок сортировки определяется сравнением хранимых 16 байтов (`memcmp`), а не канонической GUID-строкой.

### `Uuid::newChrono` {#newchrono}

Генерирует ключ, порядок которого следует времени создания с точностью до миллисекунд, а затем случайным битам суффикса. Строки, вставленные близко по времени, попадают в соседние диапазоны ключей. Это улучшает локальность при сканировании по временному диапазону и при обслуживании индексов по сравнению с полностью случайными ключами.

Необязательные аргументы-зависимости работают так же, как у [`RandomUuid()`](../../builtins/basic.md#random): они управляют моментом вычисления функции для каждой строки, а не содержимым генерируемого значения.

* `Uuid::newChrono([T1[, T2, ...]]) -> Uuid`

### `Uuid::newChronoPrefix` {#newchronoprefix}

Та же хронологическая раскладка, что у `newChrono`, но старшие биты ключа (префикс по умолчанию — 10 бит) берутся из первого аргумента, а не выбираются случайно. Используйте для закрепления общего префикса у нескольких строк в одной транзакции или пакете, чтобы они обычно попадали в одну партицию.

Первый аргумент может быть:

* `Uint64` — младшие 10 бит размещаются в поле префикса;
* `Uuid` — верхние 10 бит MSB исходного значения переиспользуются как префикс.

Далее могут следовать необязательные аргументы-зависимости.

* `Uuid::newChronoPrefix(Uint64{Flags:AutoMap}[, T1[, T2, ...]]) -> Uuid`
* `Uuid::newChronoPrefix(Uuid{Flags:AutoMap}[, T1[, T2, ...]]) -> Uuid`

### `Uuid::newSharded` {#newsharded}

Генерирует ключ, балансирующий распределение по партициям и временную локальность. При каждом вызове старшие биты префикса остаются случайными (по умолчанию около 2<sup>10</sup> ≈ 1024 корзин партиций), в следующее битовое поле записывается текущее Unix-время с точностью до секунды, остальные биты заполняются случайностью. Нагрузка на запись распределяется по партициям, а строки, созданные в близкие моменты времени, остаются относительно близко в пространстве ключей.

Необязательные аргументы-зависимости работают так же, как у [`RandomUuid()`](../../builtins/basic.md#random).

* `Uuid::newSharded([T1[, T2, ...]]) -> Uuid`

### `Uuid::newShardedPrefix` {#newshardedprefix}

Та же шардированная раскладка, что у `newSharded`, но префикс фиксируется первым аргументом. Строки с общим префиксом обычно принадлежат одной партиции, что упрощает многстрочные транзакции; встроенная временная метка по-прежнему группирует близкие по времени записи внутри корзины префикса.

Первый аргумент — `Uint64` или `Uuid` (те же правила, что у `newChronoPrefix`). Далее могут следовать необязательные аргументы-зависимости.

* `Uuid::newShardedPrefix(Uint64{Flags:AutoMap}[, T1[, T2, ...]]) -> Uuid`
* `Uuid::newShardedPrefix(Uuid{Flags:AutoMap}[, T1[, T2, ...]]) -> Uuid`

### Выбор между `newChrono` и `newSharded` {#key-friendly-choice}

| Задача | Функция |
| --- | --- |
| Максимальный хронологический порядок в первичном ключе; сканирование по временному диапазону по ключу | `newChrono` |
| Равномерное распределение по партициям при одиночных вставках с некоторой временной локальностью | `newSharded` |
| Несколько строк в одной транзакции должны делить партицию | `newChronoPrefix` или `newShardedPrefix` с одним префиксом |
| Неструктурированные случайные идентификаторы без семантики сортировки | [`RandomUuid()`](../../builtins/basic.md#random) |

## RFC UUID версии 7 {#rfc-v7}

Эти функции генерируют стандартные значения [RFC 9562 UUID версии 7](https://datatracker.ietf.org/doc/html/rfc9562): 48-битная Unix-метка в миллисекундах в старших битах и случайный суффикс. Результат хранится во внутреннем представлении `Uuid` {{ ydb-short-name }}. Используйте их при необходимости совместимости с инструментами RFC v7 или извлечения встроенной временной метки.

Поскольку v7 следует полевой раскладке RFC, его порядок сортировки в ключах {{ ydb-short-name }} хуже соответствует хронологии, чем у `newChrono`. Для производительности строковых таблиц в {{ ydb-short-name }} предпочтительнее `newChrono` или `newSharded`.

### `Uuid::newV7` {#newv7}

Генерирует v7 UUID по текущей временной метке.

* `Uuid::newV7([T1[, T2, ...]]) -> Uuid`

### `Uuid::newV7At` {#newv7at}

Генерирует v7 UUID по явно заданной временной метке. Принимает `Timestamp` или `Timestamp64`.

* `Uuid::newV7At(Timestamp{Flags:AutoMap}[, T1[, T2, ...]]) -> Uuid`
* `Uuid::newV7At(Timestamp64{Flags:AutoMap}[, T1[, T2, ...]]) -> Uuid`

### `Uuid::extractTs` и `Uuid::extractTs64` {#extract-ts}

Извлекают временную метку, встроенную в v7 UUID. Возвращают `NULL`, если аргумент не является v7 (например, ключ от `newChrono` или `newSharded`).

* `Uuid::extractTs(Uuid{Flags:AutoMap}) -> Timestamp?`
* `Uuid::extractTs64(Uuid{Flags:AutoMap}) -> Timestamp64?`

## Примеры {#examples}

Одиночная вставка со шардированными ключами (по умолчанию — случайный префикс на каждую строку):

```yql
INSERT INTO events (id, payload)
SELECT
    Uuid::newSharded(TableRow()) AS id,
    payload
FROM AS_TABLE($rows);
```

Многострочная транзакция с общим префиксом (строки обычно попадают в одну партицию):

```yql
$prefix = Uuid::newShardedPrefix(RandomNumber(1));

INSERT INTO events (id, payload)
SELECT
    Uuid::newShardedPrefix($prefix, TableRow()) AS id,
    payload
FROM AS_TABLE($rows);
```

Хронологический первичный ключ:

```yql
INSERT INTO audit_log (id, message)
VALUES (Uuid::newChrono(), "user signed in");
```

RFC v7 с извлечением временной метки:

```yql
$ts = CurrentUtcTimestamp();
$id = Uuid::newV7At($ts);
SELECT Uuid::extractTs($id) = $ts;  -- true
```
