# Uuid

Модуль `Uuid` предоставляет генераторы первичных ключей для таблиц {{ ydb-short-name }}. В отличие от [`RandomUuid()`](../../builtins/basic.md#random), которая возвращает равномерно случайный [UUID версии 4](https://datatracker.ietf.org/doc/html/rfc4122#section-4.4), эти функции формируют 128-битные значения с осмысленной раскладкой битов так, чтобы порядок ключей и распределение по партициям соответствовали партиционированию {{ ydb-short-name }}.

Все функции возвращают значение типа `Uuid` во внутреннем 16-байтовом представлении {{ ydb-short-name }} (формат Microsoft GUID / mixed-endian). Именно в таком порядке байтов сравниваются первичные ключи. Генераторы, ориентированные на ключи, записывают байты сразу в этом формате вместо формата RFC network-byte-order.

При приведении сгенерированного `Uuid` к типу `Text` получается каноническое текстовое представление GUID. Для ключевых генераторов (`newChrono`, `newSharded` и их вариантов `Prefix`) получаемая в результате такого преобразования строка не отражает, как именно временная метка и префикс размещены в хранимых байтах.

Общие рекомендации по использованию `Uuid` в качестве первичного ключа см. в разделе [UUID в качестве первичного ключа](../../../dev/primary-key/row-oriented.md#uuid-primary-key).

## Ключевые генераторы {#key-friendly}

Эти функции формируют UUID версии 8 (implementation-specific по [RFC 9562](https://datatracker.ietf.org/doc/html/rfc9562)), оптимизированные для сортировки ключей в {{ ydb-short-name }}. Порядок сортировки определяется сравнением хранимых 16 байтов, а не канонической GUID-строкой.

### `Uuid::newChrono` {#newchrono}

Генерирует ключ, порядок которого следует времени создания с точностью до миллисекунд, а затем случайным битам суффикса. Строки, вставленные близко по времени, попадают в соседние диапазоны ключей. Это улучшает локальность при сканировании по временному диапазону и при обслуживании индексов по сравнению с полностью случайными ключами.

Необязательные аргументы-зависимости работают так же, как у [`RandomUuid()`](../../builtins/basic.md#random): они управляют моментом вычисления функции для каждой строки, а не содержимым генерируемого значения.

* `Uuid::newChrono([T1[, T2, ...]]) -> Uuid`

### `Uuid::newChronoPrefix` {#newchronoprefix}

Та же хронологическая раскладка, что у `newChrono`, но старшие биты ключа (используемый размер префикса — 10 бит) берутся из первого аргумента, а не выбираются случайно. Используйте для закрепления общего префикса у нескольких строк в одной транзакции или пакете, чтобы они попадали в одну партицию.

Первый аргумент может быть:

* `Uint64` — младшие 10 бит используются как префикс;
* `Uuid` — старшие 10 бит MSB исходного значения используются как префикс.

Далее могут следовать необязательные аргументы-зависимости.

* `Uuid::newChronoPrefix(Uint64{Flags:AutoMap}[, T1[, T2, ...]]) -> Uuid`
* `Uuid::newChronoPrefix(Uuid{Flags:AutoMap}[, T1[, T2, ...]]) -> Uuid`

### `Uuid::newSharded` {#newsharded}

Генерирует ключ, балансирующий распределение по партициям и временную локальность. При каждом вызове старшие биты (префикс) остаются случайными (по умолчанию 2<sup>10</sup> ≈ 1024 групп значений), в следующее битовое поле записывается текущее Unix-время с точностью до секунды, остальные биты заполняются случайностью. Нагрузка на запись распределяется по партициям, а строки, созданные в близкие моменты времени, остаются относительно близко в пространстве ключей.

Необязательные аргументы-зависимости работают так же, как у [`RandomUuid()`](../../builtins/basic.md#random).

* `Uuid::newSharded([T1[, T2, ...]]) -> Uuid`

### `Uuid::newShardedPrefix` {#newshardedprefix}

Та же шардированная раскладка, что у `newSharded`, но префикс фиксируется первым аргументом. Строки с общим префиксом обычно принадлежат одной партиции, что упрощает многострочные транзакции; встроенная временная метка по-прежнему группирует близкие по времени записи внутри своего префикса.

Первый аргумент — `Uint64` или `Uuid` (те же правила, что у `newChronoPrefix`). Далее могут следовать необязательные аргументы-зависимости.

* `Uuid::newShardedPrefix(Uint64{Flags:AutoMap}[, T1[, T2, ...]]) -> Uuid`
* `Uuid::newShardedPrefix(Uuid{Flags:AutoMap}[, T1[, T2, ...]]) -> Uuid`

### Выбор между `Uuid::newChrono` и `Uuid::newSharded` {#key-friendly-choice}

| Задача | Функция |
| --- | --- |
| Максимальный хронологический порядок в первичном ключе; сканирование по временному диапазону по ключу | `Uuid::newChrono` |
| Равномерное распределение по партициям при одиночных вставках с некоторой временной локальностью, эффективное кеширование данных | `Uuid::newSharded` |
| Сокращение количества затрагиваемых партиций при записи множества строк в одной транзакции | `Uuid::newChronoPrefix` или `Uuid::newShardedPrefix` с одним префиксом |
| Неструктурированные случайные идентификаторы без семантики сортировки | [`RandomUuid()`](../../builtins/basic.md#random) |

## RFC UUID версии 7 {#rfc-v7}

Функции `Uuid::newV7` и `Uuid::newV7At` генерируют стандартные значения [RFC 9562 UUID версии 7](https://datatracker.ietf.org/doc/html/rfc9562): 48-битная Unix-метка в миллисекундах в старших битах и случайный суффикс. Результат хранится во внутреннем представлении `Uuid` {{ ydb-short-name }}. Используйте их при необходимости совместимости с инструментами RFC v7 или извлечения встроенной временной метки.

Поскольку UUIDv7 следует раскладке байтов RFC, его порядок сортировки в {{ ydb-short-name }} не соответствует хронологии. Для производительности таблиц в {{ ydb-short-name }} в первичных ключах предпочтительнее использование `Uuid::newChrono` или `Uuid::newSharded`.

### `Uuid::newV7` {#newv7}

Генерирует v7 UUID по текущей временной метке.

* `Uuid::newV7([T1[, T2, ...]]) -> Uuid`

### `Uuid::newV7At` {#newv7at}

Генерирует v7 UUID по явно заданной временной метке. Принимает `Timestamp` или `Timestamp64`.

* `Uuid::newV7At(Timestamp{Flags:AutoMap}[, T1[, T2, ...]]) -> Uuid`
* `Uuid::newV7At(Timestamp64{Flags:AutoMap}[, T1[, T2, ...]]) -> Uuid`

### `Uuid::extractTs` и `Uuid::extractTs64` {#extract-ts}

Извлекают временную метку, встроенную в v7 UUID. Возвращают `NULL`, если аргумент не является UUIDv7 (например, ключ от `newChrono` или `newSharded`).

* `Uuid::extractTs(Uuid{Flags:AutoMap}) -> Timestamp?`
* `Uuid::extractTs64(Uuid{Flags:AutoMap}) -> Timestamp64?`

## Примеры {#examples}

Одиночная вставка со шардированными ключами (по умолчанию — случайный префикс на каждую строку):

```yql
INSERT INTO events (id, payload)
SELECT
    Uuid::newSharded(TableRow()) AS id,
    payload
FROM AS_TABLE($rows);
```

Многострочная транзакция с общим префиксом (строки обычно попадают в одну партицию):

```yql
$prefix = RandomNumber(1);

INSERT INTO events (id, payload)
SELECT
    Uuid::newShardedPrefix($prefix, TableRow()) AS id,
    payload
FROM AS_TABLE($rows);
```

Хронологический первичный ключ:

```yql
INSERT INTO audit_log (id, message)
VALUES (Uuid::newChrono(), "user signed in");
```
